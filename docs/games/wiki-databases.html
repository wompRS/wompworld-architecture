Databases for Game Servers

WIKI 10 DATABASES COMPARISON GUIDE

Every persistent multiplayer game needs a place to store data that survives server restarts, player disconnects, and hardware failures. This page covers ten database systems used in production game servers, from traditional SQL engines to bleeding-edge approaches that merge the database and game server into one.

Introduction

The database is the backbone of any persistent game server. It determines how fast players can save progress, how reliably the world survives a crash, how many concurrent users the system can handle, and how expensive the whole thing is to operate. Pick the wrong database and you hit a wall at 500 concurrent players. Pick the right one and you might handle 50,000.

There is no single "best" database for games. The right choice depends on what kind of data you are storing, how often it changes, whether you need strong consistency or can tolerate stale reads, and how much money you want to spend. Most production MMOs use multiple databases for different jobs. Albion Online uses Cassandra for game state and PostgreSQL for its marketplace. Fortnite uses MongoDB for player profiles and probably something else for real-time match state. EVE Online runs the entire universe on SQL Server but caches aggressively in memory.

This guide covers ten database systems that appear in production game servers. For each one, we explain how it works, what it is good at, what it is bad at, and which games from our 18-game survey use it. The comparison table at the top gives you the quick overview; the deep dives below give you the details.

The Golden Rule of Game Databases

Never hit the database in the hot path. Load player data into memory at session start, process game logic against the in-memory state, and write changes back asynchronously. Every major MMO does this. The database is for persistence and recovery, not for real-time game logic. The exception is SpacetimeDB, which merges the two layers entirely.

MySQL / MariaDB

How It Works

MySQL is a relational database. You define tables with columns, rows, and relationships between them using foreign keys. When your game needs to look up a player's inventory, it runs a SQL query against the characters table, which joins to the inventory table, which joins to the items table. Data is structured, typed, and validated by the database engine before it gets stored.

MariaDB is a fork of MySQL created in 2009 by the original MySQL creator, Michael Widenius, after Oracle acquired Sun Microsystems (and MySQL with it). For most game server purposes they are interchangeable. MariaDB adds some extra features like the Aria storage engine and the Spider sharding engine, but the core SQL behavior is the same.

Storage Engines: InnoDB vs MyISAM

MySQL supports pluggable storage engines, and the two most relevant for game servers are InnoDB and MyISAM.

InnoDB is the default and the right choice for almost all game server work. It supports transactions (so a trade between two players either fully completes or fully rolls back), uses row-level locking (so two players updating their separate inventories do not block each other), and is crash-safe (the database recovers cleanly after an unexpected shutdown). InnoDB keeps frequently accessed data in a buffer pool in RAM, which is why tuning innodb_buffer_pool_size is the single most impactful MySQL performance knob.

MyISAM is an older engine that uses table-level locking. When one player writes to a MyISAM table, every other write to that same table waits. This is fine for read-heavy, rarely-updated data like a static item definition table, but it is completely unsuitable for any table that sees concurrent writes from multiple players. MyISAM is also not crash-safe. If the server dies mid-write, you can lose data or corrupt the table. Do not use MyISAM for player-facing game data.

Replication and Sharding

MySQL supports asynchronous replication where a primary server streams its write log to one or more read replicas. This is useful for offloading read-heavy queries (like leaderboard lookups or marketplace searches) to replicas while the primary handles all writes. Semi-synchronous replication adds a guarantee that at least one replica has received the write before the primary acknowledges it to the application, reducing the risk of data loss during a failover.

MariaDB extends this with Galera Cluster, which provides true synchronous multi-master replication. Every node in a Galera cluster can accept writes, and all nodes are guaranteed to have the same data. The tradeoff is higher write latency because every write must be confirmed by a quorum of nodes before it completes. MariaDB also offers MaxScale, a database proxy that can route queries to appropriate shards based on schema rules.

Horizontal sharding (splitting data across multiple independent MySQL instances) is possible but requires application-level logic. You decide that players with IDs 1-1,000,000 go to shard A and players 1,000,001-2,000,000 go to shard B, then your application code routes queries accordingly. This works but adds significant complexity to your codebase.

Performance at MMO Scale

MySQL performs well for typical game workloads when properly tuned. The buffer pool should be sized to fit your entire working dataset in RAM if possible. Connection pooling is essential because MySQL's thread-per-connection model gets expensive above a few hundred connections. ProxySQL or MaxScale can sit between your game servers and MySQL to manage connection pooling and query routing.

The fundamental bottleneck with MySQL at MMO scale is write throughput on a single primary. A well-tuned MySQL server on modern NVMe storage can handle tens of thousands of writes per second, which is enough for most games if you batch your writes and keep transactions short. But if you need to scale writes beyond what one machine can handle, you are looking at manual sharding, which is a significant engineering investment.

Which Games Use It

World of Warcraft is the flagship example. WoW private server emulators (AzerothCore, TrinityCore, VMaNGOS) all use MySQL/MariaDB with a three-database architecture: auth (account data, realm lists), characters (player saves, inventories, guild data), and world (static content like NPC spawns, item definitions, quest data). While Blizzard's production infrastructure is not public, the emulator architecture closely mirrors the original design. RuneScape's backend also historically used MySQL for player data storage.

When to Use It

  • You want a battle-tested relational database with massive community support
  • Your game needs structured data with clear relationships between entities
  • You are comfortable with SQL and want something that "just works" for most use cases
  • Write volume fits on a single primary server (up to ~50K writes/sec with good hardware)

When to Avoid It

  • You need horizontal write scaling without application-level sharding logic
  • Your data model is highly dynamic and does not fit neatly into fixed schemas
  • You are building a global game that needs multi-region write capability

PostgreSQL

How It Works

PostgreSQL is a relational database, like MySQL, but with a significantly richer feature set. It supports everything MySQL does (tables, joins, indexes, transactions) plus a bunch of advanced features that are genuinely useful for game development: JSONB columns for semi-structured data, built-in pub/sub via LISTEN/NOTIFY, full-text search, array columns, range types, and custom data types.

The core difference from MySQL is PostgreSQL's MVCC (Multi-Version Concurrency Control) implementation. When a transaction reads data, it sees a consistent snapshot of the database as of the transaction's start time. Readers never block writers and writers never block readers. This is critical for game servers where you might have one thread reading a player's inventory while another thread is updating it. In PostgreSQL, both operations proceed without locking conflicts.

JSONB: Flexible Schema Within a Relational Database

PostgreSQL's JSONB column type lets you store arbitrary JSON documents inside a relational table. This is extremely useful for game data that does not fit a fixed schema. Think player inventories where different item types have different properties, achievement metadata, quest progress with variable completion criteria, or equipment with randomized stat rolls.

You can index JSONB columns using GIN (Generalized Inverted Index) indexes, which makes querying into the JSON structure fast. Want to find all players who have a specific item in their inventory? GIN index on the inventory JSONB column, and PostgreSQL can answer that query efficiently.

The catch: JSONB columns have write amplification issues. PostgreSQL's MVCC works by creating new row versions rather than updating in place. When you update a single field inside a JSONB document, PostgreSQL writes a complete new copy of the entire row, including every index entry that points to it. For tables with large JSONB documents that get frequently updated, this can become a serious performance problem. The Heap-Only Tuple (HOT) optimization that helps with regular column updates does not apply to GIN-indexed JSONB fields.

Pub/Sub for Game Events

PostgreSQL's LISTEN/NOTIFY feature lets your game servers subscribe to named channels and receive notifications when something changes. A marketplace server can LISTEN on a channel and get notified whenever a new buy order is placed, without polling the database. This is simpler than setting up a separate message broker like RabbitMQ for basic event-driven patterns, though it does not scale as well as dedicated message queues under very high throughput.

Performance Tuning for Game Workloads

Key PostgreSQL tuning parameters for game servers: shared_buffers (typically 25% of available RAM), effective_cache_size (75% of RAM, tells the query planner how much OS cache to expect), work_mem (memory per sort/hash operation), and max_connections (keep this lower than you think and use a connection pooler like PgBouncer). PostgreSQL's thread-per-connection model is heavier than MySQL's, so connection pooling is even more important.

For read-heavy workloads, streaming replication to read replicas works well. For write scaling, the Citus extension (now part of Microsoft's Azure ecosystem) adds transparent sharding to PostgreSQL, distributing data across multiple nodes while maintaining the ability to run SQL queries across shards.

Which Games Use It

Albion Online is the clearest example. Sandbox Interactive uses PostgreSQL for their marketplace database, gold market database, and accounts database. These are query-heavy systems where PostgreSQL's advanced SQL features, strong consistency, and indexing capabilities are more valuable than Cassandra's write throughput. The marketplace in particular needs complex queries (filter by item type, price range, seller, expiration time) that are natural to express in SQL but awkward in a key-value or wide-column store.

When to Use It

  • You need complex queries beyond simple key-value lookups (marketplaces, auction houses, social features)
  • Your data model mixes structured relational data with semi-structured JSONB data
  • You want one database that can handle relational queries, full-text search, and pub/sub
  • Data correctness matters more than raw write throughput

When to Avoid It

  • You need extremely high write throughput and can tolerate eventual consistency
  • Your workload is purely key-value with no complex query requirements
  • You need seamless multi-region write capability without third-party extensions

SQL Server

How It Works

Microsoft SQL Server is a commercial relational database that runs primarily on Windows (with a Linux version available since 2017). It provides the same relational model as MySQL and PostgreSQL but with enterprise features aimed at large-scale, high-availability deployments: In-Memory OLTP (formerly Hekaton) for tables that live entirely in RAM, AlwaysOn Availability Groups for automatic failover, and deep integration with Windows Server's memory and CPU management.

SQL Server's query optimizer is considered one of the best in the industry for complex analytical queries, and its tooling (SQL Server Management Studio, Profiler, Activity Monitor) gives detailed visibility into query performance and resource usage.

EVE Online: The Definitive Case Study

EVE Online's Tranquility cluster is the most documented example of SQL Server in game production. CCP Games has published detailed accounts of their database hardware evolution, and the numbers are staggering.

The current database servers run twin Intel Gold 6346 CPUs at 3.6GHz (64 cores total across 2 NUMA nodes) with 4 TB of RAM and NVMe storage arrays that replaced racks of legacy 10K SAS disks and older SSDs. The entire EVE universe database fits in memory, which eliminates disk I/O as a bottleneck for read operations.

CCP maintains approximately 15,000 open database connections from the game cluster to SQL Server at any given time. This created a serious NUMA (Non-Uniform Memory Access) problem: the previous hardware had 4 NUMA nodes, and SQL Server's connection management was causing one NUMA node to max out while the other three sat nearly idle. The fix turned out to be surprisingly simple. CCP changed their database connection pool from a stack (last-in, first-out) to a queue (first-in, first-out), which distributed load evenly across all NUMA nodes. The new hardware with only 2 NUMA nodes further reduced the problem.

Storage was another major evolution. CCP moved from traditional fiber channel disk arrays to SSDs, and eventually to a compact NVMe array. The result was effectively zero database lag for normal operations. Server startup and shutdown times dropped dramatically.

Licensing Cost

This is the big barrier. SQL Server Enterprise Edition costs approximately $15,123 per 2-core license pack, with a minimum of 4 cores per server. A 16-core server costs roughly $120,984 in licensing alone. A 64-core server like EVE's would cost approximately $483,936 per machine. Add Software Assurance for upgrade rights and unlimited virtualization, and you are looking at a very significant ongoing expense.

SQL Server Standard Edition is cheaper (~$3,945 per 2-core pack) but has memory limits (128 GB max buffer pool) that make it unsuitable for the kind of "put the entire database in RAM" approach that EVE uses. For a game studio, the licensing math often pushes teams toward PostgreSQL or MySQL unless they specifically need SQL Server's enterprise features or are already invested in the Microsoft ecosystem.

NUMA Considerations

SQL Server is NUMA-aware and distributes buffer cache pages across NUMA nodes so that threads can access locally-allocated memory. But as CCP discovered, the way your application uses connections can defeat this optimization. If your connection pool always hands out the most recently returned connection (a stack), you end up routing most traffic through one NUMA node. The fix is simple in concept (use a queue instead) but the problem is subtle enough that it took CCP years to diagnose.

Which Games Use It

EVE Online is the primary example from our survey. CCP has been running on SQL Server since the game launched in 2003, progressing through multiple hardware generations. The combination of SQL Server's in-memory capabilities, strong ACID guarantees, and single-node performance makes it well-suited for EVE's single-shard architecture where all game data must be consistent and immediately available.

When to Use It

  • You are building a single-shard game and can throw serious hardware at one database node
  • You need in-memory OLTP for latency-critical data access
  • Your team is already in the Microsoft ecosystem with Windows servers and .NET
  • You can afford the licensing costs (or qualify for a startup program)

When to Avoid It

  • You are an indie studio with a limited budget (licensing costs will eat you alive)
  • You want to run on Linux without any Windows infrastructure
  • You need horizontal scaling across many database nodes
  • Your game design requires multi-region write capability

SQLite

How It Works

SQLite is a relational database engine that runs inside your application process rather than as a separate server. The entire database lives in a single file on disk. There is no network protocol, no authentication, no separate daemon to manage. You link the SQLite library into your game binary, open a file, and run SQL queries against it. The library is about 700 KB and has zero external dependencies.

SQLite supports full ACID transactions, foreign keys, triggers, views, and most of the SQL standard. It is one of the most widely deployed pieces of software in existence, embedded in every Android phone, every iPhone, every major web browser, and most operating systems.

WAL Mode

By default, SQLite uses a rollback journal that locks the entire database during writes. WAL (Write-Ahead Logging) mode changes this so that reads and writes can proceed concurrently. In WAL mode, writes go to a separate WAL file, and readers continue to see the previous consistent state until the write commits. This dramatically improves performance for workloads with concurrent readers and occasional writers.

WAL mode has limitations: all processes accessing the database must be on the same machine (WAL does not work over network filesystems), the WAL file can grow unbounded if checkpointing is not managed, and the database must be writable (you cannot open a WAL-mode database read-only unless the WAL index files already exist).

The Single-Writer Bottleneck

SQLite can only handle one write operation at a time. Period. If two threads try to write simultaneously, one waits. For a single-player game saving progress, this is totally fine. For a small multiplayer server with 5-10 players where writes are infrequent, it is probably fine. For anything with frequent concurrent writes from many players, it becomes a serious problem.

A game server that needs to constantly update the state of hundreds of players (positions, inventories, health, quest progress) would be crippled by this single-writer constraint. Each write blocks all other writes, creating a bottleneck that scales linearly with player count.

Which Games Use It

Minecraft Bedrock Edition does not use SQLite for world storage (it uses LevelDB), but many Minecraft server plugins and mods use SQLite for configuration and small data stores. Single-player games across the industry commonly use SQLite for save files. It is also widely used in mobile games on iOS and Android where each device needs a local database.

When to Use It

  • Single-player game saves
  • Local data storage on mobile (player preferences, cached data, offline progress)
  • Development and prototyping (swap in a real database later)
  • Small multiplayer servers with fewer than ~20 concurrent players and infrequent writes
  • Static data like item definitions, map data, or localization strings

When to Avoid It

  • Any multiplayer game with significant concurrent write load
  • Any scenario requiring network access to the database from multiple machines
  • Any game that needs horizontal scaling
  • Production MMO servers of any size

MongoDB

How It Works

MongoDB is a document database. Instead of tables with fixed columns, you have collections of documents. Each document is a JSON-like object (stored internally as BSON, a binary JSON format) that can have any structure. A player document might contain their username, level, an array of inventory items (each with different properties), a nested object for quest progress, and a list of achievement timestamps. No two player documents need to have the same fields.

This flexibility is the core appeal for game development. Games are constantly adding new features, and each feature brings new data fields. With a relational database, adding a new column to a 10-million-row table requires a schema migration. With MongoDB, you just start writing documents with the new field, and old documents without it continue to work fine.

Consistency and Transactions

A common misconception is that MongoDB is "eventually consistent." This was closer to true in older versions, but modern MongoDB (4.0+) provides strong consistency for reads and writes to the primary node. Single-document operations are always atomic. Multi-document transactions have been supported since version 4.0, and cross-shard transactions since version 4.2. MongoDB is an ACID-compliant database when configured with appropriate write and read concerns.

The "eventual consistency" reputation comes from reading from secondary replicas, which are asynchronously replicated from the primary. If you read from a secondary, you might get stale data. For game logic that needs the latest state, you read from the primary. For analytics dashboards or social feeds where slight staleness is acceptable, reading from secondaries reduces load on the primary.

Sharding

MongoDB's built-in sharding distributes data across multiple replica sets automatically. A sharded cluster consists of shards (each a replica set holding a subset of data), mongos routers (which direct queries to the right shard), and config servers (which track which data lives where). You choose a shard key, and MongoDB distributes documents across shards based on that key.

Choosing a good shard key is critical. For player data, sharding by player ID distributes load evenly but makes queries like "find all players in region X" hit every shard. Sharding by geographic region keeps regional data together but can create hotspots if one region has more players. MongoDB supports location-based sharding through Atlas Global Clusters, which associates data with specific geographic zones.

The Fortnite Outage: A Cautionary Tale

On February 3-4, 2018, Fortnite experienced a major outage when concurrent users spiked to 3.4 million. The backend at the time relied on MongoDB sharding, Memcached caching, and XMPP messaging. The Matchmaking Control Plane was handling 124K requests/sec, the database was processing 318K reads/sec and 132K writes/sec, and the authentication system was handling 100K requests/sec.

The primary database shard for the MCP service could not handle the sudden surge. The resulting cascade took down matchmaking, authentication, and social features. Epic Games' post-mortem identified that their data was not properly shardable and they were prototyping in-memory database solutions as a fix. This is a real-world lesson that MongoDB (or any database) at massive scale requires careful shard key design and capacity planning.

Which Games Use It

Fortnite used MongoDB for its primary player profile and matchmaking data, running 9 shards where each shard had a writer, two read replicas, and a hidden replica. MongoDB is also widely used in mobile games and session-based multiplayer games where the flexible document model maps naturally to player profiles and game session data.

When to Use It

  • Player profiles with frequently changing schemas as you add game features
  • Session-based games where each match generates a variable-structure document
  • You need built-in horizontal scaling without managing sharding logic yourself
  • Your team is more comfortable with JSON documents than SQL tables

When to Avoid It

  • You need complex cross-collection queries with joins (MongoDB can do this with $lookup, but it is not what the system is optimized for)
  • Your data has strong relational structure that benefits from foreign keys and referential integrity
  • You need the absolute lowest write latency for real-time game state (the network round-trip to MongoDB still exists)
"MongoDB is Web Scale"

The famous satirical video from 2010 mocked MongoDB's early marketing, which oversold its capabilities and downplayed its consistency tradeoffs. Modern MongoDB (v7+) is a genuinely capable database with ACID transactions, strong consistency on primaries, and production-proven sharding. The meme is outdated, but the underlying lesson remains: no database is magic, and understanding your consistency and scaling requirements matters more than marketing claims.

Redis

How It Works

Redis is an in-memory data structure store. All data lives in RAM, which is why Redis operations typically complete in under one millisecond. Unlike a traditional database where you store rows in tables, Redis stores data in typed structures: strings, hashes, lists, sets, sorted sets, bitmaps, and streams. Each structure has specialized commands optimized for specific operations.

The key insight with Redis is that it is not a general-purpose database replacement. It is a specialized tool that excels at specific patterns. You almost always use Redis alongside a primary database (PostgreSQL, MySQL, MongoDB), not instead of one.

Game Server Use Cases

Caching

Store frequently accessed data in Redis to avoid hitting your primary database. Player session data, item definitions, NPC stats, map metadata. The 80/20 rule applies: roughly 20% of data accounts for 80% of accesses. Caching that hot 20% in Redis can reduce primary database load dramatically.

Leaderboards

Redis sorted sets are purpose-built for rankings. Each member has a score, and Redis maintains the sort order automatically. Adding a score, getting a player's rank, and fetching the top N players are all O(log N) operations. Real-time leaderboards with millions of entries and sub-millisecond updates.

Pub/Sub and Game Events

Redis Pub/Sub lets game servers broadcast events to subscribers. A chat system, kill feed, auction notifications, or inter-server communication can all run through Redis channels. Subscribers receive messages in real time without polling. Redis Streams provide persistent message logs if you need guaranteed delivery.

Session Store

Store player session data (auth tokens, current server, party membership) in Redis instead of on individual game servers. This eliminates "sticky sessions" when load-balancing across multiple game server instances. Any server can look up any player's session state in Redis.

Persistence

Although Redis is in-memory, it provides persistence options. RDB snapshots save a point-in-time copy of all data to disk at configured intervals. AOF (Append Only File) logs every write operation, providing stronger durability at the cost of more disk I/O. You can combine both methods. But persistence is not Redis's primary strength. If you need guaranteed durability for critical game data, that data should live in your primary database, with Redis serving as a fast cache layer.

Scaling

Redis Cluster distributes data across multiple nodes using hash slots. Each key is assigned to one of 16,384 hash slots, and slots are distributed across cluster nodes. This provides horizontal read and write scaling with automatic failover. Redis Enterprise adds Active-Active replication across data centers using CRDTs (Conflict-Free Replicated Data Types) for global games that need low-latency access from multiple regions.

Which Games Use It

Redis is used across the industry as a caching and session management layer. Roblox is known to use Redis caching extensively. Most major online games use Redis or a similar in-memory store (Memcached) alongside their primary database. It is rare to find a production multiplayer game at scale that does not use some form of in-memory caching.

When to Use It

  • Always, as a caching layer alongside your primary database
  • Leaderboards, rankings, counters, rate limiting
  • Session state that needs to be shared across multiple game server instances
  • Real-time pub/sub for chat, notifications, or inter-server messaging

When to Avoid It

  • As your only database (data that matters needs durable storage)
  • For data that does not fit in RAM (Redis stores everything in memory, and memory is expensive)
  • Complex queries with filtering and aggregation (Redis is not a query engine)

Apache Cassandra

How It Works

Cassandra is a wide-column store, which sounds exotic but is conceptually similar to a hash table of hash tables. Data is organized into tables, but unlike a relational database, each row can have a different set of columns. Rows are identified by a partition key (which determines which node stores the data) and optional clustering keys (which determine the sort order within a partition).

Under the hood, Cassandra uses a Log-Structured Merge Tree (LSM tree) storage architecture. Writes go to an in-memory Memtable first (extremely fast), then periodically flush to immutable SSTables (Sorted String Tables) on disk. A commit log ensures durability in case the node crashes before a Memtable flush. This architecture is why Cassandra is so fast at writes: the write path never needs to read existing data from disk or update an index in place.

Cassandra is designed from the ground up for horizontal scaling. You add nodes to the cluster, and Cassandra automatically redistributes data using consistent hashing. There is no single master node. Every node can accept reads and writes for any data, routing requests to the appropriate replica nodes internally. This means there is no single point of failure, which is a big deal for a game that needs to stay up 24/7.

Tunable Consistency

Cassandra's consistency model is tunable on a per-query basis. When you write data, you specify a consistency level that determines how many replica nodes must acknowledge the write before it is considered successful. When you read data, you specify how many replicas must respond.

  • ONE: Only one replica must respond. Fastest, but you might read stale data if the node you hit has not received the latest write yet.
  • QUORUM: A majority of replicas must respond. This guarantees you read the latest written value (assuming a replication factor of 3 and QUORUM for both reads and writes). Slower, but strongly consistent.
  • ALL: Every replica must respond. Strongest consistency, but if any replica is down, the operation fails. Almost never used in production.

Research using Probabilistically Bounded Staleness shows that even with consistency level ONE (the weakest), Cassandra typically converges within milliseconds. With a replication factor of 3, the first read after a write returns current data about 86% of the time immediately and 99% of the time within tens of milliseconds.

Conflict Resolution

When two writes to the same data happen on different nodes, Cassandra resolves the conflict using last-write-wins based on timestamps. This means your clocks matter. If node clocks are skewed, the "wrong" write can win. Running NTP (Network Time Protocol) across all Cassandra nodes is not optional.

Albion Online's Architecture

Albion Online is the clearest example of Cassandra in game production. Sandbox Interactive chose Cassandra as their primary game database because they needed a highly scalable NoSQL store for their single-world MMO. The team uses Cassandra for core game state (player inventories, positions, world objects) while keeping PostgreSQL for query-heavy systems like the marketplace.

Albion's approach leverages Cassandra's strengths: they configure slow reads but very fast writes, because most data is loaded into the game server's memory when a player logs in and written back asynchronously. The database is optimized for the write path, and reads only happen at session start or during server-to-server handoffs when a player moves between world clusters.

The Cassandra cluster can be expanded by adding nodes as the player base grows, with no downtime and no manual data migration. This horizontal scalability was a key factor in choosing Cassandra for a game designed to have every player in one shared world.

When to Use It

  • Write-heavy workloads where game state is constantly being updated
  • You need horizontal scaling that handles growth by adding nodes, not buying bigger machines
  • High availability is critical and you cannot tolerate single points of failure
  • Your data access patterns are well-defined (Cassandra requires designing your data model around your queries)

When to Avoid It

  • You need ad-hoc queries with complex joins (Cassandra's query language, CQL, is deliberately limited)
  • Your workload is read-heavy with complex filtering (Cassandra is optimized for writes, not reads)
  • You need strong consistency without tuning (Cassandra defaults to eventual consistency)
  • Your team is small and you do not want to manage a distributed cluster

DynamoDB

How It Works

DynamoDB is Amazon's fully managed NoSQL database service. You create a table, define a partition key (and optionally a sort key), and start reading and writing items. AWS handles everything else: provisioning, replication, backups, scaling, patching, and hardware failures. You never see a server. You never manage a disk. You never configure replication. It just works.

Each item in a DynamoDB table is a collection of attributes (similar to a MongoDB document but simpler). Items are identified by their primary key and can contain strings, numbers, binary data, lists, maps, and sets. Items can be up to 400 KB in size.

DynamoDB provides two consistency options per read: eventually consistent reads (the default, which may return slightly stale data but consume half the read capacity) and strongly consistent reads (which always return the latest data but cost twice as much). Writes are always strongly consistent on the primary.

Capacity Modes

On-Demand: You pay per read and write request with no capacity planning. DynamoDB automatically scales to handle whatever traffic you throw at it. This is the simpler option and, since November 2024 when AWS cut on-demand pricing by 50%, it is now the recommended default for most workloads.

Provisioned: You specify how many read and write capacity units per second you need, and pay a flat rate. Cheaper than on-demand for stable, predictable workloads. Auto-scaling can adjust provisioned capacity based on utilization, but there is a lag between traffic spikes and scale-up. Reserved capacity offers further discounts for committed usage.

Global Tables

DynamoDB Global Tables replicate data across multiple AWS regions with active-active replication. Players in US-East read from and write to the US-East replica; players in EU-West use the EU-West replica. Conflicts are resolved using last-writer-wins with timestamps. As of November 2024, AWS reduced Global Tables pricing by up to 67%, making multi-region replication significantly cheaper.

New World: AWS-Native MMO Architecture

New World by Amazon Games is the primary example of DynamoDB in MMO production. The game writes approximately 800,000 state updates every 30 seconds to DynamoDB. AWS Lambda functions power over 150 million invocations per minute for microservices including achievements, guilds, characters, sessions, and globally unique naming. The game logs 23 million events per minute through Kinesis into S3.

New World's architecture is fully cloud-native: DynamoDB for persistence, Lambda for compute, EC2 instances as application routers, and Kinesis/S3/Athena for analytics. This approach let Amazon Games scale to over 900,000 concurrent users during launch weekend without provisioning dedicated database hardware.

Cost at Scale

DynamoDB's cost scales with usage, which can be a double-edged sword. Read and write traffic is the biggest cost driver. Each Global Secondary Index multiplies write costs because every write to the base table generates a corresponding write to the GSI. Cross-region transfer fees in Global Tables deployments add up. At massive scale, DynamoDB bills can be significant, but you also avoid the operational cost of managing your own database infrastructure.

When to Use It

  • You are building on AWS and want zero database operations overhead
  • You need predictable single-digit millisecond latency at any scale
  • Your data access patterns are simple (get item by key, query by partition + sort key)
  • You need multi-region active-active replication with minimal configuration

When to Avoid It

  • You need complex queries, joins, or aggregations (DynamoDB is a key-value store at heart)
  • You want to avoid vendor lock-in to AWS
  • Your item sizes exceed 400 KB (hard limit per item)
  • You want to run your database locally without cloud dependency (DynamoDB Local exists for dev but is not production-grade)

SpacetimeDB

How It Works

SpacetimeDB takes a radically different approach: the database is the server. In a traditional game architecture, you have game servers that hold state in memory, a database that persists state to disk, and a network layer between them. SpacetimeDB collapses all three into one. You write your game logic as Rust (or TypeScript) modules that run inside the database engine itself. These modules have direct, in-process access to relational tables with no network round-trip.

Clients connect directly to the SpacetimeDB instance via WebSocket and invoke "reducers," which are transactional functions that can read and write tables, execute game logic, and return results. Reducers are atomic, isolated, and durable, similar to stored procedures but designed for real-time game logic rather than batch data processing.

Eliminating the ORM and Networking Layer

In a traditional setup, your game server talks to the database over the network. Every database query involves serialization, network transfer, deserialization, query parsing, execution, result serialization, network transfer back, and deserialization again. This round-trip adds latency even on a local network, and it means your game logic and your data live in separate processes with separate memory spaces.

SpacetimeDB eliminates this entirely. Game logic runs in the same process as the data, accessing tables through direct memory reads. There is no ORM mapping objects to rows, no connection pool to manage, no network serialization between server and database. The BSATN (Binary SpacetimeDB Algebraic Type Notation) wire format handles serialization only for client-to-server communication, not for internal data access.

Subscription Queries

SpacetimeDB provides real-time data synchronization through subscription queries. A client subscribes to a SQL query (like "SELECT * FROM players WHERE region = 'forest'"), and SpacetimeDB pushes transactional updates whenever the result set changes. This replaces the typical game networking pattern of custom packet definitions, delta compression, and state interpolation with a declarative model: tell the database what data you care about, and it keeps you updated.

Performance

SpacetimeDB achieves over 100,000 transactions per second for TypeScript modules and up to 170,000 transactions per second for Rust modules. This performance comes from holding all application state in memory and persisting through a write-ahead log (WAL) that is used for recovery, not for real-time access. The system is optimized for many small transactions with low latency rather than batch processing.

BitCraft Online

BitCraft Online, developed by Clockwork Labs (the same team behind SpacetimeDB), is a survival crafting MMO built entirely on SpacetimeDB. The game has no traditional server infrastructure. Everything in the game, including chat, items, resources, terrain, buildings, and player positions, is stored and processed by SpacetimeDB modules and synchronized to clients in real time. BitCraft requires more persistence than a typical MMO because the entire world is player-editable, so terrain deformation, building placement, and resource depletion all need to be stored and replicated.

Licensing

SpacetimeDB is source-available under the Business Source License 1.1 (BSL 1.1). This means you can read and modify the source code, but there are restrictions on commercial use. It is not open source in the OSI-approved sense. Clockwork Labs offers hosted SpacetimeDB services for teams that do not want to self-host.

When to Use It

  • You are a small team building a multiplayer game and do not want to manage separate server and database infrastructure
  • Your game logic is naturally transactional (trades, inventory changes, combat resolution)
  • You want real-time state synchronization without building custom networking code
  • You are comfortable with Rust or TypeScript for server-side logic

When to Avoid It

  • You need horizontal scaling across multiple machines (SpacetimeDB currently runs as a single module)
  • You need a database with decades of production track record (SpacetimeDB hit 1.0 in 2024)
  • Your game requires custom networking optimizations (UDP, delta compression, client-side prediction) that do not fit the subscription query model
  • You are uncomfortable with BSL licensing and want fully open-source infrastructure
Wompworld Relevance

SpacetimeDB is the most interesting option on this list for a small team building an MMO. It eliminates the need to build separate game servers, database infrastructure, and networking code. The tradeoff is that it is new, the scaling story for truly massive player counts is unproven, and the BSL license means you are somewhat dependent on Clockwork Labs. For a project like Wompworld that needs persistent ocean world state with player-driven economies, the "database is the server" model deserves serious prototyping time before committing to a traditional architecture.

LevelDB / RocksDB

How It Works

LevelDB and RocksDB are embedded key-value stores. "Embedded" means they run as a library inside your application process, not as a separate server. You link the library, open a database directory, and read/write key-value pairs. There is no SQL, no query language, no network protocol. You put bytes in with a key, and you get bytes out with the same key. That is the entire API.

Both use a Log-Structured Merge Tree (LSM tree) storage architecture, which is the same general approach that Cassandra uses. Writes go to an in-memory buffer (MemTable), which periodically flushes to sorted, immutable files on disk (SSTables). Background compaction merges SSTables to maintain read performance and reclaim space from deleted entries.

LevelDB vs RocksDB

LevelDB was created by Jeff Dean and Sanjay Ghemawat at Google and is optimized for simplicity and correctness. RocksDB is a fork created at Facebook that adds multi-threaded compaction, column families, and various optimizations for SSD storage and high-throughput server workloads. RocksDB is generally faster and more configurable, but also more complex.

LevelDB is the better choice for embedded use in game clients where simplicity and small binary size matter. RocksDB is better for server-side use where you need maximum write throughput and can tune compaction settings.

Minecraft Bedrock Edition

Minecraft Bedrock Edition uses a modified version of LevelDB for world storage. Mojang forked Google's LevelDB to add Zlib compression and Windows support (available at github.com/Mojang/leveldb-mcpe). The entire world is stored in a LevelDB database in the db/ subdirectory of each world save.

The key structure is clever: chunk data is keyed by the chunk's X and Z coordinates (as 32-bit little-endian integers), an optional dimension identifier (1 for Nether, 2 for End, omitted for Overworld), and a tag byte indicating the record type (sub-chunk blocks, entities, block entities, etc.). This means looking up all data for a specific chunk is a simple key prefix scan, which LSM trees handle efficiently.

Non-spatial data like player information and village data uses text-encoded keys. The whole system is fast, compact, and requires zero configuration, which makes it well-suited for a game that runs on everything from phones to game consoles to PCs.

RocksDB in Server Infrastructure

While RocksDB is not commonly used as a primary game database, it appears as a storage engine inside other systems that games use. Apache Kafka uses RocksDB for state stores in Kafka Streams. Apache Flink uses it for checkpoints. CockroachDB and TiDB use RocksDB as their underlying storage engine. If you are using any of these systems in your game infrastructure, RocksDB is doing work under the hood.

When to Use It

  • You need fast, embedded key-value storage inside a game client or dedicated server process
  • Your data is naturally key-addressable (world chunks by coordinates, player saves by ID)
  • You want zero infrastructure overhead (no separate database server to manage)
  • You are building a voxel/sandbox game that needs efficient chunk storage

When to Avoid It

  • You need queries more complex than get/put/iterate-range
  • Multiple processes need to access the same database simultaneously (LevelDB supports only one process)
  • You need network access to the data from multiple machines
  • You need relational features like joins, indexes on multiple fields, or transactions spanning many keys

Choosing a Database for Your Game

The decision tree for game databases is simpler than it looks. Start with these questions:

1. What kind of data are you storing?

Structured with Relationships

Player accounts linked to characters, characters linked to inventories, inventories linked to items. If your data has clear foreign key relationships and you need to query across them, use a relational database: PostgreSQL (free, flexible) or MySQL/MariaDB (free, simpler). Use SQL Server if you have the budget and need maximum single-node performance.

Flexible / Schema-Light

Player profiles where every player has different achievements, settings, and equipment. If your data model changes frequently and you do not need complex joins, use MongoDB or DynamoDB. MongoDB if you want to self-host, DynamoDB if you are on AWS and want zero ops.

Write-Heavy State Updates

Constant position updates, health changes, resource ticks. If you have extremely high write volumes and can tolerate eventual consistency, use Cassandra. If you want the database and server combined, evaluate SpacetimeDB.

Chunk / Blob Storage

Voxel world data, procedurally generated terrain, large binary objects keyed by spatial coordinates. Use LevelDB (embedded in client) or RocksDB (embedded in server).

2. How many concurrent writers do you need to support?

  • 1 writer: SQLite is fine. Simplest possible setup.
  • Hundreds: MySQL, PostgreSQL, or SQL Server on a single well-tuned machine.
  • Thousands+: MongoDB (sharded), Cassandra (distributed), DynamoDB (managed), or Redis (in-memory) alongside a primary store.

3. How much can you spend?

  • $0: PostgreSQL, MySQL/MariaDB, SQLite, Cassandra, Redis, LevelDB/RocksDB are all free to run on your own hardware.
  • Pay-per-use: DynamoDB, MongoDB Atlas. No upfront cost, but bills scale with usage.
  • Enterprise budget: SQL Server Enterprise. High upfront cost, but unmatched single-node performance for the right workload.
  • Source-available: SpacetimeDB (BSL 1.1). Free to use within license terms, hosted option available.

4. Do you need multi-region?

If your players are spread across continents and latency matters, your options narrow. DynamoDB Global Tables and Cassandra provide native multi-region replication. MongoDB Atlas offers global clusters. Redis Enterprise provides Active-Active replication. Most other databases require manual setup for cross-region replication.

The Polyglot Persistence Approach

Most production games use multiple databases. A common pattern: PostgreSQL or MySQL for accounts and marketplace data, Cassandra or DynamoDB for high-volume game state, Redis for caching and leaderboards, and maybe LevelDB for local world storage on the client. This is not overengineering. Each database is doing the job it was designed for. The complexity cost of running multiple databases is usually lower than the performance cost of forcing one database to do everything.
