Architecture Comparison / Minecraft

Minecraft

Single-Threaded Tick Sandbox / Survival

The best-selling game of all time runs on a fundamentally single-threaded server loop ticking at 20 TPS. Two completely separate codebases (Java Edition and Bedrock Edition) use different languages, protocols, and world formats, connected by a massive ecosystem of community server software, proxy networks, and mods.

Overview

Minecraft's server architecture is unusual for a game of its scale. The core game loop is single-threaded: one thread processes all entity movement, block updates, redstone circuits, chunk ticking, and player actions sequentially, every 50 milliseconds. This 20-tick-per-second heartbeat is the fundamental constraint that shapes everything about Minecraft server performance, and it has not changed since the game's earliest multiplayer implementation.

There are two completely separate editions of the game. Java Edition runs on the JVM and uses a custom TCP-based protocol. Bedrock Edition is written in C++ and uses the RakNet UDP protocol. They cannot natively connect to each other. They use different world save formats, different networking stacks, and different modding APIs. Bedrock runs on Windows 10/11, consoles, and mobile. Java Edition runs on Windows, macOS, and Linux.

The community server ecosystem is where things get architecturally interesting. Vanilla Minecraft servers top out at maybe 50-80 concurrent players before the tick loop starts lagging. Large networks like Hypixel handle over 200,000 concurrent players by running hundreds or thousands of individual server instances behind a proxy layer (BungeeCord or Velocity) that presents a single IP address to players and routes them between backend servers transparently. Modified server software like Paper, Spigot, and Fabric optimize the vanilla tick loop to squeeze more performance out of the single-threaded constraint.

Mojang's own hosted service, Realms, provides simple multiplayer for up to 10 players per world on pre-configured cloud instances with fixed render distances and no mod support. It is the simplest way to run a Minecraft server and the least capable.

Server Infrastructure

A Minecraft server, at its most basic, is a single JVM process (Java Edition) or a single native process (Bedrock Dedicated Server) running on one machine. There is no built-in sharding, no distributed state, and no horizontal scaling. If you want to serve more players, you need to run more separate server processes and connect them with external tools.

Vanilla Server Architecture

The vanilla Minecraft server (distributed by Mojang as server.jar) is a single-threaded Java application. The main game loop runs every 50ms (20 TPS). In each tick, the server processes: entity AI and movement, scheduled block updates, random block ticks (crop growth, fire spread), redstone circuits, chunk loading and generation, player input processing, and network packet handling. All of this runs sequentially on one thread. If any single step takes too long, the entire server lags.

A vanilla server with a view distance of 10 loads (2 * 10 + 1)^2 = 441 chunks per player. With 10 players scattered across the world, that can be over 4,000 unique chunks loaded simultaneously. Each chunk is 16 blocks wide, 16 blocks long, and 384 blocks tall (since 1.18), containing up to 98,304 blocks. If even a small percentage of those blocks are tile entities (furnaces, hoppers, signs), the per-tick computation adds up fast.

Paper and Spigot

Paper is the community standard for production Minecraft servers. It is a fork of Spigot, which is a fork of CraftBukkit, which is a fork of the vanilla server. Each layer adds performance optimizations and plugin API extensions. Paper typically delivers 20-50% better TPS than Spigot on the same hardware, and uses 15-25% less RAM. Key optimizations include async chunk loading (chunk I/O moved off the main thread), entity activation ranges (distant entities tick less frequently), and aggressive timings profiling to identify bottleneck plugins.

Folia, a newer fork of Paper, introduces regionized multithreading. Instead of one thread ticking the entire world, Folia divides the world into independent regions that can tick in parallel on separate threads. This is a fundamental architectural change that breaks compatibility with most existing plugins but allows servers to scale across CPU cores for the first time.

Proxy Networks (BungeeCord / Velocity)

Large Minecraft networks like Hypixel use a proxy-based architecture. Players connect to a proxy server (Velocity or BungeeCord), which acts as a reverse proxy. Behind the proxy sit dozens or hundreds of independent Minecraft server instances, each running its own JVM. The proxy handles authentication, routing, and seamless server switching. When a player types /lobby or enters a minigame, the proxy transfers their connection from one backend server to another without the player disconnecting.

Velocity (built by the PaperMC team) is the modern replacement for BungeeCord. It offers up to 8x better throughput, uses HMAC-signed modern forwarding (preventing player identity spoofing), and supports native compression libraries. A typical network setup requires the proxy instance (1 GB RAM), a lobby server (2-4 GB RAM), and multiple backend servers scaling based on player count.

Hypixel: The Largest Minecraft Server

Hypixel has peaked at over 200,000 concurrent players. Its architecture uses multiple Velocity/BungeeCord proxies behind Cloudflare Spectrum (for DDoS protection), with every proxy connected to every backend game server. Redis or a similar key-value store tracks per-proxy player counts, and DNS-level load balancing distributes incoming connections. Game servers are generic instances that dynamically configure themselves for whichever minigame is requested. State like player statistics, friends lists, and inventories is stored in external databases (likely Redis + MySQL/MongoDB), not in the Minecraft world files.

Network & Protocols

Java Edition and Bedrock Edition use fundamentally different networking stacks. This is the core reason they cannot interoperate natively.

Java Edition: Custom TCP Protocol

Java Edition uses a custom protocol built on TCP. Every packet is reliably delivered in order. The protocol has distinct phases: Handshake, Status (server list ping), Login (authentication), and Play (gameplay). Packets are length-prefixed, optionally compressed (zlib), and optionally encrypted (AES/CFB8 after authentication).

The NMS (net.minecraft.server) package contains Mojang's proprietary Java code for all packet handling, entity management, and world simulation. Plugin developers historically accessed NMS classes directly to implement features not exposed through the Bukkit API. This was always fragile because NMS classes changed between Minecraft versions with no stability guarantees. As of Paper 1.20.5, the server ships with Mojang mappings by default instead of Spigot's reobfuscated names, making NMS access slightly more stable but still unsupported.

Bedrock Edition: RakNet (UDP)

Bedrock uses RakNet, a UDP-based protocol that adds reliability, ordering, and fragmentation on top of raw UDP datagrams. The default port is 19132 (IPv4). Unlike Java's TCP stream, Bedrock packets have explicit message boundaries. The RakNet handshake begins with MTU discovery to determine the maximum packet size the connection can support. Packets use compression by default and can be encrypted.

Because RakNet is UDP-based, Bedrock handles high-latency connections more gracefully than Java Edition's TCP stack. TCP's head-of-line blocking means one lost packet delays all subsequent packets, while RakNet can deliver packets out of order when reliability is not required. This is why Bedrock can feel snappier on mobile networks with packet loss.

GeyserMC: Protocol Translation

GeyserMC is a proxy that translates between Bedrock's RakNet protocol and Java Edition's TCP protocol in real time. A Bedrock player connects to Geyser, which translates their packets into Java protocol packets and forwards them to a Java server. This allows Bedrock players to join Java servers, but it adds latency and occasionally causes desync issues because the two editions handle entity movement and block placement slightly differently.

Authentication Flow

Both editions now use Microsoft accounts. The flow for Java Edition: the client obtains a Microsoft OAuth2 token, exchanges it for an Xbox Live token, then an XSTS token, and finally a Minecraft access token. When joining a server, the client and server perform a handshake with sessionserver.mojang.com to verify identity. Servers running in "offline mode" skip authentication entirely, which is why private servers can allow cracked clients but also opens the door to identity spoofing.

Data & Persistence

Minecraft Java Edition persists its world using the Anvil file format, a binary system based on NBT (Named Binary Tag) serialization. The world directory contains region files, player data, and world configuration.

Anvil Region Files (.mca)

The world is divided into regions of 32x32 chunks. Each region is stored in a file named r.x.z.mca where x and z are the region coordinates. The file has a 8KB header containing offset and timestamp tables for all 32x32 = 1024 chunk slots, followed by the compressed chunk data.

When a chunk is saved, it is compressed (zlib or zstd depending on version) and written to the next available sector in the region file. If the chunk has grown since its last save (more block entities, for example), the server finds a new contiguous block of sectors to fit it, or appends to the end of the file. This sector-based allocation means region files can develop internal fragmentation over time.

Chunk Format (Post-1.18)

Since the 1.18 world height expansion, chunks span from Y=-64 to Y=320 (384 blocks tall). Each chunk is divided into 24 vertical sections of 16x16x16 blocks. Empty sections are not saved. Block data uses a palette-based system: each section stores a list of unique block states (the palette) and a compacted array mapping each position to a palette index. This is significantly more space-efficient than storing full block IDs for every position.

Biome data is stored in 4x4x4 cells (64 entries per section). Each chunk also stores heightmaps (for lighting and rendering), block entities (furnaces, chests, signs), tile tick schedules (pending block updates), and a generation status field indicating whether the chunk is fully generated or still in a proto-chunk state.

Player Data

Each player's data is stored in an individual NBT file under world/playerdata/[uuid].dat. This includes inventory, position, health, experience, achievements, and other per-player state. Player data files are loaded when a player joins and saved when they disconnect or at regular autosave intervals.

World Configuration

level.dat stores world-level settings: seed, game rules, difficulty, spawn point, time, weather, data pack configuration, and the world generator type. This file is read at server startup and written periodically during play.

Bedrock Edition: LevelDB

Bedrock Edition uses Google's LevelDB instead of the Anvil format. LevelDB is a key-value store that handles its own compaction and caching. Chunk data is stored as key-value pairs where the key encodes chunk coordinates and the value contains the serialized block/entity data. This approach is better suited to the resource-constrained environments (mobile, consoles) where Bedrock runs.

Anvil Format Anatomy

Region file r.0.0.mca contains chunks (0,0) through (31,31). Header: 4KB offset table + 4KB timestamp table. Body: compressed NBT chunk data in 4KB sectors. Each chunk's NBT contains: sections[] (block palettes + packed states), block_entities[] (furnaces, chests, etc.), Heightmaps (WORLD_SURFACE, OCEAN_FLOOR, etc.), structures (references to generated features), Status (generation progress: empty through full), blending_data (for upgrading old chunks to new height format).

Scaling & Performance

The Single-Thread Bottleneck

Minecraft's performance is CPU-bound on a single core. Adding more CPU cores does not help the main game loop. Adding more RAM beyond what is needed for loaded chunks provides diminishing returns. The single most impactful hardware upgrade for a Minecraft server is a CPU with the highest single-thread performance available.

The main tick loop processes everything sequentially: it ticks every loaded entity (mobs, dropped items, minecarts), processes block updates (water flow, fire spread, gravity), runs scheduled tick events (redstone, growing crops), and handles player actions (movement validation, block breaking, chat). If any of these operations takes too long, the server cannot complete its tick within the 50ms budget and TPS drops below 20.

Common Performance Killers

Chunk generation: Generating new terrain in real time is extremely CPU-intensive. Servers that allow free exploration will experience TPS drops when players venture into ungenerated areas. The standard mitigation is pre-generating the world before players arrive.

Entity accumulation: Hundreds of mobs packed into a small area (like a spawner farm) create significant per-tick overhead. Each entity needs pathfinding, AI, and collision checks. Paper mitigates this with entity activation ranges: entities more than 32 blocks from a player tick at reduced frequency or not at all.

Redstone machines: Complex redstone circuits generate cascading block updates. A single redstone pulse can trigger hundreds of block updates that all process within the same tick. The redstone system uses a two-phase tick (produce and consume) that spans 2 game ticks, and the maximum number of scheduled ticks per game tick is 65,536.

Tile entities: Hoppers, furnaces, brewstands, and other block entities run their logic every tick. A storage system with 500 hoppers adds 500 tile entity ticks per game tick. This is why large hopper-based sorting systems are notorious for tanking server performance.

Optimization Layers

The community has built multiple optimization layers. Paper optimizes vanilla code paths (async chunk I/O, entity activation ranges, hoppper optimization). Lithium (Fabric mod) rewrites vanilla algorithms with faster implementations (pathfinding, block tick scheduling, entity collisions). Spark is a profiling tool that generates web-based reports showing exactly what is consuming tick time. The sync-chunk-writes option in server.properties (disabled by default in Paper) moves chunk saving off the main thread, preventing disk I/O from blocking the tick loop.

Simulation Distance vs. View Distance

Since Java 1.18, servers can configure simulation distance and view distance separately. View distance controls how many chunks are sent to the client for rendering. Simulation distance controls how many chunks are actively ticked (entities move, crops grow, redstone operates). Setting view distance to 12 and simulation distance to 4 lets players see far while only ticking a small area, dramatically reducing per-player server load.

Technical Deep Dives

The Tick Loop in Detail

Every 50ms, the server executes a single game tick. The order of operations within a tick is deterministic and fixed. First, the server processes incoming network packets (player movements, block interactions). Then it ticks the world: for each loaded chunk within simulation distance, the server runs scheduled tick events, random block ticks (3 per subchunk by default), and entity ticks. Then it processes chunk loading/unloading. Finally, it sends outbound packets to players (state updates, chunk data). If the total time exceeds 50ms, the server falls behind and the next tick starts immediately, causing TPS to drop.

Random ticks are the mechanism behind organic world changes. Each tick, for each loaded chunk, the server selects 3 random block positions per subchunk (configurable via randomTickSpeed gamerule). If the selected block responds to random ticks (wheat grows, leaves decay, fire spreads), it runs that block's tick logic. Most random tick targets are air blocks, which do nothing. This randomized approach distributes the cost of organic simulation across many ticks rather than doing it all at once.

Forge and Fabric: The Mod Ecosystem

Forge (2011) and Fabric (2018) take fundamentally different approaches to modding. Forge uses a heavyweight event bus system with extensive hooks into vanilla code, providing a stable(ish) API for mods. Fabric uses lightweight mixins that inject code directly into vanilla classes at load time, providing lower overhead but requiring more careful compatibility management between mods. NeoForge (2023) is a community fork of Forge that modernizes the API while maintaining compatibility with the Forge ecosystem.

Mods modify the server at the code level: adding new blocks, entities, dimensions, crafting recipes, world generators, or entirely new game mechanics. Unlike plugins (which work through an API layer), mods require matching installations on both client and server. This is why modded Minecraft is typically played in curated modpacks where everyone installs the same set of mods.

Spawn Chunks and Always-Loaded Regions

A special set of chunks around the world spawn point stays loaded at all times, even when no players are nearby. The central spawn chunks tick at the entity-ticking level, meaning mobs move, redstone operates, and farms produce output continuously. Technical players exploit this by building their most important automated farms (iron farms, crop farms, XP grinders) in the spawn chunks so they run 24/7 as long as the server is up.

Bedrock Marketplace

Bedrock Edition has an official marketplace where creators sell maps, skins, texture packs, and mash-up packs for Minecoins (Bedrock's virtual currency). Unlike Java's free modding ecosystem, Bedrock's marketplace is a curated, monetized storefront. Creators must apply and be accepted into the Minecraft Partner Program. Content goes through a review process before listing. This is a fundamentally different economic model: Java monetizes through server hosting and donations, while Bedrock monetizes through a centralized content marketplace.

Architecture Evolution

2009 Minecraft Classic multiplayer. The first multiplayer implementation, basic client-server with no persistent world saving. Custom Creative-mode only.
2010 Survival Multiplayer (SMP) added. Server-authoritative survival gameplay. World persistence through the Alpha/Infdev save format (later replaced by MCRegion, then Anvil).
2011 Bukkit project starts. Community creates the CraftBukkit server wrapper, introducing the plugin API that will define the Minecraft server ecosystem for the next decade.
2012 Anvil file format and Spigot. MCRegion replaced by Anvil (.mca), supporting block IDs up to 4096. Spigot forks CraftBukkit with performance optimizations. BungeeCord proxy created.
2014 Microsoft acquires Mojang for $2.5 billion. Pocket Edition (later renamed Bedrock) grows alongside Java Edition. Mojang authentication begins migrating toward Microsoft accounts.
2016 Paper project forks Spigot. Focuses on aggressive performance optimization and bug fixes. Becomes the de facto standard for production plugin servers.
2018 Fabric loader launches. Lightweight modding platform as an alternative to Forge. Uses mixin-based injection instead of Forge's event bus. Velocity proxy project starts.
2021 1.18: Caves and Cliffs Part 2. World height doubled to 384 blocks (-64 to +320). Chunk format rewritten to support variable height sections. Terrain generation completely overhauled. Major performance implications for servers.
2023 NeoForge splits from Forge. Community fork after leadership dispute. Paper 1.20.5 ships with Mojang mappings by default, ending the Spigot reobfuscation era. Folia introduces regionized multithreading.
2024 Mojang account migration complete. All players now use Microsoft accounts. Legacy Yggdrasil authentication deprecated. Cross-play between Bedrock platforms continues expanding.
Folia: The Multithreaded Future?

Folia (by the PaperMC team) is the most ambitious attempt to break Minecraft's single-threaded constraint. It divides the world into independent regions that tick in parallel on separate threads. Two players on opposite sides of the world no longer share a tick budget. The catch: Folia breaks nearly all existing plugins because the Bukkit API assumes single-threaded access to the world. Plugins must be rewritten to be thread-safe. This makes Folia a clean break from the existing ecosystem, and adoption has been slow. But for servers with large worlds and spread-out player bases, the potential performance gains are enormous.
