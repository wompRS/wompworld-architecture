Architecture Comparison / Second Life

Second Life

Region-Grid Virtual World / UGC

Fully migrated from on-premises data centers to AWS in 2021 after 17 years of bare-metal operations.

Overview

Second Life is one of the oldest continuously running virtual worlds, launched in 2003 and still operational in 2026. It uses a fundamentally different architecture than most online games: every 256x256 meter patch of virtual land (called a "region" or "sim") runs as a dedicated server process. The collection of all these regions forms "the grid," a contiguous virtual continent where players (called "residents") can walk, fly, or teleport between regions.

At its peak around 2007, Second Life had over 1.1 million monthly active users and was the subject of intense media hype. Concurrency peaked during the COVID-19 pandemic at roughly 60,000 simultaneous users. As of late 2024, the platform has about 500,000 monthly active users and daily concurrency averaging between 34,000 and 50,000.

What makes Second Life architecturally distinctive is that nearly all content is user-created and runs server-side. The scripting language (LSL, Linden Scripting Language) executes on the simulator process, not the client. The economy uses a real-currency-backed token (Linden Dollars, L$) with a floating exchange rate. Users retain copyright on their creations, and the platform includes basic DRM through a permissions system. The entire grid completed migration from Linden Lab's own data centers to Amazon Web Services in January 2021.

The architecture reflects design decisions from 2001-2003 that have been incrementally upgraded for over two decades. Some of these decisions, like tying one region to one process and using UDP for inter-simulator communication, have created persistent problems (region crossing lag, for example) that are deeply embedded in the system's DNA.

Server Infrastructure

Second Life uses a client-server-server architecture. The viewer (client) connects to a simulator process that holds the canonical state for one 256x256 meter region. As the player moves through the world, their connection is handed off from one simulator to the next. Backend services (inventory, presence, search) run as separate processes that the simulators call into.

Simulator (Sim) Processes

The simulator is the primary Second Life server process, written in C++. Each simulator instance is responsible for exactly one region. It stores object state (every prim, mesh, and scripted object in the region), land parcel state (ownership, permissions, access lists), and terrain heightmap state. It handles collision detection, runs the Havok physics engine, and executes all LSL scripts attached to objects in the region.

The simulator targets 45 frames per second. If it cannot maintain that rate, it applies time dilation rather than dropping frames. The sim continues to tick at 45 fps from the server's perspective, but in-world time slows down. A sim running at 50% time dilation is effectively simulating the world at half speed. Players experience this as sluggish movement and slow script execution.

Historically, Linden Lab ran 4 simulator processes per physical server (one per CPU core) on "Class 4" hardware: Debian Linux boxes with 2 dual-core CPUs. Earlier hardware classes ran fewer sims per machine. When a resident purchases a private island, they get dedicated use of a simulator process. Linden Lab moves sims between physical hosts for maintenance or after crashes, but residents do not control placement.

Space Server and Routing

The Space Server maintains the grid coordinate system. When a simulator starts up, it registers its grid (x, y) position with the Space Server. When a simulator needs to know about its neighbors (for region crossing), it queries the Space Server to find which process is running the adjacent region. This routing layer is what makes the grid feel like a contiguous world rather than isolated rooms.

Login Server

The login server is a CGI script running on login.agni.lindenlab.com. When a resident logs in, it verifies credentials, determines the target region (home location, last location, or a specified URL), finds the simulator process running that region, verifies the user is allowed to connect there, alerts the simulator to expect the connection, and tells the viewer where to connect. The login server is a single point of entry that then hands the client off to the appropriate sim.

Data Server

The Data Server handles all connections to the backend databases: the central user database, log database, inventory database(s), and search database(s). It performs queries on behalf of simulator processes, abstracting the database layer away from the sim code.

Network & Protocols

Second Life networking predates many modern game networking conventions. The simulator communicates with clients and other simulators using a UDP-based "circuit" system.

UDP Circuits

A "circuit" in Second Life terminology is a UDP network connection maintained between two endpoints. Circuits are established between the viewer and its current simulator, and between adjacent simulators that share a border. When a player is near a region boundary, their viewer maintains connections to multiple simulators simultaneously, receiving object and terrain data from neighboring regions to provide a seamless view across borders.

The server does not trust the client. All state changes must be validated server-side. It is architecturally impossible to impersonate another account or spoof another client's identity, as all access requires a login token that establishes identity and permissions.

Region Crossing: The Oldest Bug

Region crossings are Second Life's most notorious technical problem. When an avatar or a physical vehicle moves from one region to another, the following happens: the source sim packages up the avatar (or object) state, sends it to the destination sim via the UDP circuit, the destination sim instantiates the avatar, and the viewer switches its primary connection to the new sim. This handoff takes time, and during the transition, the avatar may appear frozen, teleport to unexpected locations, or lose attached objects.

This has been described as the oldest known bug in the Second Life codebase. Unlike a cell phone handoff (which can happen in milliseconds), a sim crossing involves serializing complex object hierarchies with scripts, physics state, and attachments. The problem is structural: each region is a fully independent process with its own state, so crossing a boundary is more like a mini-teleport than a smooth transition.

Voice Chat: Vivox to WebRTC

Voice chat in Second Life originally used Vivox technology based on SIP/RTP protocols. The voice data bypassed the viewer entirely: an external daemon (SLVoice.exe) handled the audio streams, communicating with Vivox's servers independently. The source code for this daemon was never available due to contractual obligations between Linden Lab and Vivox.

Second Life is currently transitioning from Vivox to WebRTC, a free and open-source real-time communication standard. The migration is incomplete, meaning some regions run WebRTC voice and others still run Vivox. Voice does not work across region boundaries when the two regions use different voice backends. Transitioning through a WebRTC region, then a Vivox region, then back to WebRTC can break voice transmission entirely until the user reconnects.

Region Crossing Lag

The region crossing problem has been present since Second Life's earliest days and has never been fully solved. It affects walking, flying, sailing, and driving across region boundaries. Vehicles are particularly vulnerable because they carry physics state, attached scripts, and passenger avatars that all need to transfer atomically. The migration to AWS improved sim performance but did not fundamentally change the crossing architecture, because the problem is in the region-per-process model itself, not the hardware.

Data & Persistence

Second Life uses MySQL as its primary database, running on Debian Linux (historically on-premises, now on AWS). Rather than building one monolithic database, Linden Lab uses a data partitioning strategy based on UUIDs. Almost everything in Second Life has a 128-bit UUID: every user, every object, every texture, every inventory item. This makes partitioning straightforward because UUIDs distribute uniformly across database shards.

Database Architecture

The databases are divided by function: user databases store profiles and authentication data, inventory databases store player-owned items (partitioned by user UUID), asset databases store the actual content (textures, meshes, sounds), region databases store object placement and terrain for each sim, and log/search databases handle analytics and in-world search indexing. The Data Server process handles all queries on behalf of simulators, with Squid caching layers between the queries and MySQL to reduce database load.

Asset System

Every piece of content in Second Life (textures, meshes, animations, sounds, scripts) is stored as an asset with a UUID. When a resident uploads a texture, the viewer converts it to JPEG 2000 format and uploads it to the asset server. The asset is then delivered to other residents' viewers through a CDN. The viewer maintains a local texture cache (LLTextureCache) so frequently seen content does not need to be re-downloaded.

Mesh uploads go through a validation process called Mesh Asset Validation (MAV) that checks for malformed geometry, excessive polygon counts, and physics shape complexity. Each mesh upload has a Linden Dollar cost determined by the model's complexity, though Premium Plus members can upload for free.

Linden Dollar Economy

The Linden Dollar (L$) is Second Life's internal currency. It has a floating exchange rate on the LindeX exchange, historically hovering between L$240 and L$270 per US dollar. Residents buy L$ through the viewer or the LindeX website, and can sell L$ back for real money. In 2013, Linden Lab reported over $3.2 billion in cumulative transaction volume over 10 years, with an average of 1.2 million daily transactions.

The economy is fully user-driven. Residents buy and sell virtual goods (buildings, vehicles, clothing, furniture), services (custom content creation, land management), and land itself. The Marketplace is an external web-based storefront with over 2.1 million listed items. Because Linden Lab operates the currency exchange, they have ultimate authority over the economy, and changes to exchange rates, fees, or policies have real financial consequences for residents who depend on Second Life income.

Permissions and DRM

Second Life includes a built-in permissions system for user-created content. Each object has flags controlling whether it can be copied, modified, or transferred. A creator can sell a "no copy, no modify, transfer" item (the buyer gets one copy they cannot duplicate or alter but can resell) or a "copy, no transfer" item (the buyer can make copies for personal use but cannot give them away). This is basic DRM enforced by the server. It does not prevent screen capture or external copying, but it controls in-world distribution.

Content Pipeline

Residents create content in external tools (Blender, Photoshop, etc.) and upload to Second Life. Textures: TGA/PNG/BMP accepted, internally converted to JPEG 2000. Max resolution 1024px (standard) or 2048px (high-res). Meshes: COLLADA (.dae) format, validated through MAV, with automatic LOD generation and physics shape extraction. Scripts: Written in LSL (or the new SLua beta), compiled to bytecode on the server, executed in the simulator's VM. All assets get a UUID and are stored in the asset database, delivered to viewers via CDN.

Scaling & Performance

The Region-Per-Process Constraint

Second Life's fundamental scaling unit is the region. Each 256x256m region is one process. You cannot split a region across multiple processes or merge multiple regions into one process. This means a region's capacity is limited by what one CPU core can simulate. If 80 avatars with complex attachments and hundreds of scripted objects converge on a single region, that region's sim will hit time dilation regardless of how powerful the underlying hardware is.

Historically, Linden Lab ran 4 sims per physical server. On AWS, the mapping between sim processes and virtual hardware is more flexible, but the one-sim-one-process constraint remains. The AWS migration gave sims access to newer, faster CPUs than the aging on-premises hardware, which improved baseline performance, but it did not change the architecture.

Time Dilation

When a sim cannot maintain its 45 fps target, it reduces the simulation rate rather than dropping frames. A sim at 0.5 time dilation is running at half speed: avatars move at half pace, scripts execute half as many instructions per real-time second, and physics objects move in slow motion. This preserves determinism (all events still happen in the correct order) but makes the world feel sluggish. Residents monitor time dilation as a health indicator for any region they visit.

Script Performance

Every LSL script in a region shares the sim's processing budget. A region with thousands of scripted objects (common in shopping areas and clubs) will perform worse than a region with minimal scripting. Script time is reported in the sim statistics: if scripts are consuming more than 50% of frame time, the region will struggle. Poorly optimized scripts (tight loops, excessive timer events, heavy use of llSensor) are a constant source of performance problems.

Because LSL executes server-side, there is no way for a resident to run computations on their own hardware to offload the sim. Every scripted door, every animated texture, every particle emitter consumes server-side resources. This is the opposite of the client-side scripting model used by most modern game engines.

Concurrency Trends

Second Life's peak concurrency has been on a gradual decline. The all-time high was during the COVID-19 pandemic in 2020, when daily peak concurrency stayed around 60,000. By 2024, the highest recorded concurrency was 53,016 (February 2024). The platform has not exceeded 50,000 since early 2024. Monthly active users sit around 500,000 as of late 2024, down from over 1 million in 2007.

Technical Deep Dives

LSL (Linden Scripting Language)

LSL is a state-and-event-driven language with C-like syntax. Every script has one or more states, and each state contains event handlers. When an event occurs (a touch, a collision, a timer tick, an incoming message), the corresponding handler executes. Scripts transition between states using the state keyword. A door script might have an "open" state and a "closed" state, with touch events triggering transitions between them.

LSL is strongly typed and compiles to bytecode that runs in a virtual machine on the simulator. The original LSL1 was BASIC-based and used during the alpha phase (when the platform was called LindenWorld). LSL2, the current version, was written by Cory Linden sometime after 2002 and has been extended by many developers since then. Scripts communicate with each other through llSay/llListen (chat channels), llMessageLinked (within an object), and HTTP requests (external communication).

Linden Lab is now beta-testing SLua, a Luau-based scripting language that will eventually complement (not replace) LSL. SLua brings modern language features like proper string manipulation, closures, and better performance. Existing LSL scripts continue to work everywhere, and no viewer changes are needed to interact with SLua-powered content.

Havok Physics Integration

Second Life uses the Havok physics engine (currently version 2011.2, previously Havok 4 since April 2008). All physics calculations happen server-side in the simulator process. Havok simulates gravity, collisions, friction, and constraints for all physical objects in a region.

To conserve processing power, Havok freezes stable objects. If an object rests on the floor and does not move beyond a minimal threshold for about two seconds, physics calculations are disabled for that object. It remains frozen until another actively physical object collides with it. This optimization is critical because a region might contain thousands of objects, but only a handful are moving at any given time.

Havok is a commercial product that Linden Lab licensed for several hundred thousand dollars. This dependency means that if Linden Lab ever wanted to open-source the simulator, Havok could not be included. While open-source physics engines exist, Second Life's deep integration with Havok makes substitution effectively impossible without rewriting the physics layer.

OpenSimulator Comparison

OpenSimulator (OpenSim) is an open-source project founded in 2007 that implements a compatible server for Second Life viewers. It is written in C# (running on Mono) and licensed under BSD. OpenSim grids allow cross-grid visits through the Hypergrid protocol. The main advantages over official Second Life are dramatically lower land costs (often 10x more land for 10x less money), custom user management, larger region sizes, and features like personal backups.

However, OpenSim lacks Second Life's built-in DRM permissions system, which means creators who rely on copy/modify/transfer protections cannot port their business models to OpenSim. OpenSim's proposed architecture moves asset control from grid owners to content owners to enable cross-world asset migration, but this is still evolving.

Sansar: The Failed Successor

In 2017, Linden Lab launched Sansar, a VR-focused virtual world intended as Second Life's next-generation replacement. Sansar was built cloud-native on AWS, using Amazon ECS (Elastic Container Service) for nearly all backend services. Unlike Second Life's region grid, Sansar used instanced "experiences" that spun up on demand.

Uptake was low. Linden Lab halted Sansar development in 2020 to refocus on Second Life. The rights to Sansar's assets were sold to Wookey Search Technologies. However, the work on Sansar was not wasted from an infrastructure perspective: the lessons learned about running on AWS directly informed Second Life's own cloud migration, which began shortly after Sansar's launch and completed in January 2021.

Architecture Evolution

2001 LindenWorld alpha. The platform begins as a physics sandbox. LSL1 (BASIC-based) is the first scripting language. Architecture based on per-region server processes from the start.
2003 Second Life public launch. LSL2 (C-like syntax) replaces LSL1. The grid runs on Class 2 servers (1 CPU, 1 sim per box) in San Francisco and Dallas co-location facilities.
2006 Open-source viewer released. Linden Lab open-sources the viewer code, leading to third-party viewers like Phoenix and later Firestorm. Server code remains proprietary.
2007 Peak hype and OpenSimulator founded. Media coverage peaks. Over 1.1 million monthly active users. OpenSim begins as an open-source server alternative. Class 4 hardware introduced (4 sims per server).
2008 Havok 4 physics deployed. Replaces the original physics engine with Havok 4 across all simulators. Voice chat (Vivox) integration goes live. Backbone presence service (Python) introduced.
2010 Mesh import support added. Residents can upload COLLADA mesh models for the first time, moving beyond primitive-based building. Backend services begin migrating from custom C++ to web services.
2013 $3.2 billion in cumulative transactions. Linden Lab reports 10 years of economic activity. Terms of service change grants Linden Lab broader rights to user-generated content, causing controversy.
2017 Sansar public beta launches. VR-focused successor built on AWS ECS. Linden Lab begins planning Second Life's own cloud migration based on Sansar infrastructure lessons.
2020 COVID pandemic peak and Sansar shutdown. Concurrency hits ~60,000. Sansar development halted to refocus on SL. Cloud migration accelerates. Region transitions to AWS begin in earnest.
2021 Full AWS migration completed (January). All regions, services, and databases now run on Amazon Web Services. Older on-premises hardware decommissioned. Performance improves due to newer AWS hardware.
2024 WebRTC voice migration begins. Transition from proprietary Vivox voice to open WebRTC standard. SLua (Luau-based scripting) enters beta testing alongside LSL. Monthly active users at ~500,000.
Cloud Migration: What Changed

The AWS migration replaced aging on-premises hardware with modern cloud instances, improving baseline sim performance. It also removed the burden of hardware procurement, testing, and commissioning from Linden Lab's operations team. The migration was remarkably smooth for a 17-year-old platform, with only minor hiccups around teleport handling and group chat. What it did not change was the fundamental architecture: regions are still one-process-per-sim, region crossings still involve full state serialization between processes, and the grid topology is unchanged. The cloud made Second Life more reliable and easier to maintain, not architecturally different.
