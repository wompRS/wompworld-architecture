Networking Middleware

These are libraries and frameworks that handle the actual transport of game data between clients and servers. They sit between your game logic and raw sockets, providing abstractions for things like reliable delivery, state synchronization, matchmaking, and room management. Picking the right one depends on your engine, your scale target, and whether you want someone else managing the infrastructure.

Photon Server / Photon Cloud (Exit Games)

What it does: Photon is the most widely used networking middleware in the game industry, built by Exit Games (based in Hamburg, Germany). It provides a complete networking stack for multiplayer games: connection management, room-based matchmaking, state synchronization, RPCs, and relay servers. Photon comes in two flavors. Photon Cloud is a fully managed SaaS where Exit Games runs the servers for you. Photon Server is the self-hosted version where you run the same core on your own machines.

How it works: The core of Photon is written in C++ for raw performance, but the application layer is C#. This means your server-side game logic is written in C#, and if your client is also C# (Unity), your entire project can share code between client and server. Photon uses UDP by default for game traffic with its own reliable-UDP protocol on top, falling back to WebSocket for web clients. The architecture is room-based: players join rooms, and the server relays or processes events within each room. For server-authoritative games, you write Photon Server plugins or use Photon Fusion, which adds tick-based simulation, client-side prediction, and lag compensation.

Products in the Photon family:

  • Photon Realtime / PUN 2 - Relay-based multiplayer. Good for co-op, casual competitive. One client is the "host" for game logic, Photon just forwards messages.
  • Photon Fusion 2 - Tick-based netcode with client prediction and server authority. Built for action games. Supports both shared (client-hosted) and dedicated server modes.
  • Photon Quantum - Deterministic lockstep simulation. All clients run identical simulation. Best for fighting games, RTS, sports. Extremely bandwidth-efficient.
  • Photon Server - Self-hosted version. You run the C++ core on your own hardware. Full control over the infrastructure and application logic.

Pricing: Photon uses CCU-based pricing (Concurrently Connected Users). Their rule of thumb: 1 CCU roughly equals 20 DAU or 400 MAU. The free tier gives you 100 CCU, enough for prototyping and small-scale testing. The 500 CCU Growth plan costs $125/month. 2,000 CCU runs $500/month. Above 2,000 CCU, you move to Premium Cloud at roughly $0.50 per CCU per month. Stumble Guys, one of the most-downloaded mobile games of 2022, was paying around $25,000/month for 50,000 CCU while serving 20 million monthly players. Each tier bundles about 3 GB of traffic per CCU.

Which of our 18 games use it: Albion Online is the flagship Photon Server deployment. Their entire server backend runs on Photon Server with C# application code on top. Albion chose Photon specifically because of the C++ core / C# app layer split, which let them write server and client in the same language while getting native performance for networking.

Photon vs. Roll Your Own

For an indie MMO like Wompworld, Photon Cloud makes sense for session-based gameplay (instanced PvP arenas, matchmade battles). For the persistent open world with hundreds of players per zone, Photon Server (self-hosted) gives you the control you need, but you are writing all the zone management, world partitioning, and persistence logic yourself. Photon handles the transport layer and room management, not the MMO architecture. Albion built all of their cluster management, economy systems, and zone transitions on top of Photon Server.

Mirror / FishNet (Unity Open Source)

What they are: Mirror and FishNet are the two leading open-source networking libraries for Unity. Both are free, both are community-driven, and both exist because Unity deprecated their built-in networking (UNET) in 2018 and left a vacuum that took years to fill.

Mirror is the direct successor to UNET. It started as a fork and has been battle-tested for over 10 years. Mirror supports UDP, TCP, WebSockets, and Steam transport. It includes spatial hashing for interest management, synced variables, remote procedure calls, automatic message batching, and built-in Transform and Physics synchronization. It ships allocation-free at runtime (except for transports). Mirror is used by thousands of Steam games, including Population: ONE (acquired by Meta in 2021) and Zooba by Wildlife Studios, which hit 100+ million downloads on Google Play.

FishNet (Fish-Networking) is the newer challenger. It was built from scratch rather than forked from UNET, which gave it a cleaner architecture. FishNet includes built-in client-side prediction and lag compensation out of the box, which Mirror does not have natively. It also provides more granular control over bandwidth, scene management, and object spawning. FishNet is gaining adoption fast, particularly among developers building action games where prediction and rollback matter.

Pricing: Both are free and MIT-licensed for commercial use. FishNet has a paid Pro tier with additional features.

Limitations at scale: Neither Mirror nor FishNet is designed for MMO-scale persistent worlds. They work well for games with rooms of up to a few hundred players, but they do not provide world partitioning, cross-server communication, database integration, or the kind of orchestration you need for thousands of concurrent players in a shared world. For an MMO, you would use Mirror or FishNet as the transport/sync layer within a single zone, then build all the multi-zone architecture yourself on top.

Which of our 18 games use them: None of the 18 reference games use Mirror or FishNet directly. These libraries are primarily used by indie and mid-tier Unity games. Large-scale MMOs typically use either commercial middleware (Photon) or custom networking stacks.

For Wompworld

If the project uses Unity, Mirror or FishNet could serve as the intra-zone networking layer, but you would still need to build the MMO-scale systems (zone transitions, world partitioning, persistent state, cross-zone communication) yourself. They are not a substitute for a game server framework.

Nakama (Heroic Labs)

What it does: Nakama is an open-source game server by Heroic Labs that handles the social and operational infrastructure of online games. It provides user accounts, authentication, friends lists, groups/clans, chat, matchmaking, leaderboards, tournaments, notifications, in-app purchase validation, and data storage. It also supports real-time multiplayer with both relayed and server-authoritative models.

How it works: Nakama is written in Go for performance and can be extended with custom server logic in Go, Lua, or TypeScript/JavaScript. The runtime executes your custom code inside the server process. For real-time multiplayer, Nakama supports authoritative match loops that tick at a configurable rate (up to 30Hz), where the server validates all player input and broadcasts state. The server manages CPU scheduling and memory allocation per match, distributing load across a single instance or a cluster.

The matchmaker is flexible: you can define required, desired, and optional characteristics for both players and matches. In 2025, Heroic Labs added a MatchmakerProcessor hook that gives developers access to the full ticket pool for custom matching logic, plus party listing APIs and matchmaker stats.

Deployment: Nakama can run on any cloud provider (AWS, Azure, GCP, DigitalOcean) or your own hardware. Heroic Labs also offers a managed cloud service called Heroic Cloud for teams that do not want to run their own infrastructure. Integration with Edgegap provides on-demand low-latency server deployment for the game simulation layer.

Pricing: The server is licensed under Apache 2.0, so self-hosting is free. Heroic Cloud (managed hosting) has paid tiers. Enterprise licenses with priority support and SLAs are available.

Which of our 18 games use it: None of the 18 directly, but Nakama is widely used by mid-tier studios for mobile and PC multiplayer games. It handles the same category of problems that PlayFab and EOS solve, just without vendor lock-in.

When to Use Nakama vs. Custom

Nakama excels when you need a proven, self-hostable solution for social features, matchmaking, and lightweight multiplayer. It does not replace a dedicated game server for high-frequency simulation (like naval combat at 20+ Hz tick rates). For Wompworld, Nakama could handle accounts, friends, chat, matchmaking, and leaderboards, while a separate game server handles the actual ship simulation and combat.

Colyseus

What it does: Colyseus is an open-source multiplayer framework for Node.js. It provides a room-based architecture with automatic state synchronization, matchmaking, reconnection handling, and an authoritative server model. If your game server needs are modest and you are comfortable with TypeScript, Colyseus gets you from zero to multiplayer in hours rather than weeks.

How it works: You define Room classes that contain game logic. Each room has lifecycle methods: onCreate, onJoin, onLeave, onDispose, plus a message handler for client input. You define your game state as a schema object on the server, and Colyseus automatically synchronizes it to all connected clients using binary delta encoding. The transport layer uses WebSockets. Serialization uses a custom binary delta format for state and MessagePack for everything else.

The server is authoritative: game logic runs on the server, clients send inputs, the server processes them and broadcasts the updated state. Colyseus supports client-side prediction patterns where you apply the same movement logic locally for responsiveness, then reconcile when the server's authoritative state arrives.

Scaling: Colyseus can scale from 10 to 10,000+ concurrent users by adding Redis for cross-process room coordination and load balancers for distributing connections. For larger scale, you would run multiple Colyseus processes behind a reverse proxy.

Client SDKs: Available for web (JavaScript/TypeScript), Unity (C#), Defold, Haxe, Cocos, and Construct 3.

Pricing: MIT licensed, free for commercial use.

Which of our 18 games use it: None. Colyseus is primarily used for web-based multiplayer games, mobile games, and smaller-scale real-time projects. Its Node.js single-threaded model is not suited for the kind of high-frequency physics simulation that an MMO with naval combat requires.

Best Fit

Colyseus is excellent for web-based games, game jams, prototypes, and projects where development speed matters more than raw server performance. For Wompworld, it is too lightweight. The single-threaded Node.js runtime would bottleneck on the physics, AI, and world simulation that a naval MMO demands.

Backend-as-a-Service

These platforms provide pre-built backend services for games: player accounts, matchmaking, leaderboards, economy management, analytics, and sometimes server hosting. They let you skip building commodity backend systems so you can focus on gameplay. The tradeoff is always vendor lock-in vs. development speed.

SpacetimeDB (Clockwork Labs)

SpacetimeDB deserves a longer treatment because it represents a genuinely different approach to game server architecture. Instead of the traditional stack (game client connects to game server, game server talks to database), SpacetimeDB collapses the game server and database into a single system. Your game logic runs inside the database. Clients connect directly to the database. There is no separate server process.

The "Database is the Server" Paradigm

In a traditional multiplayer game, you have three layers: clients, game servers, and databases. The game server sits in the middle, receiving player input from clients, running game logic, reading and writing state to the database, and sending updates back to clients. You write networking code, serialization code, ORM code, persistence code, and synchronization code. You deploy and manage server processes, database clusters, and the plumbing between them.

SpacetimeDB eliminates the middle layer. You write your game logic as a "module" that runs directly inside the database. The module defines tables (your game state) and reducers (your game logic). Clients connect to the database over WebSockets, call reducers to perform actions, and subscribe to queries to receive real-time state updates. The database handles persistence, synchronization, and transport.

This means you do not write networking code (SpacetimeDB handles the WebSocket connections and message routing). You do not write serialization code (SpacetimeDB uses its own binary format, BSATN). You do not write ORM code (your tables are native to the database, accessed directly from your module). You do not write persistence code (the database persists everything automatically via a write-ahead log). You do not deploy servers (you upload your module with spacetime publish and it runs inside the database).

How Modules Work

A module is your server-side application compiled to WebAssembly (for Rust, C#, and C++ modules) or bundled as JavaScript (for TypeScript modules). You upload it to a SpacetimeDB instance, and it runs inside the database process. A module contains:

  • Tables - Define your game state as relational tables. Player positions, inventory items, chat messages, terrain data. SpacetimeDB holds all table data in memory for sub-microsecond access and persists it to a write-ahead log (WAL) for durability.
  • Reducers - Functions that modify game state. When a player wants to move, attack, or craft an item, the client calls a reducer. Reducers execute transactionally: either all changes commit or none do. If a reducer fails (e.g., permission check fails), the entire transaction rolls back. This gives you atomicity for free.
  • Scheduled Reducers - Reducers that fire on a timer, used for game loops, NPC AI ticks, resource respawn timers, and similar periodic logic.

Modules can be written in Rust, C#, TypeScript, or C++. Rust and C++ modules compile to WebAssembly and run in a sandboxed WASM runtime inside the database. TypeScript modules run in a JavaScript engine. The 2.0 release added Procedure Functions, which let modules make HTTP calls to external APIs from within a reducer.

BSATN Wire Format

BSATN (Binary SpacetimeDB Algebraic Type Notation) is SpacetimeDB's binary serialization format. It is used for all data interchange between the database and clients. You never interact with it directly. The client SDKs and the module compilation toolchain handle encoding and decoding automatically. BSATN is a compact, typed binary format designed for the kind of high-frequency, small-message traffic that games produce.

The Subscription Model

Clients subscribe to data using SQL queries. Each subscription defines what subset of the database the client cares about. When data matching a subscription changes (because a reducer modified it), the client receives the delta automatically. The client SDK maintains a local cache of subscribed data that can be queried without network round trips.

This is the key insight: you do not write code to decide what to send to which client. You do not build update packets. You do not implement interest management manually. The client says "I care about all players within 500m of me" as a SQL subscription, and SpacetimeDB handles the rest. When a player moves into or out of range, the subscription automatically adds or removes them from the client's local cache.

In SpacetimeDB 2.0, confirmed reads are enabled by default, meaning the server waits until a transaction is confirmed durable before sending updates to clients. Any data a client receives will survive a server restart.

Client SDKs

SDKs are available for C# (Unity), C++ (Unreal Engine with Blueprint support), TypeScript (with framework integrations for React, Vue, Angular, and Svelte), and Rust. The SDKs handle WebSocket connections, authentication, subscription management, local caching, and change callbacks. In a Unity game, you subscribe to data, and your UI updates automatically when the data changes.

BitCraft MMO

BitCraft Online is the flagship project built on SpacetimeDB, developed by Clockwork Labs (the same company that builds SpacetimeDB). BitCraft is a community sandbox MMORPG with crafting, city-building, exploration, and survival in a single seamless shared world. The entire backend is a single SpacetimeDB module. There are no other servers or services. Everything: chat messages, items, resources, terrain, player locations, all of it is stored, processed, and synchronized by SpacetimeDB. BitCraft launched on Steam alongside SpacetimeDB 1.0 in June 2025.

Performance

SpacetimeDB 2.0 claims over 100,000 transactions per second for TypeScript modules and up to 170,000 transactions per second for Rust modules. All data lives in memory for sub-microsecond read access. The WAL provides durability without the latency cost of synchronous disk writes on every transaction.

License and Pricing

SpacetimeDB is source-available under the Business Source License 1.1 (BSL). This is not technically open source. It converts to AGPL v3.0 with a linking exception after a set period. The linking exception means you do not have to open-source your own game code; you only have to contribute back changes to SpacetimeDB itself. You can self-host for free. Clockwork Labs also offers Maincloud, their managed hosting service, which uses an "Energy Packs" credit system for pricing.

What SpacetimeDB Eliminates

  • Networking code (WebSocket transport is automatic)
  • Serialization code (BSATN handles it)
  • ORM / database access code (tables are native)
  • Persistence code (WAL handles durability)
  • Server deployment (upload module, done)
  • State synchronization code (subscriptions handle it)

What SpacetimeDB Does Not Solve

  • Client-side rendering and prediction
  • Physics simulation (runs server-side in your module)
  • Asset delivery and patching
  • Voice chat
  • Anti-cheat beyond server authority
  • World partitioning across multiple SpacetimeDB instances

When SpacetimeDB Makes Sense vs. When It Does Not

Use it when: You want to move fast with a small team. You are building a game where the server is authoritative and the state model fits cleanly into relational tables. You do not want to deal with server deployment, database management, or writing networking code. The BitCraft model (sandbox MMO with crafting, building, exploration) maps well to SpacetimeDB's strengths.

Be cautious when: You need world partitioning across multiple server instances (SpacetimeDB modules currently run on a single instance). You need sub-frame physics simulation at 60Hz+ with tight deterministic requirements. You need to integrate deeply with existing infrastructure. You want a proven, battle-tested foundation with years of production history at massive scale (SpacetimeDB hit 1.0 in June 2025).

For Wompworld: SpacetimeDB is worth watching closely. The "database is the server" model would eliminate a huge amount of infrastructure work. The subscription model maps well to a naval game where players need real-time updates about nearby ships, projectiles, and world state. The risk is that the technology is young, the single-instance model may not scale to a full-server MMO with thousands of concurrent players, and the BSL license means you depend on Clockwork Labs for the long term. But if the performance claims hold and clustering support matures, this could be the difference between a team of 3 building an MMO backend and a team of 10.

PlayFab (Microsoft Azure)

What it does: PlayFab is Microsoft's backend platform for live games, built on Azure. It provides a broad set of managed services: player authentication (cross-platform, including Xbox, Steam, PlayStation, Nintendo, Facebook, Google, and Apple), player profiles, virtual economy (currencies, catalogs, stores, inventory), leaderboards, matchmaking, lobbies, real-time party chat, push notifications, content management, A/B testing, LiveOps event scheduling, analytics, and multiplayer server hosting. It is the Swiss Army knife of game backend services.

How it works: PlayFab runs as a set of REST APIs on Azure. Your game client or server calls PlayFab APIs to perform backend operations. For multiplayer server hosting, PlayFab provisions and manages dedicated server containers on Azure VMs, handling auto-scaling, region selection, and matchmaking placement. The economy system (Economy v2) is a rebuild on modern Azure infrastructure designed for hundreds of millions of active players with large inventories.

Pricing: Free in development mode (up to 100,000 users, 10 titles per account). After launch, pricing is consumption-based: you pay per event, per profile read/write, per API call to matchmaking/lobby/economy, per server VM-hour, and per analytics query. The free tier is generous enough for indie prototyping and beta testing. Costs scale with usage, which can become expensive at high DAU if you are not careful about API call frequency.

LiveOps features: PlayFab's strength is running games as a service. You can schedule timed events (double XP weekends, seasonal content), manage content remotely without client patches, run A/B experiments on economy balancing, segment players for targeted offers, and track engagement metrics in real-time.

Which of our 18 games use it: Minecraft uses PlayFab for its backend services (both games are Microsoft properties). Several of the reference games use Azure infrastructure that integrates with PlayFab's ecosystem.

For Wompworld

PlayFab could handle accounts, authentication, economy management, leaderboards, and LiveOps without building any of those systems from scratch. The free tier covers development and early access. The risk is Azure lock-in and consumption-based pricing that can surprise you at scale. For an indie MMO, the economy and LiveOps features are genuinely useful, but the multiplayer server hosting is designed for session-based games, not persistent worlds.

Epic Online Services (EOS)

What it does: EOS is Epic Games' suite of free, cross-platform backend services. Originally built for Fortnite and then opened to all developers. It provides player authentication (Epic Games accounts plus third-party identity providers), crossplay between platforms and storefronts, lobbies, matchmaking, peer-to-peer and relay networking, voice chat, player sanctions, player reports, anti-cheat (Easy Anti-Cheat, covered separately below), achievements, stats, leaderboards, and title storage.

How it works: EOS is engine-agnostic and platform-agnostic. You integrate the EOS SDK into your game (available for Unreal, Unity, and custom engines via C). The services communicate with Epic's backend infrastructure. For voice chat, connections are relayed through EOS backend servers with quality-of-service management, multi-region presence, and automatic scaling. For crossplay, EOS links player accounts across Steam, Epic Games Store, PlayStation, Xbox, and Nintendo.

Pricing: Free. No royalties, no hosting fees, no per-user charges. This is the single biggest differentiator. Epic subsidizes EOS to grow its ecosystem and compete with Steam. You can use any or all services without paying Epic anything.

Which of our 18 games use it: Fortnite is the origin and proving ground for EOS. Albion Online and Ashes of Creation use Easy Anti-Cheat, which is part of the EOS suite. Multiple AAA and indie titles use EOS for crossplay and matchmaking.

For Wompworld

Hard to argue against free. EOS could provide accounts, crossplay, voice chat, lobbies, matchmaking, and anti-cheat at zero cost. The catches: you are depending on Epic continuing to offer this for free, the SDK integration adds complexity, and you are pulling data through Epic's infrastructure. For an indie project, using EOS for voice chat and anti-cheat alone would save months of development.

Amazon GameLift (AWS)

What it does: GameLift is AWS's managed service for deploying, scaling, and operating dedicated game servers. You upload your game server binary, configure your fleet (instance types, regions, scaling rules), and GameLift handles provisioning, auto-scaling, matchmaking placement, health monitoring, and DDoS protection. It supports up to 100 million concurrent players and 100,000 player adds per second.

How it works: GameLift runs your game server binary on EC2 instances (or in containers as of recent updates). The auto-scaling algorithm predicts player demand and spins servers up or down to match. Standby servers keep wait times low. FlexMatch is the built-in matchmaking service: you define rule sets (skill-based, latency-based, team composition requirements), and FlexMatch groups players and feeds matches into a game session queue that selects the best available server based on each player's location.

Spot Instances: GameLift supports Spot Instances at 50-85% savings over on-demand pricing. It uses a proprietary algorithm to optimize placement for low latency, low cost, and minimal Spot interruption rates. Combined with Graviton (ARM) instances for another 20% price-performance improvement, and intelligent auto-scaling, GameLift advertises infrastructure costs as low as $1/month per player.

Pricing: Pay-as-you-go, billed per second of VM time. Linux instances are roughly half the cost of Windows instances (no licensing fee). FlexMatch is included free with GameLift hosting; standalone FlexMatch (for non-GameLift servers) is billed separately by player packages and matchmaking hours. GameLift supports 68 instance families across 26 regions and 9 Local Zones.

New World and GameLift: Amazon's own MMO, New World, did not use GameLift. This is important context. New World ran on custom EC2 infrastructure because GameLift is designed for session-based games (matches with a start and end time). MMOs with persistent worlds, continuous simulation, and world partitioning do not fit GameLift's session model. AWS's own documentation now acknowledges this: for large-scale MMOs requiring world partitioning, custom solutions on EC2, ECS, or EKS are a better fit than GameLift.

Which of our 18 games use it: None of the 18 reference games use GameLift directly (they are mostly persistent-world MMOs). GameLift is used by session-based multiplayer games. Fortnite uses AWS infrastructure but through custom EKS deployments, not GameLift.

For Wompworld

GameLift is not the right fit for a persistent-world MMO. It is built for games where matches start and end. If Wompworld had instanced PvP arenas or session-based game modes in addition to the open world, GameLift could host those instances while the persistent world runs on custom EC2/EKS infrastructure. FlexMatch could still be used as a standalone matchmaker regardless of where your servers run.

Multiplay (Unity)

What it does: Multiplay was Unity's dedicated server hosting platform. It provided scalable, engine-agnostic game server hosting with multicloud hybrid infrastructure, auto-scaling, zero-downtime patching, QoS-based player placement, matchmaker integration, server health monitoring, and war room launch support. It operated across 70+ data center regions.

Track record: Multiplay had over 20 years of operational history. It helped Respawn scale Apex Legends from zero to 50 million players in under 24 days, peaking at 2+ million concurrent players. It hosted servers for Rogue Company, Worms Rumble, MARVEL SNAP, and Divine Knockout, among others. Unity claimed 30-40% hosting cost reduction compared to rolling your own.

The catch: Unity announced the discontinuation of Multiplay support in December 2025. The last feature update shipped in August 2024. Studios using Multiplay are migrating to alternatives like Edgegap, Hathora, AWS GameLift, or self-managed Kubernetes clusters.

Which of our 18 games use it: None of the 18 directly, though some of the studios behind the reference games used Multiplay for other titles.

Status: Discontinued

Multiplay is shutting down. Do not build new projects on it. If you are evaluating Unity's multiplayer ecosystem, note that the server hosting product is gone while the networking libraries (Netcode for GameObjects, Netcode for Entities) continue to be developed. For dedicated server hosting, look at Edgegap, Hathora, GameLift, or self-managed Kubernetes.

Anti-Cheat Systems

Anti-cheat in online games is an arms race. Cheaters reverse-engineer game clients, inject code into processes, modify memory, use external tools to read game state, and employ hardware devices to gain unfair advantages. Anti-cheat systems try to detect and prevent all of this. The major players use kernel-level drivers that operate at ring 0 (the highest CPU privilege level) to monitor for tampering that user-mode software cannot see.

Easy Anti-Cheat (EAC) / Epic Games

What it does: EAC is the most widely deployed kernel-level anti-cheat in PC gaming, protecting over 200 games as of 2025. It was originally developed by Kamu (founded 2006), acquired by Epic Games in 2018, and integrated into the Epic Online Services suite as a free service in 2021.

How it works: EAC installs a kernel driver that operates at ring 0, the most privileged access level in the CPU. This gives it unrestricted access to system memory, running processes, and hardware configuration. It can monitor memory for unauthorized modifications, detect injected DLLs, scan for known cheat signatures, and analyze hardware fingerprints. The kernel-level access is necessary because modern cheats operate at the same privilege level to hide from user-mode detection. EAC scans continuously while a protected game is running.

Linux / Steam Deck: In 2021, EAC received native Linux and macOS support as part of the EOS integration. Proton/Wine compatibility was improved in collaboration with Valve for the Steam Deck launch. However, the Linux version runs in user mode (not kernel mode), and game developers must opt in to enable Linux support. Compatibility varies per game.

Pricing: Free through Epic Online Services. No per-user or per-game fees.

Which of our 18 games use it: Fortnite (Epic's own game, the original testbed). Ashes of Creation uses EAC. Albion Online used EAC from 2019 to 2023, then switched to BattlEye. Destiny 2 uses BattlEye, not EAC.

Controversies: Kernel-level access means EAC has the same privilege level as your operating system and hardware drivers. If the anti-cheat is compromised, attackers have full system access. Past incidents include the Genshin Impact anti-cheat being exploited by ransomware operators in 2022 to disable antivirus software. This is not specific to EAC, but it is the fundamental risk of any kernel-level approach.

BattlEye

What it does: BattlEye is a proprietary kernel-level anti-cheat developed by BattlEye Innovations (Reutlingen, Germany). First released in 2004 for Battlefield Vietnam, it has since been integrated into numerous major multiplayer games.

How it works: BattlEye describes itself as a "fully proactive kernel-based protection system." It runs both kernel-mode and user-mode scanning. The kernel driver monitors process creation, DLL loading, memory modifications, and system calls. It performs continuous signature scanning against known cheat patterns, detects custom drivers that try to access game memory, and blocks applications that use vulnerable kernel drivers for injection. BattlEye can terminate processes that attempt to hook into the protected game.

A 2014-2015 update added the kernel-mode driver to strengthen protection. BattlEye has stated the driver is not intended to function as a rootkit, though the access level is identical.

Linux / Steam Deck: BattlEye supports Proton/Steam Deck, but only in user mode, and only if the game developer explicitly opts in. Many developers have not opted in, which means Linux compatibility is inconsistent even for games that use BattlEye.

Windows 11 issue: Microsoft's Kernel-mode Hardware-enforced Stack Protection feature in Windows 11 can prevent the BattlEye driver from loading. BattlEye is working with Microsoft on a fix, but as of early 2026, this remains an active compatibility issue.

Which of our 18 games use it: Albion Online switched from EAC to BattlEye in July 2023. Destiny 2 uses BattlEye. ARK: Survival Evolved uses BattlEye. PlanetSide 2 uses BattlEye. In 2025, Rockstar added BattlEye to GTA Online.

RICOCHET Anti-Cheat (Activision)

What it does: RICOCHET is Activision's proprietary, custom-built anti-cheat system for Call of Duty. Unlike EAC and BattlEye, which are third-party products licensed to many games, RICOCHET was developed internally and is exclusive to the Call of Duty franchise. It operates across multiple layers, making it one of the most comprehensive anti-cheat systems in gaming.

How it works (multi-layer model):

  • Layer 1: TPM 2.0 hardware verification - A hardware security module built into the CPU or motherboard that verifies the PC's boot process has not been tampered with. TPM 2.0 creates a hardware-backed attestation of system integrity that is difficult to spoof.
  • Layer 2: Secure Boot enforcement - Ensures only trusted, signed software loads during the Windows boot process. Together with TPM 2.0, this prevents cheats from loading before the operating system starts.
  • Layer 3: Cloud-based attestation - Added in early 2026, this uses Microsoft Azure Attestation to verify PC integrity remotely before a match begins. Activision calls it the most robust attestation technology in gaming.
  • Layer 4: Kernel-level driver - A custom kernel driver that monitors processes interacting with Call of Duty while the game runs. It activates when the game launches and deactivates when it closes.
  • Layer 5: Server-side analytics and in-game mitigations - Backend systems that analyze player behavior patterns, detect statistical anomalies, and apply in-game mitigations. This includes detection of hardware cheat devices like Cronus Zen and XIM Matrix.

TPM 2.0 requirement: As of Black Ops 6 Season 5 (August 2025), TPM 2.0 and Secure Boot are enforced in a phased rollout. They are required for Black Ops 7 and Warzone. This is controversial because it requires players to enable specific BIOS/UEFI settings, and it completely prevents Linux/SteamOS play.

Which of our 18 games use it: Call of Duty: Warzone is the only reference game using RICOCHET.

Valve Anti-Cheat (VAC)

What it does: VAC is Valve's automated anti-cheat system, integrated into Steam since 2002. Unlike the kernel-level solutions above, VAC historically operated in user mode, scanning process memory for known cheat signatures while a game is running. It is the oldest continuously operating anti-cheat system in gaming.

How it works: VAC uses signature-based detection, scanning a player's system memory for patterns that match known cheat programs. It does not scan hard drives or personal files. When a match is found, the account is flagged but not immediately banned. VAC's signature feature is the delayed ban: after detection, bans are held for days or weeks before enforcement. This serves two purposes. It prevents cheat developers from quickly identifying which detection method caught them. And it lets the cheat spread through the cheating community before the ban wave hits, catching more users.

VAC Live (2023-2025): A newer system that provides real-time detection and immediate bans during active gameplay. VAC Live uses machine learning and more sophisticated detection than the original signature-based system.

VACnet: A deep learning system deployed since 2017 that analyzes player behavior (aim patterns, movement, statistical anomalies) and flags suspicious accounts for review. VACnet does not ban directly; it feeds data into the Overwatch community review system and the VAC detection pipeline.

Trust Factor: A behavioral scoring system that influences matchmaking quality. Players with high Trust Factor get matched with other high-Trust players; low Trust Factor results in longer queues and more problematic matches. Trust Factor considers account age, game library, play history, report frequency, and VAC/ban history.

Consequences: VAC bans are permanent and non-negotiable. Banned accounts are publicly marked. Phone-number-linked accounts can trigger cross-account bans. Family Sharing privileges are revoked.

Which of our 18 games use it: VAC is specific to Steam games. None of the 18 reference games use VAC as their primary anti-cheat (most use EAC or BattlEye), though games on Steam benefit from VAC as a secondary layer.

EAC through EOS is the obvious choice for an indie project: it is free, widely deployed, and integrates with the rest of the EOS suite. The most important anti-cheat measure for an MMO is server authority: if the server validates all actions, the impact of client-side cheating is limited to information leaks (wallhacks, radar) rather than game-breaking exploits (speed hacks, item duplication).

Infrastructure

How you host and orchestrate your game servers matters as much as how you write them. The infrastructure layer determines your operational costs, your ability to scale for launch day, your latency to players around the world, and how many 3am pages your ops team gets when something breaks.

Kubernetes and Docker for Game Servers

What they do: Docker packages your game server binary and all its dependencies into a portable container that runs identically across any environment. Kubernetes orchestrates those containers across a cluster of machines, handling scheduling, scaling, health monitoring, networking, and restarts. Together, they let you deploy and manage hundreds or thousands of game server instances without manually provisioning individual machines.

Why Containers for Game Servers

Game servers are a natural fit for containers. Each game server instance is a relatively short-lived process (minutes to hours for session-based games, longer for persistent worlds). It needs a fixed set of dependencies. It should be isolated from other instances on the same machine. And you need to spin up many of them quickly when player demand spikes. Docker containers provide lightweight isolation, fast startup (seconds rather than minutes for VMs), consistent environments, and efficient resource utilization through bin-packing (running multiple containers on one machine).

How Fortnite Uses Kubernetes (EKS)

Epic Games runs Fortnite on Amazon EKS (Elastic Kubernetes Service). Their architecture is microservices-based: different components of the game backend run as separate containerized services orchestrated by Kubernetes. Key Kubernetes features they use include Horizontal Pod Autoscaling (HPA) for dynamic resource allocation, self-healing through automated pod restarts and rescheduling, rolling updates and canary deployments for seamless version transitions, service discovery and load balancing, and secrets management. Epic monitors the infrastructure with Prometheus, FluentD, InfluxData, and Telegraf.

Agones and Open Match

Google developed Agones, an open-source Kubernetes extension specifically for game server orchestration. Agones handles game server lifecycle management (allocating, ready, running, shutdown), fleet management (maintaining a pool of ready servers), and fleet autoscaling. AWS published a reference architecture for running Agones and Open Match on EKS, with AWS Global Accelerator for edge entry, Shield for DDoS protection, and CloudWatch for monitoring.

Open Match is an open-source matchmaking framework (also from Google) that integrates with Agones. It handles the player matching logic, while Agones handles the server allocation.

Cost Optimization Patterns

  • Spot/Preemptible Instances: Run game server containers on Spot Instances for up to 90% cost reduction. Containers start fast enough to handle the 2-minute Spot interruption notice gracefully. The game server drains players to another instance before shutdown.
  • Graviton/ARM Instances: ARM-based processors (AWS Graviton, Azure Arm) deliver up to 40% better price-performance for network-heavy workloads like game servers.
  • Container Packing: Configure resource limits per container to pack multiple game server instances onto a single machine, maximizing utilization.
  • Serverless Containers: AWS Fargate and Google Cloud Run abstract away the cluster entirely. You just deploy containers and pay per-second of compute. Good for bursty, session-based workloads.

When Kubernetes Makes Sense for Games

  • Session-based games with variable player counts
  • Games that need rapid horizontal scaling
  • Teams with DevOps expertise or willingness to learn
  • Multi-region deployments requiring automated orchestration
  • Microservices architectures with many independent components

When Kubernetes Is Overkill

  • Single-server persistent worlds (just run on EC2/bare metal)
  • Small teams with no ops experience
  • Prototyping and early development
  • Games with fewer than 1,000 concurrent players
  • Projects where managed services (GameLift, Multiplay) are a better fit
For Wompworld

Kubernetes is the right call if and when the game reaches the scale where you need to run many zone servers across multiple regions with auto-scaling. But it introduces significant operational complexity. For early development and alpha/beta, running game servers directly on a handful of dedicated machines or VMs is simpler and sufficient. Adopt Kubernetes when you actually need the orchestration, not because it sounds good on a tech stack slide.

Cloud Providers for Game Hosting

The three major cloud providers (AWS, Azure, GCP) all want your game hosting dollars. Each has game-specific services, different pricing models, and different strengths. Here is how they compare for multiplayer game hosting as of 2025.

AWS (Amazon Web Services)

Market share: ~29% of the global cloud market (Q3 2025). The largest cloud provider.

Game-specific services: GameLift (dedicated server hosting + FlexMatch matchmaking), Local Zones (ultra-low latency metro edge data centers), EKS (managed Kubernetes), and a broad game tech blog with reference architectures. AWS also published EKS Blueprints with pre-configured cluster patterns for game workloads.

Latency: Lowest global average latency of the three providers. Local Zones in cities like Dallas, Atlanta, and Los Angeles reduce ping further. Epic Games worked with AWS to deploy Fortnite's NA-Central region using Local Zones in Texas.

Pricing (bandwidth): ~$0.09/GB for the first 10 TB egress. Compute pricing is competitive, especially with Graviton (ARM) instances.

Notable games hosted: Fortnite (EKS), PUBG (partial), New World (EC2, now shutting down).

Microsoft Azure

Market share: ~20% of the global cloud market.

Game-specific services: PlayFab (the most comprehensive game backend platform), Azure Multiplayer Servers (integrated with PlayFab), and strong integration with Xbox Live services. Azure has the most complete package for game-as-a-service operations.

Latency: Good with multi-region deployment. Cross-US latency floor around 50-70ms, which stays under 50ms with strategic region placement (East, Central, West).

Pricing (bandwidth): ~$0.05/GB, the cheapest of the three for bandwidth-heavy games. ARM instances offer the largest price gap between x86 and ARM of any provider (65% savings on-demand).

Notable games hosted: Minecraft, Halo, Sea of Thieves, PUBG (partial). All Microsoft first-party titles run on Azure.

Google Cloud Platform (GCP)

Market share: ~13% of the global cloud market.

Game-specific services: Agones (open-source game server orchestration for Kubernetes) and Open Match (open-source matchmaking). GCP's game tools are open-source, which means lower vendor lock-in. You can run Agones and Open Match on any Kubernetes cluster, not just GKE.

Latency: Excellent private backbone between regions. Fewer total regions than AWS, but the internal network provides low-hop routing. GCP does not have a mainland China region, which is a dealbreaker for games targeting the Chinese market.

Pricing (bandwidth): ~$0.08-$0.12/GB, the most expensive for bandwidth. Compute pricing is competitive, especially for Spot/preemptible instances.

Notable games hosted: Apex Legends (primary), Pokemon GO.

For Wompworld

For an indie MMO, the choice between cloud providers comes down to: what services do you want to use, and where are your players? If you want PlayFab's LiveOps and economy tools, Azure is the natural fit. If you want the broadest infrastructure with the most game-specific reference architectures, AWS is the default. If you want to avoid vendor lock-in and use open-source orchestration, GCP's Agones/Open Match stack is appealing. For the Wompworld prototype phase, any provider works. Pick based on what managed services you want to lean on, not on raw infrastructure pricing.

One practical note: bandwidth costs matter for MMOs. A naval game with hundreds of entities broadcasting position updates generates substantial egress traffic. At scale, Azure's $0.05/GB vs. GCP's $0.12/GB is real money. Budget for bandwidth from day one.
