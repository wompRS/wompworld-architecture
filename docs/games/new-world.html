Architecture Comparison / New World

New World

Cloud-Native Grid Open-World MMO

The first AAA MMO built from scratch on AWS cloud infrastructure, using a grid-based hub system with stateless EC2 instances, serverless microservices, and DynamoDB for state persistence. A case study in what happens when a cloud company builds a game.

Overview

New World is Amazon Game Studios' flagship MMO, set on the supernatural island of Aeternum. What makes it architecturally significant is that it was designed from day one as a cloud-native game running entirely on AWS. Werner Vogels, Amazon's CTO, showcased the game's architecture during his AWS re:Invent 2021 keynote, calling it "a MMORPG truly born in the cloud."

The game's world is divided into 14 territories across the island of Aeternum, mapped onto a grid of compute cells serviced by 7 stateless EC2 "hubs." Each world supports up to 2,000 concurrent players while simulating over 7,000 AI entities and hundreds of thousands of objects. The server ticks at 30 Hz, six times faster than the 5 Hz rate common in traditional MMOs, which enables New World's action combat system to feel responsive.

Non-gameplay services like character creation, company management, and the trading post run as serverless Lambda functions behind API Gateway. Game state writes to DynamoDB at a rate of roughly 800,000 writes every 30 seconds, and telemetry flows through Kinesis Data Streams at 23 million events per minute into S3 for analytics.

The result is a system that scaled from single test servers during development to over 500 production worlds at launch without code changes. It also produced one of the most dramatic population crashes in MMO history, dropping from 913,000 concurrent players to under 20,000 within three months.

Server Infrastructure

Every New World "world" (what players call a server) is actually a cluster of EC2 instances working together. The architecture breaks down into several layers.

Remote Entry Points (REPs)

Each world has exactly 4 EC2 C5.2xlarge instances serving as Remote Entry Points. These are the only public-facing IP addresses in the entire world cluster. REPs handle connection security, authentication, and resilience. If a player connects to New World, their traffic hits one of these four REPs first, which then routes them to the appropriate backend hub.

Game Hubs

Behind the REPs sit 7 stateless EC2 instances called "Hubs." Each hub runs the actual game simulation for a portion of Aeternum. The world map is overlaid with a grid, and each hub is responsible for 2 grid cells. Critically, these 2 cells are not adjacent. If you walk from grid cell 1 to grid cell 2 on the map, you transition from one hub to another. This interleaving pattern ensures that geographic hotspots (like a popular town) spread their load across different physical machines rather than hammering a single instance.

Because the hubs are stateless, they can be replaced instantly if one fails. All persistent state lives in DynamoDB, so a replacement hub can spin up, pull the latest state, and resume simulation without data loss.

Instance Pool

Session-based content like Expeditions (dungeons) and Outpost Rush (PvP battlegrounds) runs on a separate instance pool. These are dynamically provisioned EC2 instances that spin up when players queue for instanced content and release when the session ends. This keeps the open-world hubs focused on the persistent game state while instanced content scales independently.

Network & Protocols

New World's networking follows a server-authoritative model, though this was a point of significant controversy at launch. Amazon Game Studios explicitly stated that "from a simulation standpoint, New World is entirely server based." Clients send controller inputs to the server, the server validates those inputs against movement and ability limits, and only then processes them within the server-side simulation.

Client-Authority Controversy

Despite Amazon's claims, players discovered early exploits that suggested the client had more authority than it should. Movement speed hacks, invincibility exploits, and item duplication bugs circulated within the first weeks of launch. The community accused AGS of running a client-authoritative architecture, which Amazon denied. The reality was likely a hybrid model where certain client-side predictions were trusted more than they should have been, particularly around movement and collision.

Hit Detection

New World uses hitscan detection for many abilities rather than physics-based projectiles. This was not the original design. In the 1.0.1 patch, AGS converted turret projectiles in Territory Wars from physical projectiles to hitscan specifically to address performance issues. Spawning hundreds of physical projectile objects during a 50v50 war was tanking server performance, so they moved to a raycasting approach that requires far less simulation overhead.

The 30 Hz Gamble

New World runs its game simulation at 30 ticks per second. Traditional MMOs like World of Warcraft or EverQuest run at roughly 5-10 Hz. AGS chose 30 Hz because New World's action combat (dodge rolls, active blocking, aim-based attacks) demands faster state updates than tab-target games. The tradeoff is that simulating 2,500 players at 30 Hz across 7 hubs requires significantly more compute per world than a 5 Hz MMO would.

Data & Persistence

Amazon DynamoDB is the backbone of New World's persistence layer. DynamoDB's schemaless design suits game data well because game objects tend to have low coherency and variable structure. A player character, a dropped item, a settlement's upgrade state, and a faction's territory control data all have different shapes but live in the same database.

State Management

Because the game hubs are stateless, all meaningful game state must flow through DynamoDB. When you craft an item, the hub writes that state to DynamoDB. When you kill a mob, the loot table result and your XP gain write to DynamoDB. When a territory changes hands after a war, the new ownership state writes to DynamoDB. This means the database handles roughly 800,000 writes every 30 seconds per world cluster.

The advantage is resilience. If a hub crashes, it loses nothing. A replacement spins up, pulls the latest state from DynamoDB, and the world continues. Players on the failed hub may rubber-band or disconnect briefly, but their progress is safe.

Settlement Persistence

Settlements are persistent structures that companies (guilds) own and upgrade. Each settlement contains crafting stations (Forge, Workshop, Kitchen, etc.) that can be upgraded from tier 1 to tier 5. These tiers determine what items players can craft. Settlement state includes the tier of every crafting station, the current tax rates set by the owning company, the territory standing of every player who has gained reputation there, and the progress on active town projects.

When a territory is lost in a war, or when a Corrupted Invasion succeeds and downgrades crafting stations, those state changes propagate through DynamoDB and become visible to all players in the settlement within seconds.

Trading Post

The Trading Post is New World's player-driven marketplace, functioning like a commodities exchange with buy orders and sell orders. All trading posts across all settlements are linked into a single global marketplace per world. Players can post up to 100 active orders per character, with a maximum listing duration of 14 days. Fees depend on which settlement you post from (sell fees) and which settlement you buy from (purchase taxes).

The Trading Post runs as a serverless microservice through Lambda and API Gateway, decoupled from the game simulation. This means trading operations do not consume hub compute cycles. Order matching, fee calculation, and item transfer all happen in the serverless tier.

DynamoDB Scaling

One of the most impressive technical achievements was scaling DynamoDB from a single test server during development to over 500 production worlds at launch without code changes. DynamoDB's auto-scaling provisioned throughput handled the ramp from zero to hundreds of thousands of concurrent write operations without manual intervention. This is the kind of scaling that on-premise database infrastructure cannot match.

Scaling & Performance

Launch Scaling Crisis

New World launched on September 28, 2021, and hit 913,634 concurrent players on Steam within the first week. Amazon had provisioned several hundred worlds, but it was not enough. Queue times stretched into hours. Some servers had 20,000+ player queues. AGS scrambled to add more worlds, eventually reaching over 500 at peak.

The irony of Amazon's cloud company struggling with launch capacity was not lost on the community. The technical challenge was not raw compute (AWS has effectively infinite EC2 capacity), but rather the game's per-world player cap of 2,500. You cannot solve an MMO population surge by making individual servers bigger. You can only add more worlds, which fractures the player base across parallel instances of the same game.

50v50 Territory Wars

Territory Wars pit two groups of 50 players against each other in a siege-style battle for control of a territory. The attacker must capture the defender's fort, using siege weapons like cannons and repeaters. These 100-player battles represented the highest-density combat the game could produce, and they routinely caused severe performance problems.

The main issue was AoE (Area of Effect) ability spam. When 100 players cluster on a single capture point, the server has to simulate dozens of overlapping AoE effects, each checking collision against 100 players, every tick. At 30 Hz, that is an enormous computational load concentrated in a single grid cell on a single hub.

Players discovered they could intentionally crash wars by coordinating mass AoE spam. The defending faction would dump every AoE ability on the point simultaneously, tanking the server's tick rate until the war instance crashed. This "war lag" exploit flipped territories across many servers before AGS addressed it.

World Merges

As the population crashed from 913K to under 50K within months, AGS faced the opposite problem: too many empty worlds. Their world merge system transfers all player data (gold, items, companies, housing) from smaller worlds into a larger host world. The only data that does not transfer is territory ownership, which stays with the host world's existing factions. This means merged players lose their territory claims and have to compete for them again on the host world.

AGS performed multiple rounds of merges through 2022 and 2023, consolidating hundreds of worlds down to a much smaller number. The merge process itself was handled through DynamoDB data migration, moving player records from one world's partition to another.

World Sets and Transfers

New World introduced "World Sets," clusters of worlds in the same region that allow character transfers between them. A player can transfer their character from one world to another within the same set, with cooldowns to prevent abuse. This system runs through the serverless tier and involves migrating the player's DynamoDB records to the target world's data partition.

Technical Deep Dives

Faction Control System

Three factions (Marauders, Syndicate, Covenant) compete for control of Aeternum's territories. A company claims a territory by paying 100,000 gold at its fort. Faction control is tracked server-side through influence counters. PvP missions in a territory generate influence for the attacking faction, and when influence reaches 100%, the territory enters a Conflict state. A lottery then selects which attacking company leads the war declaration.

The Influence Race system runs daily at fixed server times (6 PM, 7 PM, 8 PM local). Each race lasts 45 minutes. The attacking faction must earn enough influence through PvP missions, Influence Towers, and Forts before time expires. An underdog mechanic grants bonuses to the weakest faction to prevent permanent domination by a single group.

Corrupted Invasions

Every 4 days, each territory triggers a Corrupted Invasion: a PvE siege where waves of AI enemies attack the settlement's fort. Up to 50 players defend. The territory's Governor selects 10 "Hero" slots, and the remaining 40 defenders are randomly drawn from registered players. If the invasion succeeds, crafting stations in the settlement are downgraded, forcing players to spend resources rebuilding them.

The invasion system creates a persistent tension between PvE maintenance and PvP conquest. A company might win a war to take a territory, only to lose their crafting stations to an invasion they could not defend because half their players were from rival factions who intentionally threw the fight.

Fort Control Points

Each territory has a fort with a capture point that requires PvP flagging to enter. Capture works on a majority-presence timer: if one faction has more players in the capture radius than any other faction, they slowly capture the point. A solo player takes 5 minutes. A full group of 5 takes 2 minutes 30 seconds. Fort control grants a 5% XP bonus and 20% influence bonus to faction members in that territory.

Control points deactivate during Wars and Invasions and reactivate 60 minutes after the event ends. This prevents capture point manipulation during scheduled large-scale events.

Observability and Data-Driven Design

Every minute, 23 million telemetry events flow from the game servers through Kinesis Data Streams into Kinesis Data Firehose, which deposits them in S3. From there, Amazon Athena and Redshift provide queryable analytics. Data analysts can query specific insights, like which wolf in a particular forest has been fought the most. Game designers then use these insights to modify the game in near-real-time.

On the operations side, the telemetry pipeline enables proactive problem detection. If a particular hub starts showing elevated tick times or memory usage, the ops team can investigate before players notice degraded performance.

Architecture Evolution

2016 Project announced as a survival sandbox built on Amazon Lumberyard, a fork of CryEngine 3. Original vision was a full-loot PvP survival game.
2018 Alpha testing begins. AGS identified blind spots in the architecture through early player testing. The survival direction was controversial, and the game began pivoting toward a more traditional MMO structure.
2020 Major pivot to PvE/PvP hybrid MMO. The game shifted from open-world full-loot PvP to a flagged PvP system with PvE questing, expeditions, and the territory war system. Settlements and crafting stations became persistent, upgradable structures.
2021 Q3 Launch (September 28). 913,634 peak CCU on Steam. Over 500 worlds deployed. Massive queue problems. Client-authority exploits, duplication bugs, and war lag exploits surface within weeks.
2021 Q4 Werner Vogels showcases the architecture at AWS re:Invent 2021. Population begins rapid decline. First round of emergency patches to address exploits, including converting war turret projectiles from physics objects to hitscan.
2022 World merges begin. Hundreds of underpopulated worlds consolidated into larger ones. Character transfer system and World Sets introduced. The Trading Post is unified across all settlements in a world.
2023 Continued merges and Brimstone Sands expansion. Population stabilizes at a much lower level. Additional server regions consolidated.
2024 New World: Aeternum relaunch. Console release (PS5, Xbox) alongside a PC refresh. Cross-play infrastructure added. The game continues running on the same AWS architecture with fewer worlds.
Lumberyard to O3DE

Amazon Lumberyard, the engine New World was built on, is no longer offered by Amazon. In July 2021, Amazon and the Linux Foundation announced its successor, Open 3D Engine (O3DE), an open-source engine under the Apache 2.0 license. By that point, roughly 70% of Lumberyard had been rewritten. New World itself remains on the original Lumberyard codebase and has not migrated to O3DE.
