Architecture Comparison / RuneScape

RuneScape / Old School RuneScape

World-Based Shards Sandbox MMORPG

A Java-based MMO running on the custom RuneTek engine since 2001, where each world is an isolated server process simulating the entire game map at a fixed 0.6-second tick rate. Its two-tier architecture separates game worlds from centralized backend services like login, the Grand Exchange marketplace, and player saves.

Overview

RuneScape is one of the longest-running MMOs in existence, launching as a browser-based Java applet in January 2001. Its server architecture reflects that heritage: every game world is an independent Java process that simulates the entire game map, handling up to 2,000 simultaneous players (1,400 on free-to-play worlds). Players pick a world at login and can hop between worlds freely, but each world is its own isolated simulation with its own NPC states, resource spawns, and ground items.

The engine that powers all of this is RuneTek, developed in-house by Jagex. OSRS still runs on RuneTek 3, the same engine branch from 2007, while RS3 uses RuneTek 7 with a C++ NXT client. The server side remains Java in both cases. Content is authored in RuneScript, a domain-specific scripting language that compiles to bytecode and runs inside the Java server. RuneScript was designed so that designers without Java experience could build quests, NPCs, and game mechanics without touching engine code.

Jagex operates a two-tier infrastructure. The front-end tier hosts game world processes in regional data centers across the UK, US, Canada, Australia, Germany, and other countries. The back-end tier handles login authentication, player saves, the Friends system, Clan Chat, the Grand Exchange, billing, and analytics. Back-end services have been progressively migrating to AWS, with the Jagex Accounts system running on EKS (Elastic Kubernetes Service) and RDS (Relational Database Service).

World Model

Each world is a containerized VM running the full Java game application. A single world simulates the entire game map with all NPCs, resources, and up to 2,000 players. Worlds are isolated from each other; a player can only be logged into one world at a time, enforced by the login server.

The 0.6s Game Tick

Every action in RuneScape is governed by a 600ms tick cycle. The server processes all queued player actions, updates NPC AI, resolves combat, and sends state updates to clients each tick. Observed tick lengths range from 0.606s on quiet worlds to 0.618s on full ones, because the server must process more entities when crowded.

RuneScript

Jagex's proprietary scripting language for content authoring. It compiles to bytecode that the Java engine interprets. Delays are expressed in tick counts, not milliseconds. The syntax is simple by design, allowing non-programmers to create quests, skill interactions, and NPC behaviors. The RS codebase contains over 6 million lines of engine code.

Two Clients, One Server

RS3 runs the NXT client (C++, released 2016) with deferred rendering, long draw distances, and GPU-accelerated effects. OSRS runs a modernized Java client through RuneLite (community) or the official client. Both connect to the same Java server backend, just different engine branches (RT3 for OSRS, RT7 for RS3).

Server Infrastructure

Jagex hosts game worlds as containerized virtual machines on ESXi hypervisors in commercial data centers distributed across the UK, US, Canada, Netherlands, Australia, Sweden, Finland, and Germany. Each physical machine runs multiple world VMs. This virtualization approach replaced dedicated physical servers to reduce hardware-related downtime and simplify maintenance.

The Java Virtual Machine (JVM) running each world process has been tuned with a pauseless garbage collector to prevent GC pauses from stalling tick processing. This is critical because a GC pause longer than 600ms would cause the world to skip a tick entirely, creating visible stutter for all 2,000 connected players.

Jagex has been progressively migrating backend services to AWS. The Jagex Accounts system runs on EKS (Kubernetes) and RDS (managed database). AWS Direct Connect circuits provide dedicated networking between Jagex's on-premise data centers and AWS, allowing regional edge locations to burst into cloud capacity when needed. The company has also begun testing entirely cloud-hosted game worlds, starting with US East Coast, to enable expansion into regions like South America and Southeast Asia without deploying physical hardware.

World Capacity and Performance

Each OSRS members world supports up to 2,000 concurrent players, while free-to-play worlds cap at 1,400. RS3 recently reduced its cap to 1,500 from 2,000. A single world simulates tens of thousands of NPCs, manages all resource spawns, and processes every player action each tick. The more players on a world, the longer each tick takes to process, which is why tick lengths vary from 0.606s on quiet worlds to 0.618s on full worlds like World 2.

When Jagex's total concurrent player count exceeds their existing world capacity, they can spin up additional worlds. During OSRS's record peak of 240,756 concurrent players in August 2025, the system operated across 150+ worlds distributed globally.

Network & Protocols

RuneScape uses a custom TCP-based binary protocol for client-server communication. The server sends state updates to the client every 600ms (once per tick), but the client sends packets to the server immediately based on user input. When a player clicks a tree and then immediately moves, two packets are sent in rapid succession. The server processes them in order, with the second overriding the first walking request.

The game logic is single-threaded. Each tick, the server reads all queued input packets from connected clients, processes game logic sequentially, and then sends update packets back. This single-threaded model simplifies the engine dramatically but means the server's throughput is fundamentally limited by how much work it can do in one tick on one core.

Login Flow

When a player launches the client, they first connect to a login server that handles authentication and loads their character data. The login server assigns a default world based on server latency, language preference, and physical distance. The client then establishes a direct TCP connection to the selected game world. The login server enforces the one-character-per-world rule: if your character is already logged in on another world, the login is rejected.

The login server is separate from game worlds. The flow is: Client connects to Game World, which forwards login to the Login Server for authentication and character retrieval, then returns the session to the client. The game world essentially acts as a gateway during the login handshake.

World Hopping

Players can switch worlds without fully logging out by using the world-hop feature. This disconnects from the current world and reconnects to a new one, but your character briefly disappears from both. There is a deliberate cooldown on world hopping to prevent abuse, and some content (like Wilderness PvP) imposes additional cooldown timers.

Data & Persistence

Player data is stored centrally on the back-end tier and loaded into a game world when the player logs in. When they log out or hop worlds, their character state is serialized and saved back to the central database. This means your bank, inventory, quest progress, and skill levels follow you across worlds. World-specific state (NPC positions, dropped items, resource respawn timers) lives only in the memory of that world's process and is not persisted between server restarts.

The Grand Exchange

The Grand Exchange (GE) is RuneScape's central marketplace and operates as a cross-world service. An offer placed on World 301 can match with an offer on World 450. The GE functions as a centralized order book: buy and sell orders include a quantity and price, and the system matches them using price priority with chronological tiebreaking. When a buyer's price meets or exceeds a seller's price, the trade executes at the earlier offer's price.

Items and coins are held in escrow when an offer is placed. Completed or partially completed trades can be collected from any bank in the game, not just the GE location. Each player has 8 GE slots (3 for free players), and per-item trade limits reset on 4-hour rolling windows to prevent price manipulation.

The 2023 GE rework introduced client-side prediction for the interface. Previously, every interaction with the GE required a server tick round-trip (600ms minimum). The new system handles UI state client-side and only contacts the server when an offer needs to be placed or collected, making the experience significantly more responsive.

The OSRS GE also exposes a public API with /5m, /1h, and /24h endpoints providing historical price and volume data for every tradeable item, with records going back to 2021.

Scaling & Performance

RuneScape's scaling model is horizontal through world replication. You don't make one world handle more players; you add more worlds. Each world is an independent process, so adding a new one means spinning up another containerized VM. This is simple and reliable but means no individual world can exceed its 2,000-player cap, and cross-world interactions are limited to specific services (GE, Friends, Clan Chat).

The 0.6-second tick rate is both a feature and a constraint. It means the server only needs to send updates roughly twice per second, keeping bandwidth requirements low. A full world with 2,000 players generates manageable network traffic because players only receive data about nearby entities, not the entire world state. But it also means the game feels sluggish compared to modern action games, and changing the tick rate would require rewriting millions of lines of RuneScript where timing is expressed in tick counts.

Horizontal Scaling

New worlds can be spun up as containerized VMs without code changes. Each world is fully independent. This approach scaled OSRS from 50 worlds at 2013 launch to 150+ worlds supporting 240K concurrent players by 2025.

Tick Budget

Each world has 600ms to process all player actions, NPC AI, combat resolution, and resource respawns. On full worlds, the tick budget is nearly consumed, causing tick elongation. The pauseless GC and VM tuning exist specifically to protect this budget.

Cloud Burst Capacity

AWS Direct Connect circuits allow regional edge locations to burst into AWS when extra capacity is needed. This will eventually support entirely cloud-based regions, enabling Jagex to deploy worlds in places like South America or Southeast Asia without physical infrastructure.

Data Center Migration

Jagex recently migrated from an aging data center to a new Tier 3 facility for improved power reliability. The core backend is progressively moving to AWS, while game worlds remain on virtualized on-premise hardware for now.

Technical Deep Dives

Bot Detection: Botwatch

RuneScape has fought bots since its earliest days. The primary detection system is Botwatch, which profiles accounts using multiple layers of analysis. The client sends heuristic data to the server continuously, including mouse movement patterns, click timing, camera adjustments, and input sequences. If the server stops receiving heuristic data, the player is logged out after exactly five minutes, since the server uses this data stream as its primary indicator of activity.

Jagex also deploys code-level traps against different bot types. For injection bots (which inject code into the client), Jagex writes "fake code" that does nothing in the real game but is visible to injected scripts. When a bot interacts with this fake code, Botwatch flags it. Reflection bots are harder to catch because they mirror the client's loaded classes and read from the copy rather than injecting into the original. Packet bots (which sent raw commands to the server) were eliminated years ago when Jagex encrypted and changed the communication protocol.

The community-built OSRS Bot Detector plugin uses machine learning trained on data provided by Jagex moderators. It runs through the RuneLite client and sends likely bot accounts to Jagex's anti-cheating team. The plugin analyzes nearby players' behavior patterns and compares them against known bot signatures.

The 2007 Backup and OSRS

Old School RuneScape exists because of a single backup created on August 10, 2007. This backup contained the complete source code for RuneScape as it existed on that date, running on RuneTek 3. Jagex did not routinely keep full backups until 2012, making this one exceptionally fortunate.

The backup included not just the live game but also content under development at the time, like early versions of the God Wars Dungeon and the Summoning skill. When OSRS launched in February 2013, players started fresh on Tutorial Island regardless of their RS3 progress. The game hit 50,000 concurrent players within four hours of launch.

OSRS has since grown to surpass RS3 in population. The team also uses a partial May 2012 backup for backporting post-2007 content into OSRS. Every major addition is voted on by players through in-game polls, with a 75% approval threshold required for new content (later reduced to 70%).

Instanced Content

While the overworld is shared across all players on a world, certain content is instanced. Boss encounters like the Theatre of Blood, Chambers of Xeric, and various slayer bosses create private instances for a player or group. These instances are spun up within the same world process and share its tick cycle. When the instance completes (boss killed or party wipes), it is destroyed. The number of concurrent instances a world can support is limited by its tick budget.

Deadman Mode Infrastructure

Deadman Mode is a seasonal PvP tournament where players compete on special worlds with permadeath mechanics. The finals involve hundreds of players in a shrinking safe zone, creating extreme server load concentrated in a small area. Jagex has historically run these tournaments on dedicated hardware with modified world configurations, but the events have been plagued by connectivity issues and DDoS attacks due to the high stakes involved (real-world prizes).

Architecture Evolution

2001
RuneScape Classic launches as a browser Java applet. The original engine is simple, with RuneScript Classic using a line-prefix syntax (= for triggers, + for conditionals). Servers are physical machines in the UK.
2004
RuneScape 2 launches on RuneTek 3 with a complete visual overhaul, 3D graphics, and modernized RuneScript. The server architecture remains world-based but scales to support growing player counts.
2007
The August 2007 backup is created. RuneScape hits peak membership numbers. Later that year, Wilderness PvP is removed and free trade is restricted, causing massive player backlash.
2007
Grand Exchange launches, replacing player-to-player trading as the primary commerce method. It runs as a cross-world centralized service from day one.
2012
Botwatch deployed alongside Botany Bay. A mass ban wave permanently removes hundreds of thousands of bot accounts. The Evolution of Combat (EoC) update overhauls the combat system, pushing many players away.
2013
Old School RuneScape launches from the August 2007 backup with RuneTek 3. Over 449,000 players vote in the poll. OSRS hits 50K concurrent on day one. RS3 (the main game) is rebranded and continues on RuneTek 5/7.
2016
NXT client released for RS3, replacing the Java client with a C++ application featuring deferred rendering, OpenGL 2.0+ support, and multi-threaded asset loading. Server side remains Java.
2018
OSRS Mobile launches on Android and iOS, bringing the game to mobile platforms. The same Java server backend handles mobile connections with no architectural changes.
2020+
AWS migration begins. Jagex Accounts system moves to EKS/RDS. Direct Connect circuits link on-premise data centers to AWS. Cloud-hosted game worlds are tested for US East Coast.
2024
Data center upgrade to Tier 3 facility. Migration from ESXi to KVM/Linux CFS planned for improved tick scheduling. Game worlds begin containerized deployment in new DC.
2025
OSRS hits 240K concurrent players, an all-time record. Cloud-hosted worlds expected to enable new regional deployments. RS3 reduces world caps to 1,500 players.
