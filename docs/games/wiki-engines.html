Game Engines for Multiplayer

Technology Wiki Engine Comparison

A technical survey of game engines used in multiplayer and MMO development. Covers commercial engines like Unreal and Unity, open-source options like Godot and O3DE, the lineage of id Tech through modern Call of Duty, purpose-built MMO engines from studios like CCP and Blizzard, and the ForgeLight engine behind PlanetSide 2's record-breaking battles.

Unreal Engine

Architecture Overview

Unreal Engine, developed by Epic Games, is the most widely used AAA game engine in the industry. Written in C++ with a visual scripting system called Blueprint, it has powered everything from small indie shooters to massive open-world games. UE5 introduced several landmark systems: Nanite (virtualized micropolygon geometry that eliminates manual LOD authoring), Lumen (fully dynamic global illumination and reflections using software and hardware ray tracing), Mass AI (crowd simulation handling 10,000+ NPCs with individual behaviors), and Chaos Physics (a replacement for PhysX that handles destruction, cloth, hair, and ragdolls).

The engine structures multiplayer games around a server-authoritative client-server model. The server maintains the true game state. Clients send input to the server via Remote Procedure Calls, and the server replicates the resulting state changes back to clients. Actors (the base object class in UE) can be marked for replication, and the engine handles serialization, delta compression, and priority-based bandwidth allocation.

UE5's default networking was designed for session-based matches with 64 to 100 players, not for persistent MMO worlds with thousands. Studios building larger-scale projects typically integrate external backend frameworks for horizontal scaling, sharding, and container orchestration through tools like Kubernetes or Agones.

Networking Model

Unreal uses a property replication system where the server marks certain Actor properties as replicated. When those properties change, the engine serializes the delta and pushes it to relevant clients. Three types of RPCs handle event-driven communication:

  • Server RPCs are called by clients and executed on the server. Used for input validation and action requests.
  • Client RPCs are called by the server and executed on the owning client. Used for player-specific feedback.
  • Multicast RPCs are called by the server and executed on all connected clients. Used sparingly due to bandwidth cost.

The Replication Graph is an optimization plugin that replaced the default actor-by-actor relevancy checks. In standard replication, every replicated actor evaluates every connected client each frame to determine if it should send data. This scales poorly. The Replication Graph groups actors into spatial cells and caches relevancy decisions, dramatically reducing CPU overhead. Epic's own Fortnite Battle Royale uses the Replication Graph to handle 100 connected players and roughly 50,000 replicated actors per match.

Movement prediction uses client-side prediction with server reconciliation. The client predicts movement locally for responsiveness, the server validates and corrects when needed, and the client smoothly interpolates to the corrected position.

Key Strengths

  • Best-in-class rendering pipeline with Nanite, Lumen, and Virtual Shadow Maps
  • Blueprint visual scripting lowers the barrier for non-programmers
  • Mature dedicated server support with built-in replication
  • Massive marketplace of plugins, assets, and community knowledge
  • Cross-platform deployment to PC, consoles, mobile, and VR

Limitations for MMOs

  • Replication system was built for match-based games (64-100 players), not persistent worlds
  • Scaling beyond a single server requires significant custom engineering
  • Heavy memory footprint and long compile times for large C++ projects
  • Past Unreal-based MMOs like TERA struggled with large-scale PvP performance
Licensing

Free to download and use. A 5% royalty applies to lifetime gross revenue above $1 million. Revenue from Epic Games Store sales is royalty-free. The "Launch Everywhere with Epic" program reduces the royalty to 3.5% for games that ship on EGS at or before other stores (effective January 2025). Non-game commercial use requires an annual seat license of $1,850.

Notable Multiplayer Games

Fortnite (Epic Games), PUBG (Krafton), Valorant (Riot, heavily modified UE4), Rocket League (Psyonix), Sea of Thieves (Rare), The Finals (Embark Studios), Ashes of Creation (Intrepid Studios, UE5 MMO in development).

Unity

Architecture Overview

Unity is the most widely used game engine by project count, powering roughly half of all mobile games and a substantial share of indie and mid-tier PC titles. The engine is written in C++ with gameplay scripting in C#. Unity has been undergoing a major architectural shift from its traditional GameObject/MonoBehaviour paradigm to DOTS (Data-Oriented Technology Stack), which includes the Entity Component System (ECS), the Burst compiler for high-performance C#, and the Job System for safe multithreading.

For multiplayer, Unity offers two networking solutions that target different use cases. Netcode for GameObjects (NGO) uses the familiar MonoBehaviour workflow and targets casual cooperative games with smaller player counts. Netcode for Entities is built on top of DOTS/ECS and targets competitive, large-scale multiplayer with server-authoritative prediction, interpolation, and lag compensation.

Both networking solutions use the Unity Transport Package (UTP) as their underlying transport layer. Unity also provides companion services: Relay (for NAT punchthrough and connectivity), Lobby (matchmaking), and Vivox (voice chat).

Networking Model

Netcode for GameObjects supports both client-authoritative and server-authoritative models. It synchronizes scenes and GameObject data across clients using a host/listen-server model where one client acts as the host. This is the simpler path, well-suited for co-op games with 4-16 players.

Netcode for Entities is strictly server-authoritative with client prediction. It introduces "ghosts" as the network representation of entities, supports automatic delta compression and relevancy, and can strip rendering assets from dedicated server builds. Unity has demonstrated 128+ player cross-platform matches using this system. The entity-based approach allows far more state to be synchronized over the network compared to the GameObject approach.

Unity also integrates extensively with third-party networking solutions, most notably Photon (by Exit Games), which provides cloud-hosted rooms-based multiplayer with global relay infrastructure. Many Unity multiplayer games ship with Photon rather than Unity's first-party networking.

Key Strengths

  • Largest ecosystem of tutorials, assets, and community knowledge
  • C# is more accessible than C++ for smaller teams
  • DOTS/ECS provides a path to high-performance server code
  • Strong cross-platform support, particularly for mobile
  • Photon integration provides battle-tested cloud networking

Limitations for MMOs

  • DOTS/ECS is still maturing and requires a steep learning curve
  • Netcode for GameObjects was not designed for large player counts
  • No built-in dedicated server hosting (requires third-party services)
  • The 2023 Runtime Fee debacle damaged developer trust significantly
  • Garbage collection pauses in C# can cause frame hitches in server code
Licensing (Post-Runtime Fee)

Unity canceled the controversial Runtime Fee in 2024 and reverted to seat-based subscriptions. Unity Personal is free for developers earning under $200K/year. Unity Pro costs $2,200/year per seat (required above $200K revenue). Unity Enterprise costs $4,000/year per seat (required above $25M revenue). No royalties.

Notable Multiplayer Games

Escape from Tarkov (Battlestate Games), Rust (Facepunch Studios), Among Us (InnerSloth), Fall Guys (Mediatonic), Hearthstone (Blizzard), Genshin Impact (miHoYo), VRChat (VRChat Inc.), Albion Online (Sandbox Interactive, Unity client with custom Java server).

Godot Engine

Architecture Overview

Godot is a free and open-source game engine released under the MIT license. It supports two scripting languages: GDScript (a Python-like language designed for the engine) and C# (via Mono/.NET). The engine uses a scene-tree architecture where everything is a node, and games are composed by nesting scenes within scenes.

Godot saw a massive surge in adoption after Unity's 2023 Runtime Fee controversy, with contributions doubling and high-profile donations arriving from studios like Re-Logic ($100,000 plus ongoing monthly support). It has produced genuine commercial hits: Brotato sold over 10 million copies, and multiple Godot games have cleared $4-6 million in revenue.

The engine is governed by the Godot Foundation, a non-profit entity. There are no fees, no royalties, and no restrictions on commercial use. You own every line of your game's code, and you can modify the engine itself if you need to.

Networking Model

Godot provides multiplayer networking at three tiers. The high-level API offers RPC functions, the MultiplayerSpawner node (for remote object instantiation), and the MultiplayerSynchronizer node (for property synchronization). The mid-level MultiplayerPeer abstraction provides protocol-agnostic networking. The low-level tier gives direct access to the transport protocol.

Three transport backends are available:

  • ENet (default): UDP-based with optional reliability and ordered delivery. Good performance for action games. Hard connection cap of 4,095 clients, with practical instability appearing around 1,300 connections due to CPU spikes that cause mass disconnects.
  • WebSocket: TCP-based, works in web browsers. Higher latency because TCP forces all RPCs to be reliable, making it poorly suited for fast-paced games.
  • WebRTC: UDP-based peer-to-peer, works in browsers. More complex setup requiring a signaling server, but offers low latency for browser-based games.

Godot 4.0 rewrote the scene replication API to be more user-friendly, and Godot 4.4 (March 2025) introduced Jolt physics integration. Steam networking is supported through GodotSteam, with a full MultiplayerPeer implementation available since GodotSteam 4.16.

The "Godot for MMOs" Debate

This comes up constantly in the Godot community, and the honest answer is that Godot was not designed for MMO-scale networking. The ENet backend hits a hard wall at 4,095 connections and becomes unstable well before that. Developers attempting MMO architectures in Godot have found that the server drops all connected clients at once when a mysterious CPU threshold is crossed, rather than degrading gracefully.

That said, developers have built zone-based sharding systems on top of Godot, using Nginx to route connections to per-zone server instances. The approach works, but it requires building most of the MMO infrastructure yourself. At that point, Godot is functioning as a client renderer rather than a multiplayer framework, and the networking layer is largely custom code.

Wompworld Relevance

For a project like Wompworld that targets a persistent single-shard ocean world with hundreds or thousands of concurrent players, Godot's networking ceiling is a serious constraint. You would need to build a custom server backend and use Godot purely as a client. The engine is excellent for smaller-scale multiplayer (co-op, small PvP), but it is not an out-of-the-box MMO solution.

Notable Multiplayer Games

Brotato (Blobfish), Dome Keeper (Bippinbits), Cassette Beasts (Bytten Studio), Halls of Torment (Chasing Carrots). Most of these are single-player or local co-op. Large-scale multiplayer titles in Godot remain rare.

CryEngine / Lumberyard / O3DE

The Lineage

This is a story of one engine forking into three distinct codebases over two decades. CryEngine was created by Crytek in 2002 and gained fame with Far Cry and the Crysis series. In 2015, Amazon licensed CryEngine for a reported $50-70 million and released their modified version as Amazon Lumberyard in February 2016. In July 2021, Amazon and the Linux Foundation announced that Lumberyard would become O3DE (Open 3D Engine), an open-source engine under the Apache 2.0 license managed by the Open 3D Foundation.

By the time O3DE launched, roughly 70% of the engine had been completely overhauled. CryPhysics was replaced with NVIDIA PhysX. CryNetwork and GridMate were replaced with an entirely new AzNetworking stack. CryRenderer was replaced with the Atom renderer. As Derek Reese of Red Hat put it: O3DE is not CryEngine/Lumberyard, but its own independent engine that continues some of the best features and code from Lumberyard.

How Star Citizen Forked It

Cloud Imperium Games began building Star Citizen on CryEngine 3 and internally named their heavily modified version Star Engine by September 2016. In December 2016, CIG announced the switch to Amazon Lumberyard, primarily to escape the limitations of CryNetwork, CryEngine's built-in networking.

Chris Roberts explained the rationale: CryNetwork was a single-server architecture designed for FPS matches, not for a massively online space simulation. Amazon's cloud infrastructure and AWS integration offered a path toward the distributed server mesh that Star Citizen's design requires. CIG preserved all their custom work (64-bit precision, planet tech, local physics grids, zone system, object containers) and swapped only the like-for-like engine foundation from CryEngine to Lumberyard.

This move triggered a lawsuit from Crytek in December 2017, alleging copyright infringement and breach of contract. The case was eventually settled.

O3DE Networking

O3DE's networking is built in two layers. AzNetworking is the low-level transport that focuses on reduced code size, low latency on send/receive, and low message processing overhead. The Multiplayer Gem sits on top of AzNetworking and provides entity-based asynchronous networking using event-driven network properties and RPCs for state synchronization.

The engine supports dedicated server builds, cloud integration for player authentication and telemetry, and scripting via Lua and Python. Script Canvas provides visual scripting for multiplayer logic.

Key Strengths

  • O3DE is fully open-source (Apache 2.0) with no fees or royalties
  • Modular architecture lets you include only the components you need
  • Modern Atom renderer with hardware-accelerated ray tracing
  • Backed by major industry partners (AWS, Intel, Huawei, Niantic, Epic, Microsoft, Tencent)

Limitations for MMOs

  • Small community and limited ecosystem compared to Unreal or Unity
  • Very few shipped commercial games (MadWorld is the most notable)
  • Documentation is sparse and the learning curve is steep
  • The CryEngine heritage means some subsystems still carry legacy complexity
  • Hiring developers with O3DE experience is extremely difficult
CryEngine Itself (Current Status)

CryEngine continues to exist as a separate product from Crytek, but its relevance has diminished. Crytek shifted to a 5% royalty model on gross revenue above $5,000 per year. The engine powers Hunt: Showdown (Crytek) and was used for Kingdom Come: Deliverance (Warhorse Studios), but new large-scale projects choosing CryEngine are rare. Crytek appears focused primarily on their own games rather than engine licensing.

Custom Engines

Why Studios Build Their Own

Every major MMO that has survived for more than a decade runs on a custom engine. This is not a coincidence. The unique demands of persistent worlds with thousands of concurrent players, server-authoritative anti-cheat design, specialized world management, and real-time economic simulation make general-purpose engines a poor fit. Veterans of mass PvP games know that you cannot use an FPS engine for an MMORPG without radical modification.

The core argument for custom engines comes down to control. A studio building an MMO needs precise authority over the networking stack, memory management, level loading, asset streaming, input synchronization, and server-side simulation. Off-the-shelf engines give you a renderer and a physics engine, but the parts that make an MMO work are exactly the parts you need to build yourself.

World of Warcraft (Custom Blizzard Engine)

WoW runs on a custom engine that evolved from the Warcraft III codebase. The engine is written in C++ with Lua handling UI and game logic scripting. It has undergone continuous updates across two decades of expansions, but the fundamental architecture was designed specifically for WoW's needs: zone streaming, raid instancing, auction house infrastructure, and a server-authoritative model that processes combat on the server.

The engine is optimized for wide hardware compatibility, which is a deliberate design choice. WoW has always prioritized running well on low-end machines to maximize its addressable player base. The tradeoff is that the engine carries significant legacy code that limits what can be modernized without a ground-up rewrite.

EVE Online (Carbon / Stackless Python)

EVE Online is the most architecturally unusual MMO in production. CCP Games chose Stackless Python for game logic, which provides lightweight microthreads (tasklets) for handling massive concurrency without the overhead of OS threads. CCP has stated that their commercial success was built on the decision to use Stackless Python as their foundation.

The engine runs every player in a single shared universe on one server cluster (Tranquility). Python handles game logic while C++ handles the rendering pipeline and performance-sensitive systems. The tradeoff is that Python's Global Interpreter Lock means each game node is single-threaded, and CCP has spent twenty years engineering around this constraint rather than rewriting the core.

RuneScape (RuneTek)

Jagex built the RuneTek engine from scratch, now on version 7. The engine's most notable feature is RuneScript, a proprietary domain-specific scripting language used for server-side event handling. RuneTek evolved through seven major versions, from the original DeviousMUD graphics engine through the RuneTek 5 rewrite that enabled targeting multiple graphics APIs (DirectX, OpenGL, console hardware) from a single engine.

Jagex's founder built the entire engine from the ground up, and it remains one of the clearest examples of how a purpose-built platform can serve a game for decades. The engine handles RuneScape's tile-based world, real-time combat calculations, and a player-driven economy with millions of registered accounts.

Guild Wars 2 (ArenaNet Proprietary Engine)

Guild Wars 2 runs on a custom engine that evolved from the Guild Wars 1 codebase. The client is written in C++ and the server runs C++ on Linux. The engine was specifically designed to handle large-scale open-world events (World Bosses, WvW battles with hundreds of players) without trusting the client. As one developer noted, building a correctly server-authoritative engine is genuinely hard to do, and ArenaNet invested heavily in getting it right rather than bolting security onto a third-party engine.

The downside is technical debt. GW2's engine uses DirectX 9, which limits it to effectively single-core rendering. Major graphics improvements are unlikely without a complete engine rewrite, which realistically means waiting for a sequel.

The Trade-offs

Advantages

  • Complete control over networking, memory, and performance
  • Tailored server-authoritative design for anti-cheat
  • No royalties or licensing fees to third parties
  • Can build features that off-the-shelf engines cannot support
  • Unique technical capabilities become competitive advantages

Disadvantages

  • Enormous upfront cost in time and engineering talent
  • Hiring developers for a proprietary engine is much harder
  • Technical debt accumulates and is nearly impossible to escape
  • Onboarding new team members is slow and expensive
  • Engine migration is a multi-year project, if it happens at all
Wompworld Relevance

A custom engine is the right choice for established studios with dedicated engine teams and multi-year timelines. For a team without professional game development experience, building a custom engine would consume the entire development budget before gameplay even begins. The realistic path is choosing an existing engine and building custom networking and server infrastructure on top of it.

id Tech Lineage

The Family Tree

The id Tech engine family, created by John Carmack and id Software, is the most influential lineage in FPS history. Every major engine family in competitive shooters traces back to this codebase in some way.

id Tech 1 (1993) powered Doom and Doom II using BSP trees for rendering. id Tech 2 (1996-1997) powered Quake and Quake II, introducing true 3D rendering and the client-server networking model that would define online shooters. Valve licensed id Tech 2 and modified it into GoldSrc, the engine behind Half-Life, which later became the Source engine.

id Tech 3 (1999) powered Quake III Arena and introduced the snapshot-based networking system that became the industry standard. The server runs the simulation at a fixed tick rate, captures snapshots of the game state, and sends delta-compressed updates to each client over UDP. This architecture was licensed to dozens of studios and directly spawned two major engine families.

The Call of Duty Branch (IW Engine)

Infinity Ward used id Tech 3 to build the original Call of Duty in 2003. They then evolved it into the IW Engine, which has powered every mainline Call of Duty title since. Each version added substantial features:

  • IW 3.0 (CoD4: Modern Warfare, 2007): Bullet penetration, improved AI, lighting upgrades, particle system overhaul
  • IW 4.0 (MW2, 2009): Texture streaming technology
  • IW 8.0 (Modern Warfare 2019): Complete overhaul with PBR rendering, photogrammetry, DX12, ray tracing

Infinity Ward's Mark Rubin has stated that there is nothing left in the engine from Quake anymore, as they have rewritten every section. While you can technically say that Modern Warfare uses a heavily modified Quake engine, that description is incredibly reductive after 20+ years of rewrites.

The Valve Branch (GoldSrc / Source / Source 2)

Valve derived GoldSrc from id Tech 2 (Quake II engine) for Half-Life (1998). This evolved into the Source Engine, which powered Half-Life 2, Counter-Strike: Source, Team Fortress 2, Left 4 Dead, and Dota 2. Source 2 now powers Counter-Strike 2, Dota 2 (post-Reborn), and Half-Life: Alyx.

id Software continued evolving the engine through id Tech 4 (Doom 3, 2004), id Tech 5 (Rage, 2011, featuring MegaTexture), id Tech 6 (Doom 2016, Vulkan rendering), id Tech 7 (Doom Eternal, 2020, 10x geometry detail over id Tech 6), and id Tech 8 (Doom: The Dark Ages, May 2025, path tracing and destructible environments). Since Microsoft's acquisition of ZeniMax/Bethesda, id Tech is now Xbox-exclusive and no longer licensed to third parties.

Legacy Impact

The snapshot-based delta compression networking that id Tech 3 introduced in 1999 is still the foundation of how most competitive shooters synchronize state. When people argue about tick rates in Call of Duty or Counter-Strike, they are debating parameters of an architecture that John Carmack designed for Quake III Arena over 25 years ago.

ForgeLight Engine

Architecture Overview

ForgeLight is a proprietary engine developed by Sony Online Entertainment (now Daybreak Game Company) specifically for massive-scale online games. It was built from the ground up to solve a problem that general-purpose engines could not: running an MMOFPS with thousands of concurrent players in a seamless open world without loading screens.

The engine was announced as the platform for all future SOE MMOs, with its biggest technical achievement being the elimination of discrete zones. Previous EverQuest games loaded separate zone files when players crossed boundaries. ForgeLight replaced this with continuous world streaming, real-time radiosity lighting, advanced particle systems, and volumetric fog across 64 km² continents.

ForgeLight was originally single-threaded but was modified to support multithreading for the PlayStation 4 port of PlanetSide 2, which improved PC performance as a side effect. The renderer was upgraded from DirectX 9 to DirectX 11 during PlanetSide 2's lifecycle. Physics uses NVIDIA PhysX, including water buoyancy and projectile drag systems added in post-launch updates.

Networking and Scale

ForgeLight's defining feature is its ability to support extreme player density. PlanetSide 2 supports up to 2,000 concurrent players per continent across open-world combat zones. In 2015, PlanetSide 2 set the Guinness World Record for most players in an online FPS battle, with 1,158 players recorded in a single fight during a sponsored event.

The engine uses a server-authoritative model where the server validates all combat interactions, positions, and state changes. The challenge ForgeLight solved was not just handling 2,000 connections, but processing combat calculations, vehicle physics, projectile trajectories, and base capture mechanics for all of those players simultaneously across a seamless map with no instancing.

Games Using ForgeLight

  • PlanetSide 2 (2012): The engine's flagship title. MMOFPS with three factions fighting over continents. Still running in 2026 under Rogue Planet Games.
  • H1Z1 (2015, later split into H1Z1: Just Survive and H1Z1: King of the Kill): The battle royale that popularized the genre before PUBG and Fortnite took over.
  • EverQuest Next (canceled 2016): Was planned as the next EverQuest MMO using ForgeLight with voxel-based procedural destruction via Voxel Farm integration. Canceled after SOE was sold and became Daybreak.
  • Landmark (2016): A voxel building sandbox spun off from EverQuest Next technology. Shut down in 2017.
  • Free Realms (2009) and Clone Wars Adventures (2010): Earlier, lighter SOE titles that used ForgeLight.

Limitations

  • Proprietary to Daybreak (now under Enad Global 7). Not available for licensing.
  • Originally single-threaded, and the multithreading retrofit has limits
  • DirectX 11 ceiling means no modern rendering features (ray tracing, mesh shaders)
  • PlanetSide 2's performance issues at large scale are well-documented, with frame rates dropping significantly in large fights
  • Engine development has slowed as Daybreak's resources have been spread thin across multiple studios
Wompworld Relevance

ForgeLight is the closest existing engine to what Wompworld needs in terms of player density in a seamless world. PlanetSide 2 proved that thousands of players could fight in real-time across a shared map without instancing. The problem is that ForgeLight is not available for licensing, and the engine's age shows. What ForgeLight demonstrated is that purpose-built engines for massive multiplayer can achieve things general-purpose engines cannot, but you have to build (or license) the technology yourself.

Key Takeaways for Wompworld

Looking at the full landscape of engine options through the lens of a persistent ocean-world MMO:

Unreal is the Strongest Commercial Option

UE5 has the best rendering pipeline, the most mature dedicated server support, and the largest talent pool. The replication system needs extension for MMO scale, but the Replication Graph provides a proven optimization path. The 5% royalty is the cost of not building your own engine.

Unity is Viable but Risky

Netcode for Entities on DOTS/ECS is architecturally promising for server performance, but DOTS is still maturing. The Runtime Fee debacle created trust issues that Unity is still recovering from. C# is easier to work with than C++, which matters for a small team.

Godot is Wrong for This Scale

Godot's networking ceiling of roughly 1,300 stable connections makes it unsuitable as a full-stack MMO solution. It could work as a client-only renderer with a custom server backend, but at that point you are building your own networking anyway and losing Godot's main advantage (simplicity).

Custom Engines are Right but Expensive

Every long-lived MMO runs on a custom engine for a reason. The problem is that building a custom engine requires years of development and a team of experienced engine programmers. For a team without game dev experience, this is not the first project to attempt it on.

The practical reality for a project like Wompworld is that Unreal Engine 5 with a custom server backend and sharding layer offers the best balance of visual quality, existing tooling, available talent, and extensibility. The alternative is Unity with DOTS, which trades rendering quality for an easier language (C#) and lower upfront complexity. Both will require significant custom networking work to reach MMO scale.
