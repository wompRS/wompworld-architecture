Overview

Star Citizen's server architecture is arguably the most technically ambitious in the MMO space. Rather than using traditional instancing or zone-based sharding, Cloud Imperium Games (CIG) is building a system where multiple game servers collaborate to simulate a single continuous universe. The key technologies that make this possible are Object Container Streaming (OCS), the Replication Layer, Persistent Entity Streaming (PES), and Server Meshing.

The foundation of all this is the StarEngine, CIG's heavily modified fork of CryEngine 3 (later incorporating elements from Amazon Lumberyard). CIG converted the engine to use 64-bit double-precision coordinates, replacing the standard 32-bit floats that limit most game engines to maps of roughly 8km x 8km. With 64-bit precision, Star Citizen can represent star systems millions of kilometers across with sub-centimeter accuracy.

As of Alpha 4.0 (December 2024), static server meshing is live, supporting up to 500 players per shard across two star systems (Stanton and Pyro). Dynamic server meshing, which will allow servers to split and merge in real time based on player density, remains in development.

64-Bit Coordinate System

CIG spent roughly 8 months converting CryEngine from 32-bit to 64-bit world coordinates. The physics system was refactored alongside positioning. GPU rendering was made camera-relative (since GPUs still work in 32-bit), calculating positions relative to the camera in 32-bit while the CPU tracks the true 64-bit position. This enables game worlds spanning entire star systems.

Object Container Streaming

OCS allows both the client and server to only hold in memory the minimum set of entities required for gameplay. Objects are grouped into "containers" that stream in and out based on player proximity. Client-side OCS shipped in 2018; server-side OCS (S-OCS) followed in late 2019. This was the first required step toward server meshing.

Persistent Entity Streaming

Introduced in patch 3.18, PES allows every dynamic object in a shard to persist even when players leave the area or log out. Dropped items, parked ships, cargo, and wreckage remain in the universe. The state of every entity is stored in the EntityGraph database and streamed back in when players return.

Server Meshing (Static)

Live since Alpha 4.0 (December 2024). Multiple servers collaborate to simulate one shard, with each server responsible for a fixed region of space. Stanton and Pyro each run on separate servers. Players can travel between systems via jump points (a ~2 minute tunnel traversal) without loading screens. Up to 500 players per shard.

Server Infrastructure

Star Citizen is hosted entirely on Amazon Web Services (AWS). CIG uses AWS EC2 instances for dedicated game servers (DGS), with servers deployed across multiple AWS regions. One significant benefit of AWS hosting is that inter-server communication uses Amazon's CDN backbone, which provides roughly 10% lower latency compared to routing over the public internet.

The Shard Model

A "shard" in Star Citizen terminology is a logical grouping of servers that together simulate one copy of the universe. Before server meshing, each shard was a single DGS that ran the entire Stanton system, limited to about 50-100 players. With static server meshing (Alpha 4.0), a shard consists of multiple DGS nodes, each responsible for a fixed region, supporting up to 500 players total.

DGS (Dedicated Game Server) Nodes

Each DGS runs the full game simulation for its assigned region: physics, AI behavior, game logic, vehicle systems, and environmental interactions. DGS nodes do not communicate directly with clients. Instead, the Replication Layer sits between clients and DGS nodes, handling entity replication and state synchronization.

Crash Recovery

If a DGS node crashes, the Replication Layer spins up a replacement node. The new node loads entity state from the EntityGraph database and reconnects to the Replication Layer. Client gateways redirect to the new node, and gameplay resumes. Players may experience brief freezing and some position snapping, but the process targets sub-minute recovery. If a Replicant crashes, a similar recovery process occurs. The Gateway layer is even simpler and recovers in seconds.

Server Meshing Load Tests

CIG has conducted several large-scale meshing tests. In March 2024, an 800-player test was run. On October 10, 2024 (Star Citizen's birthday), "Meshing Test E" pushed to 1,000 concurrent players and was further stressed to 2,000 to measure stability and identify crash conditions. These tests inform the stability targets for live deployment.

Network & Protocols

Star Citizen's networking is built around the Replication Layer, a middleware system that sits between clients and game servers. The Replication Layer is designed to be event-driven rather than tick-based. As packets arrive, it processes and forwards them immediately, rather than batching updates on a fixed tick cycle like a traditional game server.

Entity Authority

At any given time, exactly one DGS has authority over a given entity. Authority is allocated on a first-come-first-served basis and managed by the Atlas service. When a player moves from one DGS's region to another, authority over that player's entity (and their ship, cargo, etc.) migrates to the new DGS. The Replication Layer handles this handoff seamlessly.

Streaming Bubbles

Each connected player has a "streaming bubble" around them. The Replication Layer determines which entities fall within each player's bubble and only sends updates for those entities. Entity hierarchies (like a ship containing players, cargo, and equipment) are treated as streaming groups to ensure a ship always streams in as a complete unit.

Tick Rate Challenges

CIG targets 30 Hz for DGS tick rate, but in practice the rate has dropped as low as 10 Hz during high-load scenarios. The primary bottleneck is entity count: a server simulating physics for thousands of entities (including individual pieces of cargo, ship components, weapons, NPCs, and player characters) hits CPU limits faster than a server tracking simple player positions. Server meshing helps by reducing the number of entities per DGS, but entity-dense scenarios like fleet battles still stress individual nodes.

Multicrew Physics Grids

Each ship has its own local physics grid. Players walking inside a moving ship experience physics relative to the ship, not the universe. When a player transitions between physics grids (entering or exiting a ship, EVA transitions), the server must reconcile two different reference frames. This was historically a major source of bugs and freezes, especially in version 3.0 where crossing physics grid boundaries could cause multi-second pauses.

Data & Persistence

Star Citizen's persistence is handled by the EntityGraph, a graph database that stores the state of every network-replicated entity in the universe. EntityGraph evolved from the original "iCache" concept, which was CIG's early design for a central entity cache.

EntityGraph Structure

Each entity in EntityGraph is a node with properties: object class, item type, legal owner, physical orientation, and precise 64-bit position within the game world. Relationships between entities are stored as edges (e.g., "this weapon is mounted on this ship hardpoint" or "this cargo box is inside this ship's hold"). When a DGS needs to simulate a region, it queries EntityGraph for all entities in that area, streams them in, and begins simulating.

Persistent Entity Streaming (PES)

Before PES, logging out or leaving an area would cause loose objects (dropped items, parked ships) to disappear. PES changed this by persisting every dynamic entity's state back to EntityGraph. Players can now drop items, park ships, and log out, then return later to find everything where they left it. Mission progress (like partially delivered cargo) persists across sessions.

The Hybrid Service

Rather than building each Replication Layer microservice (Replicant, Atlas, Scribe, Gateway) as separate deployable services from the start, CIG initially built the "Hybrid" service: a single process containing the logic of all four services plus EntityGraph integration. This reduced the number of moving parts during development and testing. The plan is to split these into independent, separately scalable services over time.

Quantum Economy Simulation (StarSim)

Quantum is a background economy simulation that runs millions of lightweight virtual NPCs called "Quanta." Each Quanta has personality traits (morality, aggression), job proficiencies, and makes decisions about what work to do. Quanta simulate mining, trading, piracy, and security work. Their collective behavior drives commodity prices, mission availability, and NPC encounter frequency across the universe.

Probability Volumes

Quantum tracks "probability volumes" rather than individual Quanta positions. Each volume (a region of space, a planet surface area, a station) has statistical probabilities of events occurring. The simulation only resolves to specific NPCs or events when a player looks at a specific location, collapsing the probability into concrete game state.

Ship Component Damage

Each ship has a detailed component hierarchy: power plants, coolers, shield generators, weapons, thrusters, quantum drives, and more. Damage is tracked per-component, with each component having its own health, degradation state, and functional output. All of this state must be replicated across the network and persisted to EntityGraph, adding significant per-entity data overhead.

Odin Visualization Tool

CIG built "Odin," an internal web-based tool for visualizing Quantum simulation data in real time. Odin displays commodity price fluctuations, goods flow between locations, Quanta population distributions, and economic health metrics. It serves as both a development tool and a monitoring dashboard for the live economy.

Scaling & Performance

Star Citizen's scaling story is the story of server meshing. Before meshing, the entire Stanton system ran on a single server, hard-capped at around 50-100 players. Everything in the 4.x series is about breaking past that limit.

Static Server Meshing (Live)

In static meshing, each DGS is assigned a fixed region of space. The assignment does not change based on load. If 400 players crowd into one region, that region's DGS must handle all of them. The mesh structure is: one server per major location (Crusader, Hurston, etc.) plus servers for Pyro's regions. Players cross server boundaries seamlessly as they travel.

Dynamic Server Meshing (Planned)

Dynamic meshing will allow server boundaries to shift in real time. If a large group of players gathers at one location, the system will split that region across multiple DGS nodes. If an area empties out, servers merge back together. This requires the Atlas service to continuously monitor load and reassign entity authority on the fly, while the Replication Layer maintains seamless connectivity for clients.

Cost Scaling

The computational cost of simulation varies dramatically by scenario. CIG has noted that 100 players in FPS combat are cheaper to simulate than 100 players fighting in single-seater spaceships with missiles and energy weapons. Large multicrew ships with dozens of components, multiple players, NPCs, and cargo are the most expensive entities per-unit. Dynamic meshing will need to account for this variable cost when deciding how to partition servers.

Technical Deep Dives

The 64-Bit Conversion

CIG Technical Director Sean Tracy (a Crytek veteran) led the conversion from 32-bit to 64-bit coordinates. The key insight was that not every engine module needed conversion. The AI module, for instance, has no benefit from 64-bit execution, so it was left operating in 32-bit while receiving 64-bit position data. Physics and positioning were refactored together, since physical simulation at astronomical scales requires the same precision as positioning.

The GPU side required a different approach entirely. GPUs operate natively in 32-bit, and converting them to 64-bit would be prohibitively expensive. Instead, CIG implemented camera-relative rendering: all positions sent to the GPU are calculated relative to the camera in 32-bit, while the CPU tracks true world positions in 64-bit. This preserves rendering performance while enabling millimeter-accurate positioning across millions of kilometers.

The Stanton-Pyro Jump Point

The jump point between Stanton and Pyro is the first inter-system connection in the live game. Traversing the jump tunnel takes approximately 2 minutes and requires active piloting to avoid collisions with tunnel walls and obstacles. From a server architecture perspective, the jump point is where entity authority transfers from a Stanton DGS to a Pyro DGS, coordinated by the Replication Layer. The client experiences seamless travel with no loading screens.

Early testing revealed bugs with jump point transitions that initially prevented smooth crossings. A week after the first meshing test, the first successful jump point traversal was achieved, marking a major milestone in proving the meshing architecture worked.

How Quantum Works Under the Hood

Quantum was originally written in an unspecified language and later rewritten in C#, yielding a performance improvement of over two orders of magnitude. This was necessary because simulating the Stanton system alone requires a significant number of Quanta to achieve acceptable economic fidelity, and future systems like Pyro, Nyx, and beyond will multiply that requirement.

Each Quanta chooses jobs based on a value calculation: base payout multiplied by proficiency multipliers, modified by trait alignment with the job type. A Quanta with high aggression and low morality perceives piracy jobs as more valuable than a law-abiding Quanta would. This creates natural economic diversity without requiring hand-tuned distributions.

Tony Zurovec, CIG's director of persistent universe design, described the system as inspired by real quantum mechanics: the simulation tracks probability volumes rather than individual positions, only "collapsing" to specific entities when a player observes them. This allows the simulation to scale to millions of Quanta without tracking millions of individual positions.

Network Bind Culling

Before OCS, a predecessor technology called Network Bind Culling was proposed at a 2015 tech summit involving engineers from CIG studios, Illfonic, Behaviour Interactive, and Wormbyte. Development started in May 2015 in Frankfurt and was deployed to the mainline codebase in June 2015. NBC was the first step toward filtering what data each client receives, reducing bandwidth by not sending updates for entities the player cannot interact with.

Architecture Evolution

2012
Kickstarter campaign. CIG begins development on what is then CryEngine 3 with 32-bit coordinates and traditional single-server architecture.
2015
64-bit conversion completed. ~8 months of work converts the engine to double-precision coordinates. Network Bind Culling proposed and implemented.
2016
CIG transitions from CryEngine to Lumberyard. Amazon's fork of CryEngine provides AWS integration and licensing benefits. Engine is increasingly customized into "StarEngine."
2018
Client-Side OCS ships (Alpha 3.3). First step toward server meshing. Clients can now dynamically stream entities in/out of memory, dramatically reducing RAM usage and improving FPS.
2019
Server-Side OCS ships (Alpha 3.8). Servers can now stream entities dynamically, paving the way for multiple servers to each hold a subset of the world.
2023
Persistent Entity Streaming ships (Alpha 3.18). EntityGraph database goes live. Every dynamic entity in a shard now persists. Dropped items, parked ships, and cargo survive across sessions.
2024
Static Server Meshing goes live (Alpha 4.0). Stanton and Pyro connected via jump points. Up to 500 players per shard across multiple DGS nodes. Replication Layer (Hybrid) deployed.
2024
Meshing stress tests. 800-player test in March, 1,000-2,000 player test in October. Testing reveals crash conditions and stability improvements needed for higher player counts.
2025
CitizenCon: Replication Layer 2.0 and Dynamic Mesh 2.0 announced. Plans to split the Hybrid into independent microservices and implement dynamic server boundary adjustment. Playability and stability focus declared for 2025.
