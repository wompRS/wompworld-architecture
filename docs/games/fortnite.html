Architecture Comparison / Fortnite

Fortnite

Dedicated Server Shards Battle Royale / Creative Sandbox

The world's most-played battle royale runs entirely on AWS, sharding 100 players per match across tens of thousands of dedicated Unreal Engine servers. Epic built a Replication Graph to keep 50,000 replicated actors synced without melting the CPU, and turned the whole thing into a platform that hosts concerts for 12 million concurrent viewers.

Overview

Fortnite Battle Royale drops 100 players onto an island where they loot weapons, build structures, and fight until one player or squad remains. Every match is a self-contained shard running on a dedicated server instance in AWS, with the server acting as the authoritative simulation. When a match ends, the server instance terminates and its resources return to the pool.

The game launched as Save the World, a PvE tower-defense mode, in July 2017. The free-to-play Battle Royale mode shipped two months later and rapidly became the most popular game in the world, hitting 125 million registered players by mid-2018 and peaking at 350 million by 2020. That growth forced Epic to solve scaling problems at a pace few studios have ever faced.

Fortnite's architecture is defined by three pillars: AWS cloud infrastructure providing elastic compute that scales up to 30x during peak events, Unreal Engine's Replication Graph that makes 100-player matches computationally feasible, and Epic Online Services (EOS) acting as the backend platform for matchmaking, accounts, analytics, and social features. The same EOS platform is now offered to third-party developers, battle-tested by Fortnite's scale.

Unreal Engine 5 Dedicated Servers

Each match runs a headless (no-rendering) Unreal Engine server process that simulates physics, validates actions, and replicates state to all 100 clients. The server is authoritative over the game simulation: player positions are predicted client-side but corrected by the server. Building placement, damage, loot spawns, and storm progression are all server-validated.

Replication Graph

Standard UE replication requires every replicated actor to check every connected client each tick. With 100 clients and 50,000 actors, that's 5 million evaluations per tick. The Replication Graph plugin replaces this with spatial hashing and priority sorting, so actors only replicate to clients that can actually see them. A dormant tree sends zero updates until a player hits it.

AWS Elastic Infrastructure

Fortnite runs on tens of thousands of Amazon EC2 instances powered by Graviton processors. The fleet auto-scales to handle demand spikes, supporting up to 30x normal load during events. Epic uses Elastic Kubernetes Service (EKS) for container orchestration, and AWS Local Zones for low-latency regional servers (like NA-Central).

Epic Online Services

EOS is the backend platform extracted from Fortnite's infrastructure. It handles authentication, matchmaking, friends lists, voice chat, analytics, anti-cheat integration, and cross-platform play. The system processes matchmaking requests through the Matchmaking Control Plane (MCP), routing players to the lowest-latency server available for their region and skill bracket.

Server Infrastructure

Epic went all-in on AWS in 2018, migrating the entirety of Fortnite's infrastructure to Amazon's cloud. The game runs on tens of thousands of EC2 instances powered by AWS Graviton2 ARM processors, which Epic adopted after proving their performance and cost efficiency at Fortnite scale. After validating Graviton2 through Fortnite, Epic rolled out support for the processor architecture to the broader Unreal Engine ecosystem.

Compute Fleet

Each Battle Royale match runs as an isolated dedicated server process on an EC2 instance. A single instance can host multiple match processes depending on the instance size. The fleet auto-scales based on demand using AWS Auto Scaling Groups, with capacity pre-warmed ahead of known events (patch days, season launches, live concerts) and dynamically adjusted for organic fluctuation. During peak events, the fleet scales to roughly 30x baseline capacity.

Container Orchestration

Epic uses Elastic Kubernetes Service (EKS) for orchestrating Fortnite's game servers and backend microservices. Kubernetes handles scheduling server processes across the fleet, health checking running instances, and spinning up replacements when containers fail. This was cited as a key dependency when players in underserved regions (like Africa) requested local servers, because EKS availability in a region is a prerequisite for Fortnite server deployment there.

Regional Coverage

Fortnite operates game servers across all major AWS regions worldwide. In 2023, Epic deployed a new NA-Central region using AWS Local Zones, placing compute closer to players in the central United States and Mexico. Local Zones are essentially micro data centers positioned near population centers, designed specifically for latency-sensitive workloads like game servers. This brought sub-30ms ping to millions of players who previously connected at 60-80ms to either coast.

Creative Mode: A Different Scale Problem

Creative islands are each hosted on their own dedicated server, separate from the Battle Royale fleet. When UEFN (Unreal Editor for Fortnite) launched, it massively expanded the number of player-created experiences, and Graviton2 proved critical for cost-effectively hosting the explosion of Creative server demand. Each island runs its own game loop with custom Verse scripting, and islands can support up to 100 simultaneous players depending on the experience's complexity.

Network & Protocols

Fortnite uses the standard Unreal Engine networking stack with significant custom modifications for Battle Royale scale. The server is authoritative over the game simulation, and clients communicate exclusively with the server (not with each other). All traffic flows over UDP for gameplay data, with HTTPS used for backend API calls (matchmaking, inventory, shop).

Server Authority Model

The dedicated server runs the canonical game simulation at 30 ticks per second. Clients send inputs (movement, shooting, building placement) to the server, which validates and executes them. The server then replicates the resulting state changes back to all relevant clients. Client-side prediction handles movement and weapon fire locally for responsiveness, with server corrections applied when the prediction diverges from the server's simulation.

Replication Graph

The Replication Graph is the single most important networking optimization in Fortnite. A Battle Royale match starts with 100 connected clients and approximately 50,000 replicated actors (players, weapons, buildings, trees, loot, vehicles, projectiles). The standard Unreal replication approach requires each actor to evaluate whether it should send updates to each client every tick. At this scale, that evaluation alone bottlenecks the server CPU.

The Replication Graph replaces per-actor client evaluation with a spatial data structure. Actors are categorized by type and behavior:

  • Always-relevant actors (the local player, game state) replicate to their owner every tick
  • Spatially-relevant actors (other players, vehicles) replicate only to clients within a relevance radius
  • Dormant actors (pre-placed trees, rocks, buildings) send zero updates until interacted with
  • Destroyed actors send a final destruction event and are removed from the graph entirely

This system means a tree in Wailing Woods never consumes network bandwidth until a player hits it with a pickaxe. When damaged, only clients that can see the tree receive the update. When destroyed, clients get a single destruction notification and the actor is removed from replication permanently.

Tick Rate History

Fortnite originally ran at 20 Hz when Battle Royale launched. In 2018, Epic increased the tick rate to 30 Hz after substantial server-side performance optimization and upgraded hardware. Processing 100 players with real-time building and destruction physics at higher tick rates (60 Hz or 128 Hz) remains computationally infeasible for the player count and map scale involved. For context, Valorant runs at 128 Hz but only supports 10 players per match.

Data & Persistence

Fortnite separates ephemeral match state from persistent player data. Match state lives entirely in the dedicated server's memory and is discarded when the match ends (unless a replay is saved). Persistent data (player profiles, inventory, V-Bucks balance, Battle Pass progression, cosmetics) is stored in MongoDB with sharding across the player base.

Matchmaking Control Plane (MCP)

The MCP is the backend service responsible for processing all matchmaking requests. It receives queue requests from clients, evaluates region, platform, input method, and skill bracket, and assigns players to a game server. During the infamous 3.4 million CCU spike in early 2018, MCP's MongoDB write queue backed up catastrophically because matchmaking writes queued indefinitely waiting for database resources, causing system-wide blocking. Epic's post-mortem identified multiple compounding failures: a thread pool misconfiguration reduced available MCP workers, a proxy timeout against Memcached blocked login requests, and the XMPP service's single load balancer became overloaded.

Replay System

Fortnite's replay system is built on Unreal Engine's DemoNetDriver. The driver captures the same network replication data that would be sent to a client, writing it to a file instead. Replays use the Local File Streamer, saving asynchronously to a single .replay file on local storage. The file contains baseline state, periodic checkpoints (snapshots of world delta from baseline), and incremental replication data between checkpoints. Replay data is compressed with Oodle, and checkpoint frequency defaults to every 30 seconds. The replay can be reconstructed at any point by loading the baseline, applying checkpoints up to the target time, then applying incremental changes.

V-Bucks Economy

V-Bucks (Vinder-Bucks) are managed entirely server-side. Client-side generation is architecturally impossible. The Item Shop catalog rotates daily at 00:00 UTC, with the rotation determined by server-side configuration. Purchases are validated against the player's V-Bucks balance in the backend before items are granted. Refunds are processed within a 24-hour window or via limited refund tickets. The Battle Pass costs 1,000 V-Bucks and awards back 1,500 V-Bucks through gameplay, creating a retention loop where active players can self-fund future passes.

MongoDB Sharding

Player profiles and inventory are distributed across MongoDB shards. During the 2018 scaling crisis, backed-up write queues in Mongo cascaded into matchmaking failures. Epic has since implemented aggressive write batching and connection pooling to prevent queue starvation under peak load.

Memcached Layer

Hot data like auth tokens, session state, and recently accessed profiles are cached in Memcached to avoid hitting MongoDB on every request. The 2018 outage revealed that proxy timeouts against Memcached could block the entire authentication pipeline, leading to a redesign of the caching proxy layer.

Scaling & Performance

Fortnite's scaling challenges are defined by two axes: match-level density (100 players in a single server process) and platform-level concurrency (millions of simultaneous players across the fleet). The solutions for each axis are fundamentally different.

Match-Level: 100 Players

Each server process handles exactly one match. The Replication Graph keeps CPU usage manageable by eliminating unnecessary actor evaluations. Physics simulation (Chaos in UE5) runs at the server's tick rate for building/destruction events. The storm circle progression is a simple server-side timer with geometric shrinkage, computationally trivial compared to the replication workload. The actual bottleneck is the late-game scenario where many players are in close proximity and most actors become spatially relevant to most clients simultaneously.

Platform-Level: Millions of CCU

Horizontal scaling solves the fleet problem. More players means more match server instances, and AWS auto-scaling handles this linearly. The harder problem is the backend services that every player touches regardless of which match they're in: authentication, matchmaking, friends list, item shop. These services must scale independently of the game server fleet and handle thundering-herd scenarios (millions of players logging in simultaneously after a patch deploys).

The Travis Scott Stress Test

In April 2020, rapper Travis Scott performed a virtual concert inside Fortnite that drew 12.3 million concurrent players for the premiere showing (27.7 million unique across five showings). Each shard was capped at 50 players (down from the usual 100) due to the visual complexity of the event. The concert was a pre-programmed sequence broadcast synchronously to all shards, with the same choreography, particle effects, and music playing in lockstep. The reduced player cap per shard meant roughly 246,000 simultaneous server instances running the event. Over 3 million concurrent Twitch/YouTube viewers watched alongside the in-game audience.

Cross-Platform Matchmaking

Fortnite supports cross-platform play across PC, PlayStation, Xbox, Nintendo Switch, and mobile. The matchmaking system factors in input method (controller vs. keyboard/mouse) in addition to platform, region, and skill rating. Controller players are grouped together by default, with opt-in for mixed-input lobbies. This creates a matrix of matchmaking pools that the MCP must balance: filling matches quickly while maintaining fair input-method matching and acceptable latency bounds.

Technical Deep Dives

Building & Destruction Sync

Building is the mechanic that sets Fortnite apart from other battle royales, and it's the single most expensive networking operation in the game. When a player places a wall, the server validates the placement (collision check, material cost), spawns the building actor, and replicates it to all clients within relevance range. Destruction works similarly: damage events are validated server-side, and when a structure's health reaches zero, a destruction event is replicated. The Replication Graph handles this efficiently because buildings are spatially indexed, and only clients who can see the building receive updates.

The real challenge is cascading destruction. Destroying a floor piece can cause the wall and roof pieces above it to lose structural support and collapse. The server must resolve these dependency chains in a single tick to prevent desync where one client sees the structure standing while another sees it collapsed. Epic handles this with a structural integrity graph that evaluates connected pieces when any piece is destroyed.

Live Events Architecture

Live events (concerts, story finales, seasonal transitions) are the most extreme scaling scenarios Fortnite faces. The architecture treats events as pre-programmed sequences that are loaded as level content and triggered by a server-side timer. All shards receive the same trigger simultaneously, and the sequence plays out identically in every instance. Player interaction during events is typically limited (no building, restricted combat) to reduce the unpredictable state that would cause shards to diverge.

The Marshmello concert in February 2019 (10.7 million concurrent) was essentially a programmed character on a stage with looped animations and a lightshow. The Travis Scott "Astronomical" event in April 2020 was orders of magnitude more complex: the stage disappeared, players were transported through multiple environments with altered gravity, and the avatar appeared to be motion-captured rather than looped. Epic reduced the shard cap from 100 to 50 players to accommodate the visual complexity.

EasyAntiCheat Integration

Fortnite uses EasyAntiCheat (EAC), which Epic acquired in 2018 and subsequently made free for all developers through EOS. EAC operates as a client-side agent that monitors the game process for unauthorized memory manipulation, injected DLLs, and known cheat signatures. It runs alongside the game process (not as a kernel driver like Vanguard or RICOCHET) and reports violations to Epic's backend. Server-side behavioral analysis supplements EAC by flagging statistically improbable gameplay patterns (impossible headshot rates, movement anomalies) for manual review or automated action.

UEFN and Verse

Unreal Editor for Fortnite (UEFN) allows creators to build custom game experiences using Unreal Engine tools and publish them directly into Fortnite. Each UEFN island runs on a dedicated server with its own game loop. Creators script gameplay using Verse, a new programming language developed by Epic that compiles to the Fortnite runtime. Verse is deliberately constrained compared to C++ or Blueprints: it prevents creators from writing code that could crash the server, degrade performance for other players, or access unauthorized systems. The language enforces memory safety and computation budgets per tick.

Architecture Evolution

2017 Save the World launches (July), followed by Battle Royale (September). Initial infrastructure on a mix of cloud providers; server tick rate is 20 Hz. The backend is a traditional monolithic architecture that will soon buckle under exponential growth.
2018 3.4M CCU spike crashes the backend. MongoDB write queues, Memcached proxy timeouts, and XMPP load balancer failures combine into a cascading outage. Epic begins wholesale migration to AWS. Server tick rate increases from 20 Hz to 30 Hz. Player count hits 125 million registered accounts. Epic acquires EasyAntiCheat.
2019 Marshmello concert (10.7M concurrent) proves the live event model. Epic Online Services announced at GDC, extracting Fortnite's backend into a free platform for all developers. Replication Graph improvements continue as map sizes and actor counts grow.
2020 Travis Scott's Astronomical (12.3M concurrent, 27.7M unique). 350 million registered players. Fortnite goes all-in on AWS with Graviton2 ARM processors for game servers and voice chat. The Galactus event hits 15.3 million concurrent, the all-time record at that point.
2021 Unreal Engine 5 transition begins. Chapter 3 introduces Nanite and Lumen to Fortnite, requiring server-side adjustments for new geometry streaming and destruction physics. Creative mode continues to grow as a platform within a platform.
2023 UEFN launches, turning Fortnite into a full game creation platform. Verse programming language ships. Creative server demand explodes, with Graviton2 critical for cost-effective hosting. AWS Local Zones deployed for NA-Central servers. 650 million registered players.
2024 Remix: The Finale concert hits 14.3M concurrent players, plus 3 million concurrent Twitch/YouTube viewers. Fortnite OG (Chapter 1 map return) drives massive re-engagement. The platform now hosts tens of thousands of Creative islands simultaneously alongside the Battle Royale fleet.
