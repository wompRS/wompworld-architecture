Ashes of Creation

Server Meshing Dynamic Node MMO

Overview

Ashes of Creation is a node-driven MMO developed by Intrepid Studios, founded by Steven Sharif with significant personal investment. The game's central premise is that player activity shapes the world: as players quest, gather, and fight in a region, the local "node" levels up from wilderness to expedition to encampment to village to town to city and finally to metropolis. Only five metropolises can exist per server, each with unique superpowers based on its type (Economic, Military, Scientific, or Divine).

The technical challenge behind this design is massive. A 480 km2 overworld plus an additional 100 km2 Underrealm must be simulated as a single seamless space with no loading screens, no sharding, and no layering. To accomplish this, Intrepid built IntrepidNET, a custom networking backend that sits between the game logic and Unreal Engine 5's built-in networking. IntrepidNET implements server meshing, where the world is divided into a grid of cells, each managed by a separate game server "worker." Players move seamlessly between workers with inter-server replication handling the handoff.

The most ambitious piece of the architecture is dynamic gridding: when a worker becomes overloaded with players (like during a siege), it can split into smaller workers. When players disperse, workers recombine. This allows the system to concentrate compute resources where the action is, rather than statically partitioning the world and hoping the load stays balanced.

Ashes of Creation entered Steam Early Access on December 11, 2025, reached a peak of 31,882 concurrent players, and was removed from Steam on February 2, 2026 following the resignation of Steven Sharif and subsequent studio closure. The technology it built, particularly the IntrepidNET server meshing approach, remains one of the most detailed publicly documented attempts at solving the MMO server density problem.

Server Infrastructure

The core innovation in Ashes of Creation is that a single "realm" (what players call a server) is not a single process. It is a mesh of many game server workers, each responsible for a geographic slice of the world. All workers combined form one seamless world with no loading screens, no channels, no layers, and no sharding.

Server Workers

Each worker manages a rectangular cell of the game world. At their largest, workers cover a 5 km by 10 km area. At their smallest (when split by dynamic gridding), they shrink to 500m by 500m. Each worker runs its own game simulation loop, its own AI, its own physics, and its own multi-threaded replicator for sending state to connected clients.

During Alpha-2 spot testing, a single worker was able to accommodate 300 players alongside approximately 2,500 NPCs while maintaining roughly 30 ticks per second on the server simulation. Note that the server tick rate is distinct from client frame rate. The server can tick at 30 Hz while individual clients render at 60, 120, or 144 FPS based on their hardware.

IntrepidNET

IntrepidNET is the custom networking layer that Intrepid Studios built to sit between the game logic and Unreal Engine 5's built-in networking. The upgrade to UE5 (from the original UE4 build) was primarily a front-end change: rendering, animation (motion warping, Lumen, Nanite), and client-side systems. The backend networking code (IntrepidNET) remained largely unaffected by the engine migration.

IntrepidNET handles three critical functions:

  • Inter-server replication: Seamlessly replicates entity data between workers within a realm so that a player standing on a worker boundary can see entities simulated by the neighboring worker.
  • Custom replication graph: Optimizes what data each player receives based on relevance. As player density increases, the system reduces replication distances (from 150m down to 60m in low-performance mode) to maintain throughput.
  • Worker handoff: When a player crosses from one worker's geographic cell into another, IntrepidNET handles the seamless transfer of that player's entity and connection from one server process to another.

No Sharding, No Layers

Intrepid was explicit that there will be no channels, layers, or shards on server realms. Every player on a realm exists in the same single instance of the world. This is a fundamentally different approach from games like World of Warcraft (which use sharding/phasing) or Guild Wars 2 (which use overflow maps). The tradeoff is that server meshing and dynamic gridding must actually work under extreme load, because there is no fallback mechanism to split players into parallel copies of the same zone.

Server Workers

Network & Protocols

IntrepidNET's approach to networking builds on Unreal Engine's native replication system but replaces major components to handle MMO-scale player counts. The key insight is that Unreal's default networking was built for games with 64 or 128 players, not thousands. Intrepid had to build custom solutions for relevancy, replication throughput, and cross-server communication.

Relevancy and Culling

The IntrepidNET replication graph determines what data each connected player receives. Under normal load, the net relevancy distance (how far away an entity can be and still be replicated to a player) is 150 meters. When a worker enters low-performance mode due to high player density, this distance drops to 60 meters. This aggressive culling is essential for maintaining throughput during sieges where hundreds of players concentrate in a small area.

There are also significance-based optimizations. Entities that are close to the player and moving toward them get higher replication priority than distant, stationary entities. Certain data is always replicated regardless of distance, like directionality of nearby players (so you can see which way someone is facing even at reduced detail).

Default Appearance Fallback

During large-scale PvP events like sieges, the system may apply a default player appearance automatically. Instead of replicating the full equipment and cosmetic data for every visible player (which is expensive), the client renders a standardized appearance for distant players. This reduces the replication payload and the client-side rendering cost simultaneously.

SpatialOS: The Road Not Taken

Intrepid Studios was once associated with Improbable's SpatialOS platform, which provides a similar concept of distributing a game world across multiple servers. However, Ashes of Creation does not use SpatialOS. IntrepidNET is a fully proprietary system built in-house by Intrepid's engineering team. The decision to build custom rather than license SpatialOS gave Intrepid full control over the networking stack, but it also meant years of development time on infrastructure before the game itself could be feature-complete.

Data & Persistence

The Node System

Every piece of persistent data in Ashes of Creation connects to the Node system. The world contains a fixed set of nodes, each with a predetermined type (Economic, Military, Scientific, or Divine) that never changes. What does change is the node's level, which progresses through 7 stages based on player activity in its Zone of Influence.

The server must track XP accumulation per node in real time. Every quest completed, every mob killed, every resource gathered within a node's zone contributes to its XP pool. When a node levels up, the server triggers a cascade of state changes: NPC spawns change, buildings appear, services become available, the zone's visual appearance transforms, and the node's relationship to neighboring nodes updates.

Node Levels

  • Stage 0 (Wilderness): Default state at launch. No NPCs, no structures, no services.
  • Stage 1 (Expedition): A small group of NPCs and companions appear.
  • Stage 2 (Encampment): Basic services and small structures.
  • Stage 3 (Village): First stage where the Vassal system activates. Nearby nodes become vassals.
  • Stage 4 (Town): Permanent structures, societies, and organizations with ranking systems.
  • Stage 5 (City): Temples, guild halls, embassies. Relics can appear in city vaults.
  • Stage 6 (Metropolis): Maximum level. Only 5 can exist per server. Unlocks rarest dungeons and world functions.

Vassal System

Starting at Stage 3, when a node advances, it "enslaves" nearby nodes, making them vassals. Vassals must always be at least one stage below their parent node. A vassal cannot level up past its parent's stage minus one. Excess XP from a capped vassal flows upward to the parent node. Each Metropolis can have up to 13 nodes in its vassal hierarchy, for a total of 65 nodes inside the vassal system across all 5 Metropolises.

Metropolis Superpowers

Each node type grants a unique realm-wide ability at Metropolis stage:

Economic Metropolis

Normally, marketplaces are local to each node. An Economic Metropolis links its marketplace with all vassal nodes, creating a much larger unified market that spans its entire zone of influence. This is a massive economic advantage, centralizing trade across a wide region.

Scientific Metropolis

Grants teleportation between the metropolis and any of its vassal nodes regardless of stage, as long as the vassal is not at war. If multiple Scientific Metropolises exist on a server, an airship provides fast travel between them. In a game with no other fast travel, this is transformative.

Military Metropolis

Reduces the duration of the Corruption debuff (earned by killing non-combatant players). Intrepid discussed a cross-server "hero" competition for specific archetypes as a potential additional feature for Military Metropolises.

Divine Metropolis

At City stage, may unlock a procedurally generated "mega catacomb" dungeon beneath the settlement that connects to its divine vassal nodes. These dungeons can house unique bosses with exclusive drop tables.

Government Persistence

Each node type has a different method for electing its mayor, and the server must track all of it:

  • Scientific: Democratic election. The server runs a voting period and tallies citizen votes.
  • Divine: The citizen who accumulates the most PvE "favor" through religious quests becomes mayor.
  • Economic: Blind-bid auction. The citizen who bids the most gold wins. All bids are hidden until resolution.
  • Military: Gladiatorial combat. Citizens fight in an arena, and the winner becomes mayor.

Scaling & Performance

Dynamic Gridding

Dynamic gridding is the most technically ambitious feature of IntrepidNET. Under static server meshing, each worker owns a fixed geographic cell. If a siege happens in Worker C's territory and 500 players show up, Worker C must handle all of them. Dynamic gridding solves this by allowing Worker C to split into smaller workers. The overloaded cell subdivides, and new server workers spin up to handle the partitioned areas.

When a worker gets overloaded, the first response is to enter "low performance mode": reduce replication distances from 150m to 60m, reduce the significance of distant entity updates, and prioritize critical game state. If load remains too high, the worker splits. The minimum split size is 500m by 500m. When players disperse and the load drops, the split workers recombine into their original configuration.

Dynamic gridding was expected to be tested on the PTR in July 2025. Before its implementation, server population was estimated at 8,000-10,000 CCU per realm. With dynamic gridding, the target was up to 50,000 peak concurrent users per realm. The Shol realm reached 8,100 CCU on May 5, 2025, before dynamic gridding was available.

Siege Architecture (250v250 Target)

Castle and node sieges are the peak stress events for IntrepidNET. The design target was 250 attackers versus 250 defenders in a single geographic area, with the possibility of 500v500 at "randomly different locations simultaneously." This requires multiple workers to jointly simulate the siege area through dynamic gridding, with inter-server replication handling entities near cell boundaries.

During sieges, additional optimizations kick in: default player appearances replace customized gear visuals, replication distances shrink, and non-essential NPC AI may be suspended in the siege zone to free up simulation cycles for player combat.

Crash Recovery

If a game server worker crashes, it can be replaced automatically. Players connected to the crashed worker must relog, but players on other workers in the same realm are unaffected. This is a significant advantage over monolithic server architectures where a server crash disconnects every player. In a meshed architecture, a crash affects only the players in one geographic cell.

The Caravan System

Caravans are mobile economic targets that create rolling PvP zones in the open world. A caravan can carry 100 times what a player's backpack holds, making them critical for supplying nodes with building materials and for long-distance trade. When a caravan launches, it becomes a sanctioned PvP event: anyone can attack without suffering the Corruption penalty that normally applies to killing non-combatants.

From an architecture perspective, caravans are moving entities that cross worker boundaries. As a caravan travels from one worker's cell into another, IntrepidNET must seamlessly hand off the caravan entity, all its cargo data, and the combat state of any players fighting around it. The caravan's position is only visible on the world map to its owner and raid group, adding a fog-of-war element that reduces the data the server must broadcast to all players.

The Underrealm Problem

The Underrealm is a 100 km2 subterranean layer beneath the overworld. From a server meshing perspective, it requires its own separate grid of workers operating on a different vertical plane. The overworld grid is a 2D slice across the surface. The Underrealm needs its own 2D grid at a different elevation, with connection points (entrances/exits) where entities transfer between the two grids. Intrepid acknowledged this requires "additional tech development on the networking side" and that Underrealm workers were not yet fully integrated when early access launched.

Technical Deep Dives

Node Destruction and World Reshaping

When a node is destroyed through a siege, it does not simply reset to Stage 0. The destruction cascades through the vassal hierarchy. All vassal nodes are released from their vassal bonds. The destroyed node drops to Stage 0, and all the buildings, NPCs, services, and visual changes that came with its higher level disappear. Players who owned freeholds (housing) near the node lose their placement. The dungeon content unlocked by the node locks again.

This means a successful siege against a Metropolis can reshape a massive region of the server. The server must handle a cascade of state deletions (buildings despawning, NPCs despawning, services disabling), vassal relationship recalculations, node XP redistribution, and housing displacement for potentially hundreds of players, all in a short window.

Naval Architecture

Naval content includes ship construction from components, water navigation, ship-to-ship combat, and ship-to-player combat. Ships cannot be permanently captured (only their cargo can be looted after destruction). Raft caravans transition between land and water at coastlines through a 1-2 minute construction timer that creates a vulnerability window.

The naval system places ships on the same server mesh as land content. There are no separate "ocean servers." A ship sailing along the coast will cross worker boundaries just like a character walking on land, and IntrepidNET handles the handoff the same way. NPC ships also populate the ocean as PvE targets, adding to the simulation load on coastal workers.

Anti-Cheat

Ashes of Creation uses Easy Anti-Cheat (EAC), a kernel-level anti-cheat solution. EAC runs as a background process with elevated system access, which drew criticism from the community over privacy and performance concerns. Players reported EAC-related launch failures, random disconnects, and validation errors throughout the early access period.

The server-authoritative nature of IntrepidNET provides a natural layer of cheat prevention: the server simulates all game state, and client inputs are validated against server-side rules. However, EAC provides an additional layer against memory manipulation, speed hacks, and other client-side exploits that server validation alone cannot catch.

Economy and Marketplace Architecture

Marketplaces are local by default. Each node has its own marketplace where only citizens and visitors of that node can buy and sell. This design creates geographic economic fragmentation that drives trade caravans: a resource abundant in one region must be physically transported to sell it in another. The server must maintain independent marketplace inventories, price histories, and order books per node.

The exception is the Economic Metropolis superpower, which unifies all vassal node marketplaces into a single linked market. This is a massive database operation: all buy/sell orders across potentially 13 nodes must be queryable as a single order book, with transactions settling across node boundaries.

Architecture Evolution

2016 Kickstarter campaign raises $3.2 million. Steven Sharif founds Intrepid Studios. Initial development begins on Unreal Engine 4.
2017 Alpha-0 testing. Basic gameplay systems tested on a traditional single-server architecture. The Node system concept is demonstrated in early form.
2018 Alpha-1 begins. SpatialOS explored as a potential networking solution. Intrepid eventually decides to build its own custom networking stack instead.
2020 IntrepidNET development ramps up. Custom server meshing architecture takes shape. The team recognizes that UE4's built-in networking cannot scale to MMO player counts.
2021 Unreal Engine 5 migration announced (December). Front-end moved to UE5 for Nanite, Lumen, and improved animation. IntrepidNET backend remains largely unchanged.
2024 Alpha-2 testing with server meshing. First large-scale tests of IntrepidNET under real player load. Server meshing technology preview shown in June livestream. 300-player stress tests with 2,500 NPCs at 30 Hz.
2025 Q2 Phase 3 of Alpha-2 launches. Two 24/7 realms: Shol (US East) and Lotharia (EU West). Shol reaches 8,100 CCU on May 5. Dynamic gridding targeted for July PTR testing.
2025 Q4 Steam Early Access launch (December 11). 31,882 peak CCU. Mixed reviews. Over 200,000 new players in the first week. Technical issues with login, account linking, and server stability.
2026 Q1 Studio closure. Steven Sharif resigns (January 31). WARN Act notices filed indicating permanent closure of Intrepid Studios. Game removed from Steam (February 2). The IntrepidNET technology and Node system survive only in documentation and community archives.
Relevance to Wompworld

Ashes of Creation's IntrepidNET system is directly relevant to Wompworld's design challenges. Both games need to support large-scale PvP (naval battles vs. sieges), both have player-driven economies with physical cargo transport (trade ships vs. caravans), and both aim for a single-shard world without instancing. The dynamic gridding approach, where server workers split under load and recombine when density drops, is particularly applicable to Wompworld's ocean world where player density will naturally cluster around island settlements and trade routes. The lesson from Ashes' closure, however, is sobering: even with years of development and significant funding, getting server meshing to production quality is extremely difficult.
