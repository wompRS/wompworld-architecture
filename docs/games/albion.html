Architecture Comparison / Albion Online

Albion Online

Single-Shard Cluster Sandbox MMORPG

A cross-platform sandbox MMO built on Unity and Photon Server, where every player shares one persistent world. Its single-threaded zone architecture, Apache Cassandra database, and fully player-driven economy make it one of the closest modern analogues to EVE Online's design philosophy, applied to a medieval fantasy setting with full-loot PvP.

Overview

Albion Online is built around a principle that CTO David Salz has stated plainly: there is only a single game world shared by all players on all platforms. No separate shards, no separate servers, not even separate instances for different platforms. A player on an Android phone, a player on a Mac, and a player on a Windows PC all exist in the same world, fighting over the same resources, and trading on the same markets.

The game world is divided into roughly 600 zones ("clusters"), each approximately 1x1 km in size. Each cluster is a self-contained simulation running on a game server. When a player crosses a zone boundary, their client disconnects from one game server and reconnects to another, producing a brief loading screen. This is the fundamental unit of Albion's architecture: one cluster, one game server process, one single-threaded simulation.

The tech stack is pragmatic rather than bespoke. The client runs on Unity, chosen because it publishes to Windows, Mac, Linux, Android, and iOS with minimal per-platform effort. The server backend is written in C#, running on the Photon Server framework, which provides low-level networking and a C++ core with C# application code on top. The database layer uses Apache Cassandra for high-throughput key-value storage and PostgreSQL for query-heavy features like the marketplace. Hosting uses bare-metal dedicated machines rather than virtual instances, chosen for guaranteed CPU time and memory.

The architecture makes deliberate tradeoffs. Single-threaded cluster simulation means that ZvZ (Zerg vs. Zerg) battles with hundreds of players create genuine performance bottlenecks. There is no Time Dilation system like EVE Online. Instead, Albion uses cluster queue caps (currently ~350 players per zone) and has been incrementally optimizing spell processing and network event systems to push those limits higher. The honest reality is that this is the hardest unsolved engineering problem in the game.

Single World, All Platforms

One world for Windows, Mac, Linux, Android, and iOS. No platform-specific servers. Characters transfer seamlessly between devices. This is possible because the client is a thin view layer, and the server has absolute authority over all game logic.

Cluster-Based Zones

~600 zones, each 1x1 km. Each cluster runs as an independent simulation on a game server. Zone transitions produce brief loading screens. A single cluster can contain 300+ players, 500+ monsters, and 10,000+ interactive objects.

Full-Loot PvP

Death in PvP zones means losing your entire inventory to attackers. This design requirement means the server must be absolutely authoritative. Client-side prediction is minimal because cheating in a full-loot game has devastating consequences for victims.

Player-Driven Economy

Nearly every item in the game is crafted by players, in player-built crafting stations, from player-gathered resources. The developers do not sell items. They act only as a central bank exchanging currencies. Local marketplaces with region-specific pricing create trade route opportunities.

Server Infrastructure

Albion Online runs on bare-metal dedicated servers rather than cloud VMs. Sandbox Interactive made this choice to guarantee consistent CPU time and memory, which matters when your game simulation is single-threaded and performance-sensitive. The original hosting used Softlayer (now IBM Cloud), with 8 machines running 10-Core Intel Xeon processors and 32 GB RAM each, targeting approximately 15,000 concurrent users.

The server farm is managed by a central BackOffice Server that controls startup, shutdown, and provides an API for communicating with the entire farm. It acts as the single point of access for all other servers in the deployment. Sandbox Interactive initially tried hosting across 2 data centers for better ping distribution, but abandoned the multi-datacenter approach due to inter-server disconnects and latency issues between sites.

As the game grew (reaching 350,000 daily active users by May 2024 with the Albion Europe launch), the infrastructure has scaled substantially beyond those original 8 machines. The game now operates regional server deployments for Americas, Asia, and Europe, each running as a separate world. Asset delivery uses Gcore's CDN, which Sandbox Interactive credits with reducing infrastructure expenses by 40%.

Game Servers

Run the cluster simulations. Each game server hosts multiple zones. The game logic per zone is single-threaded, with async thread pools for database I/O, pathfinding, and logging. All incoming player commands enter a synchronized event queue and are processed sequentially.

World Server

The single authority for all cross-server features: parties, guilds, GvG battles, territory ownership, and global events. When a feature needs to coordinate across multiple game servers, the World Server handles it. This is a single point of coordination, not a single point of failure.

Marketplace Server

Handles all buy and sell orders across the game's local marketplaces. Backed by PostgreSQL for complex order matching queries. Prices vary by location, creating economic incentives for trade routes. A separate Gold Market server handles the premium currency exchange.

Chat & Login

The chat server runs on TCP, separate from game traffic. The login server handles authentication and initial routing to a game server. Both communicate with the BackOffice server for farm-wide coordination.

Instance Servers

Player islands, guild islands, and instanced content like Corrupted Dungeons are dynamically created as needed. The World Server coordinates instance creation, while the actual simulation runs on available game server capacity.

Statistics & Ranking

Separate servers handle leaderboard calculations, kill statistics, and seasonal rankings. These are read-heavy workloads that would waste game server CPU if co-located with combat simulation.

Network & Protocols

Albion Online uses Photon Server as its networking middleware. Photon provides a C++ core with C# application code on top, giving Sandbox Interactive the ability to share code between the Unity client and the server backend. They use Photon at the lowest possible level: only the server framework and the network protocols. Photon also offers a cloud hosting solution, a chat system, and an MMO framework, but Sandbox Interactive chose to use none of those, building their own implementations instead.

Most game traffic uses Photon's reliable UDP protocol. Movement messages are the exception: they use unreliable UDP because a missed movement update is immediately superseded by the next one. Chat and inter-server communication use TCP for guaranteed delivery. Message content is mostly simple binary packets, with JSON used only in rare cases.

The architecture follows a strict server-authoritative model. All game world manipulations happen on the server. The client does validity checks and sometimes prediction, but ultimately sends all requests to the server for final authority. The client is essentially a view layer. Sandbox Interactive even maintains a fully functional command-line client (no Unity, no graphics) used for load testing. This proves the point: the game logic is entirely server-side, and the client is just a display.

Data & Persistence

Albion uses a dual-database strategy: Apache Cassandra for high-throughput game state storage and PostgreSQL for query-heavy features that need relational capabilities.

Cassandra was chosen because it is a distributed hash table that scales horizontally. You can add server nodes to the Cassandra cluster, and it automatically redistributes the hash space with redundancy. Multiple nodes can fail and the cluster stays operational. Data is retrieved by hash key, with optional secondary indexes on additional columns. It is in-memory as much as possible and optimized for high throughput.

The tradeoff is that Cassandra is only "eventually consistent." Due to its distributed nature, it is possible to read outdated data. This can be fixed by reading from a quorum of nodes (slower reads) or writing to a quorum (slower writes). Albion Online chose the fast-writes approach: game servers keep player data in memory while a player is logged in, and write state to Cassandra frequently but accept that reads might occasionally be stale. This works because the authoritative state lives in the game server's memory, not in the database.

PostgreSQL handles the marketplace, where order matching queries need proper relational operators (joins, range queries, sorting by price). Cassandra's key-value model is not suited for these kinds of queries.

Zone transitions use the database as a handoff mechanism. When a player moves between zones, the source game server writes their complete state to Cassandra, and the destination game server reads it. This approach works even when both zones happen to run on the same physical machine, keeping the architecture uniform.

The Cassandra Incident

During a closed alpha test, all player-built structures in the game disappeared. The cause was a Cassandra bug involving secondary index loss. The secondary indexes that tracked building ownership were silently corrupted, and without those indexes, the game could not find the buildings. This is the kind of risk that comes with using bleeding-edge open source technology for game persistence. Sandbox Interactive has since implemented additional safeguards and monitoring, but the incident is a cautionary tale about trusting NoSQL secondary indexes for critical game state.

Apache Cassandra

Primary game state storage. Player inventories, character data, building ownership, territory state. Distributed hash table with horizontal scaling. Eventually consistent with fast writes. Originally developed by Facebook, open source under Apache License.

PostgreSQL

Marketplace and Gold Market data. Order matching requires relational queries that Cassandra cannot efficiently handle. Separate database instances for marketplace and gold market to distribute query load.

In-Memory Game State

While a player is logged in, their authoritative state lives in the game server's memory. The database serves as permanent storage and handoff mechanism. This design means fast-writes to Cassandra are acceptable because the in-memory state is always current.

Zone Transition Handoff

Player crosses zone boundary. Source server writes full player state to Cassandra. Client disconnects and reconnects to destination server. Destination reads state from Cassandra. Brief loading screen covers the transition. Works identically for same-machine and cross-machine transitions.

Scaling & Performance

Albion's biggest performance challenge is ZvZ (Zerg vs. Zerg) battles. When hundreds of players converge in a single zone, the single-threaded game simulation becomes a bottleneck. Unlike EVE Online, which slows time with TiDi, Albion has no mechanism to trade real time for simulation accuracy. Instead, the game uses cluster queue caps to hard-limit zone population.

When approximately 350 players enter a zone, the zone becomes "cluster queued." The zone is locked, low Item Power (IP) players inside the zone can be kicked out, and higher IP players outside can queue to enter. This is a harsh but effective mechanism: it prevents the simulation from being overwhelmed at the cost of excluding some players from the fight.

The underlying problem scales non-linearly. An area-of-effect spell hitting 50 opponents generates over a thousand individual checks per cast (damage calculation, resistance evaluation, buff/debuff application, etc.). With 300+ players casting spells simultaneously, the event queue fills faster than the single thread can drain it. Sandbox Interactive's Technical Director has identified two primary bottlenecks: the spell processing system and the network event system that broadcasts results to clients.

Interest Management

To keep network traffic manageable, Albion uses spatial hashing for interest management. All objects in a cluster are placed into a grid of approximately 10x10 meter cells. Each cell maintains a list of objects and fires events when objects enter or leave. Each player has an "Interest Area" consisting of two grid-aligned rectangles: a smaller inner rectangle for detailed updates and a larger outer rectangle for presence awareness. Only objects within a player's Interest Area generate network events for that player. This prevents the server from broadcasting every action to every player in the zone.

No Traditional Game Loop

Albion does not use a traditional game loop with a fixed-rate Update() call on every object. Instead, game objects create timers when they need periodic processing. A monster creates a recurring AI update timer when a player enters its detection range and deletes the timer when all players leave. A spell creates a timer for its duration. A crafting station creates a timer for the crafting time. This event-driven approach means idle objects consume zero CPU, which is important when a single cluster contains 10,000+ interactive objects but only a fraction are active at any moment.

Technical Deep Dives

Single-Threaded By Design

The decision to run each cluster's game logic on a single thread is deliberate. Making it multi-threaded would be error-prone and not necessarily more efficient, because the complex interactions between game objects (a spell hitting a player wearing equipment that has passive effects that trigger abilities that affect nearby allies) would require extensive locking. The performance cost of acquiring and releasing locks, plus the debugging cost of deadlocks and race conditions, would likely offset any parallelism gains. Instead, all incoming player commands enter a synchronized event queue and are processed in strict order of arrival. Work that can be offloaded (database operations, pathfinding, logging) goes to separate thread pools that complete asynchronously and post results back to the event queue.

Server Authority in a Full-Loot Game

Albion's full-loot PvP rules make server authority non-negotiable. If a player dies in a red or black zone, everything in their inventory is dropped and can be looted by the attacker. A client-side cheat that prevents death or duplicates items would be catastrophic. The server validates every action: movement speed, damage dealt, items in inventory, cooldown timers, and position. The client does basic prediction for responsiveness (showing movement immediately before server confirmation), but any discrepancy is resolved in the server's favor. Sandbox Interactive acknowledges that any information sent to the client can and will be extracted by cheaters, so they restrict what the client knows about to objects on-screen or very close to it.

Cross-Platform Architecture

The ability to run on five platforms (Windows, Mac, Linux, Android, iOS) comes from two architectural decisions. First, Unity handles the cross-platform rendering and input. Second, and more importantly, the Albion client is structured as a library (a DLL) that communicates with the server. The client mirrors the part of the data model visible to the player and provides a controller interface similar to the server's own. This means the "game" is the server. The client is just a viewport. The command-line test client proves this: you can interact with the full game simulation without any graphics, running through a terminal. Shared C# code between client and server (via Photon) reduces platform-specific bugs.

Guild Territories and Hideouts

Guild territories in the Outlands are controlled through GvG combat, managed globally by the World Server. Territory ownership determines access to high-tier resources and provides passive bonuses. Hideouts are persistent guild bases that exist as separate instanced maps within a zone. A guild can have up to 6 hideouts across the world but no more than 1 per zone. Hideouts can only be placed in zones with a watchtower territory in the Outlands, or in Avalonian Rest zones in the Roads of Avalon. They have defense points that protect them from attack, and are invulnerable most of the day. If all defense points are destroyed, the hideout is lost and all contents are destroyed, except items in players' personal banks, which transfer to the nearest public bank (claimable after 4 weeks, with a fee).

Anti-Cheat in a Full-Loot World

Albion has gone through two anti-cheat solutions. Easy Anti-Cheat (EAC) was added in February 2019 and removed in July 2023. BattlEye replaced it. The community strongly opposes bots and cheaters, and in a full-loot game the stakes are higher than in a conventional MMO. A bot farming high-tier resources in a safe zone devalues the time of legitimate gatherers. A speed hack in PvP means someone loses their entire gear set to a cheater. Sandbox Interactive's position is that banning bots completely is the only viable choice because the line between botting and cheating is too thin to enforce any lesser policy.

The DDoS War

During early access or beta (the exact timing is partially obscured), Albion Online was hit with sustained DDoS attacks exceeding 60 Gbps for over 4 consecutive weeks. The team describes sleepless nights and an upset community before they secured external mitigation services. This experience informed their infrastructure decisions going forward, including investment in DDoS protection and the eventual partnership with Gcore for CDN and infrastructure services. For an indie studio running a single-shard game, a successful DDoS takes down the entire world for every player, making it a higher-impact attack than against a sharded game where only one shard might go down.

Architecture Evolution

2012Founded: Sandbox Interactive founded in Berlin by David Salz and others. Small team begins building an MMO with Unity client and C# server on Photon.
2014Alpha testing: Hosted on Softlayer bare-metal machines. 8 servers, 10-core Xeon, 32 GB RAM. Cassandra chosen for distributed persistence. The building-disappearance bug caused by Cassandra secondary index corruption.
2016Quo Vadis talk: David Salz presents "Software Architecture of an MMO" at Quo Vadis conference in Berlin. Detailed public disclosure of tech stack, threading model, database strategy, and design philosophy.
2017Launch (July 17): Albion Online goes live with a single server for all platforms. Approximately 250,000 players at launch. DDoS attacks exceeding 60 Gbps for 4+ weeks force emergency mitigation purchases.
2019Free-to-play: Game goes free-to-play in April 2019, massively expanding the player base. Easy Anti-Cheat (EAC) added in February. Mobile (Android) launch.
2021Stillfront acquisition / Hideouts: Sandbox Interactive acquired by Stillfront Group. Hideout system introduced, adding persistent guild bases with defense mechanics and territory integration. iOS version launched.
2023Albion Asia / BattlEye: Albion Asia server launches (April), breaking 300,000 DAU. Easy Anti-Cheat replaced by BattlEye (July). ZvZ performance optimization becomes a stated engineering priority.
2024Albion Europe / 350K DAU: European server launches (April), pushing daily active users past 350,000. Steam peak CCU: 27,227. Ongoing investment in ZvZ spell system and network event optimization. Gcore CDN reduces infrastructure costs by 40%.
