Architecture Comparison / Destiny 2

Destiny 2

Hybrid P2P / Server Authority Shared World Shooter / MMO-Lite

Bungie's shared world shooter uses a networking model that is genuinely unlike anything else in the industry. It is not peer-to-peer, it is not dedicated servers, and Bungie deliberately avoids both terms. Every activity runs on Bungie's servers for physics and mission logic, but each player is authoritative over their own movement and abilities. The result is a game that feels like a single-player shooter even in co-op, at the cost of PvP integrity and vulnerability to network manipulation.

Overview

Destiny 2 is a shared-world first-person shooter where players explore a solar system filled with PvE activities (story campaigns, strikes, raids, dungeons, patrols, public events) and PvP modes (Crucible, Trials of Osiris, Iron Banner). The game launched in 2017 under the Activision publishing deal, moved to self-publishing in 2019, and migrated from Blizzard's Battle.net to Steam. It operates as a live service with seasonal content updates and annual expansions.

The game runs on the Tiger Engine, a proprietary engine forked from the Halo-era "blam!" engine in 2008 and developed over six years before Destiny 1 shipped. The Tiger Engine was built from the ground up to support Bungie's vision of a "shared world shooter," where players seamlessly encounter strangers in patrol zones, transition between activities without loading screens, and play cooperative content that feels as responsive as a solo FPS.

Destiny 2's networking model is the architectural decision that defines everything about the game's technical character. Bungie calls it a hybrid, but that undersells how unusual it is. The server handles physics simulation, mission scripting, and game state authority. Each client handles its own movement and ability execution with full local authority. Other players' positions are communicated peer-to-peer between clients, not relayed through the server. This means there is no rubber-banding on your own character, ever, regardless of latency. But it also means PvP hit registration depends on what each player's client thinks happened, and disputes are resolved by a system that intentionally favors both the shooter and the target.

Tiger Engine

Forked from the Halo "blam!" engine in 2008 during Halo: Reach development. Code-named "Tiger" after a piece of concept art featuring a space tiger. Supports physics simulation, rendering, animation, and the unique hybrid networking model. Major engine overhauls came with Beyond Light (2020), which shifted mission scripting to the Physics Host and improved social space transitions.

Client-Authoritative Movement

Each player's position and movement is calculated locally on their own machine and communicated directly to other players. The server never overrides your position. This eliminates rubber-banding entirely, making the game feel responsive regardless of latency. The trade-off: lag switches and network manipulation can make players appear to teleport, and there is no server-side position validation to catch it.

Server Authority: Physics + Mission

While movement is client-authoritative, Bungie's servers run the Physics Host (world physics simulation) and Mission Host (scripting, objective progression, loot drops). In Destiny 1, the Physics Host ran on a player's console, causing host migration problems. In Destiny 2, both hosts run in Bungie's data centers, eliminating mid-raid host migrations.

Shared World Design

Patrol zones silently matchmake players into shared instances of up to 9 players (3 fireteams). As you walk toward a new zone, the game contacts the matchmaking server to find an existing instance and seamlessly loads you into it. Social spaces (the Tower) support 20-26 players per instance. Transitions happen without loading screens, and other players appear and disappear at zone boundaries.

Server Infrastructure

Bungie operates its own server cloud to host Destiny 2. When Bungie split from Activision in January 2019, they invested in new server infrastructure to handle the transition to self-publishing. The game moved from Blizzard's Battle.net to Steam in October 2019, but the game servers themselves were always Bungie-hosted, separate from the platform storefront.

Virtual Headless Consoles

Bungie describes their server infrastructure as running "hundreds of thousands of virtual headless consoles" in their cloud. Each activity instance (a raid, a Crucible match, a patrol zone) runs on one of these virtual machines. The server process runs the same game code as the client but without rendering, acting as the Physics Host and Mission Host for the activity. This is the key architectural change from Destiny 1, where the Physics Host ran on a player's console and the Mission Host ran in the data center.

What the Server Controls

The Bungie server is authoritative over:

  • Physics simulation (world physics, environmental objects, vehicles)
  • Mission scripting (objective progression, encounter triggers, boss mechanics)
  • AI behavior (enemy spawning, pathfinding, aggro targeting)
  • Loot drops (what items drop, their stats/rolls)
  • Game state (match score, zone control, public event progress)
  • Economy (Glimmer, Legendary Shards, currency transactions)

What the Client Controls

Each player's client is authoritative over:

  • Own movement and position (no rubber-banding, ever)
  • Own ability execution (grenades, supers, class abilities)
  • Hit registration (the shooter's client determines hits)

Social Space Instancing

The Tower (the primary social hub) runs as a separate instance type with different player limits. Historically, Tower instances supported 40-50+ players, but persistent performance issues (particularly on PC since Beyond Light) led Bungie to temporarily reduce the cap to 20 players per instance. The performance problems stemmed from menu operations (inspecting armor stats, changing shaders) requiring server round-trips that scaled poorly with instance population. After the cap reduction, players reported significant improvement in menu responsiveness.

Network & Protocols

Destiny 2's networking is, by Bungie's own admission, "uniquely complicated." The core principle is a peer-to-peer mesh for player state overlaid on a server-authoritative backbone for game logic. Clients send position and action data directly to other clients (not through the server), while the server manages physics, AI, loot, and mission state. All traffic flows over UDP.

Why Bungie Chose This Model

Bungie's explicit design goal was for Destiny to "feel like a single-player shooter." In a pure client-server model, all physics calculations happen on the server, and the client is largely a rendering endpoint. This introduces latency between input and outcome that makes the game feel sluggish, especially for movement and abilities. Bungie wanted instant feedback: when you press jump, you jump now, not 50-100ms from now. When you throw a grenade, the trajectory starts immediately, not after a server round-trip. By making each client authoritative over its own actions, the game feels perfectly responsive regardless of latency.

The trade-off is that PvP becomes fundamentally less competitive. In Crucible, both players in a gunfight are seeing slightly different versions of reality (each is authoritative over their own position, but sees the other player with latency delay). Destiny's resolution system intentionally supports "trades" (both players killing each other) rather than forcing one player to be wrong. This is a conscious design decision: Bungie would rather both players feel like their shots hit than have one player feel cheated by lag compensation.

Client Update Rates

Destiny 2 does not have a traditional "server tick rate" in the way that games with pure dedicated servers do. A player's PC sends approximately 40 updates per second to other players via the P2P mesh. However, the rate at which a given player receives updates from other players varies based on the sender's hardware and network conditions. On console, update rates can be noticeably lower and less consistent than on PC. Since Destiny 2 does not relay player positions through a central server, there is no single authoritative tick rate to point to.

IP Exposure and DDoS

The peer-to-peer component of the network model historically exposed players' IP addresses to every other player in the activity. This made DDoS attacks trivial in competitive modes like Trials of Osiris, where a player could force opponents to disconnect by flooding their IP. Bungie addressed this on PC by migrating from Windows Sockets to Steam Network Sockets, which route traffic through Steam's relay infrastructure. Under this system, players are addressed by Steam ID rather than IP, preventing direct IP exposure. Console platforms have their own NAT traversal but are not immune to network-level attacks.

The "Kill Trading" Problem

In Destiny 2 PvP, it is common for two players to kill each other simultaneously. This happens because both clients are authoritative over their own hit registration. Player A shoots Player B on Player A's screen. Player B shoots Player A on Player B's screen. Both clients report valid kills. The server accepts both. Neither player is "wrong" in the traditional sense of the word, but the experience feels frustrating to players accustomed to games where the server picks one winner. Bungie considers this an acceptable trade-off for the responsiveness of their hybrid model.

Data & Persistence

Destiny 2 maintains an extensive persistent data layer for characters, items, progression, and collections. The Bungie.net API exposes this data through a comprehensive REST interface that powers both the official companion app and a thriving ecosystem of third-party tools.

Inventory and Vault System

Each player has three characters (one per class), each with their own equipped items and a 10-slot-per-type inventory. A shared Vault provides additional storage across characters, along with a general inventory with three 50-item categories. Items are represented as DestinyItemComponent objects with properties including itemHash (the item definition), itemInstanceId (unique per drop), bucketHash (the inventory slot), and transferStatus flags. Location constants track where each item resides: character inventory, vault, vendor, or postmaster.

Bungie.net API Architecture

The API uses three fundamental endpoints: GetProfile, GetCharacter, and GetItem. These replaced the older large specialized endpoints and use a component-based query system where callers specify which data layers they want via the ?components= query parameter. Authentication uses OAuth2 scopes, with ReadDestinyInventoryAndVault being the primary scope needed for inventory and progression data access.

Rate limiting is enforced per-endpoint. For example, the InsertSocketPlugFree endpoint (for applying mods) is limited to 2 actions per second per user. Certain items are protected from API manipulation through hash arrays in the Destiny2CoreSettings.

Third-Party Tool Ecosystem

The open API has spawned a rich tool ecosystem. Destiny Item Manager (DIM) is the most prominent, built as a React/Redux/TypeScript web application that connects to the Bungie.net API for safe inventory manipulation without accessing game memory. DIM enables players to transfer items between characters and vault without visiting the Tower, manage loadouts, and analyze item stats. Other tools include Braytech, light.gg, and d2gunsmith.com for weapon analysis, plus damagecheck.io for DPS calculations.

Damage Calculation Model

Destiny 2's damage system is server-validated with significant complexity. Base damage per bullet is a linear function of the weapon's attack stat, with the Impact stat affecting the function's coefficient and intercept. Additional modifiers are applied as percentage multiplications on the base damage. Weapon archetypes within a frame type deal identical damage (all Adaptive Sniper Rifles do the same DPS), with Exotic weapons as exceptions. Power level differences affect damage asymmetrically: being overleveled provides no damage bonus, but being underleveled applies steep penalties (~33% less at 1 level below, ~53% at 3 levels below). This system forces engagement with the power level grind even for veteran players.

Cross-Save System

After splitting from Activision, Bungie implemented cross-save allowing a single set of characters to be used across all platforms (PC, PlayStation, Xbox). The system links platform accounts to a Bungie.net account and designates one platform's characters as the "active" set. DLC purchases remain platform-specific, but Silver (premium currency) and season passes are shared. This replaced the old model where PC progression was locked to Battle.net.

Content Vault (Sunset Architecture)

In 2020, Bungie introduced the Destiny Content Vault, removing older campaigns, raids, and destinations from the active game. This was an architectural necessity: the Tiger Engine struggled with the accumulated weight of years of content, causing long patch times, excessive storage requirements, and testing burden. Vaulted content could theoretically return (as some raids have), but the vault represents a permanent reduction in live game size to keep the engine manageable.

Scaling & Performance

Destiny 2's scaling challenges are different from traditional MMOs or battle royales because the game never puts more than 26 players in a single instance. The scaling problem is the sheer number of simultaneous activity instances running globally, each requiring its own server-side Physics Host and Mission Host.

Patrol Zone Matchmaking

The most technically interesting scaling challenge is patrol zones. When a player moves toward a new zone boundary, the game contacts the matchmaking server to find an existing instance of that zone. If one exists with available capacity, the player is silently loaded into it. If not, a new instance is spun up. Zone state (enemy spawns, public event progress, patrol mission state) is maintained per-instance by the Zone Host on the server. When the last player leaves a zone instance, the instance is cleaned up.

Zone instances are independent server processes that do not communicate with each other. The geometry you see is loaded locally on your machine. Server-side, each zone runs its own mini version of the Destiny executable. This means a patrol destination like the EDZ might have hundreds of simultaneous zone instances running globally, each tracking 3-9 players, each with its own AI spawns and event timers. Players can walk between zones seamlessly because the transition is handled by the matchmaking server finding or creating an instance on the destination side.

Raid Complexity

Raids represent the highest server-side computational cost per player. A raid encounter like Riven of a Thousand Voices or The Witness involves multiple scripted phases with complex AI behavior, environmental mechanics (moving platforms, rotating geometry, damage-check timers), and strict state machines governing encounter progression. The Mission Host runs all of this scripting, while the Physics Host handles the environmental physics. A wipe (party kill) requires resetting the encounter state machine to its initial configuration, which must be fast enough that players don't experience a long reload.

Gambit's Dual-Arena Problem

Gambit's PvEvP mode is architecturally distinctive because it maintains two parallel PvE arenas plus a PvP invasion link between them. Each team of four fights AI enemies in their own arena, collecting motes to bank toward summoning a boss. When a player invades the opposing team, their client must seamlessly transition from one arena's state to the other. The server tracks both arenas simultaneously: separate enemy spawns, separate mote counts, separate boss health. The invasion mechanic requires cross-arena player state transfer, which adds complexity to the already-unusual P2P mesh model.

Trials of Osiris: Card-Based Matchmaking

Trials uses a unique matchmaking model where players purchase a "passage" card and attempt to win 7 consecutive matches to "go flawless." Matchmaking prioritizes pairing players with the same number of wins on their current card, mimicking a tournament bracket. Bungie has oscillated between card-based matchmaking (match by wins on card) and win-based matchmaking (match by total wins that weekend). The card system creates naturally escalating difficulty since you face opponents on similar win streaks, while the win-based system caused controversy because matches got harder even after losses. Additional refinements include fireteam-size matching (solo players face other solos) and post-flawless pool separation.

Technical Deep Dives

The GDC 2015 Talks

Two GDC 2015 presentations provide the most authoritative public documentation of Destiny's networking architecture:

"Shared World Shooter: Destiny's Networked Mission Architecture" by Justin Truman covered the spatial structure of Destiny's game world, the various networked experiences a player is simultaneously connected to, their overlapping responsibilities, rules for host handoff at regional boundaries, and how ungraceful host migrations and disconnections are handled. The key insight is that Destiny's world is built from overlapping "bubbles" of networked state, and the challenge is stitching these bubbles together so players can seamlessly move between them.

"Lessons from the Core Engine Architecture of Destiny" by Chris Butcher covered the six-year development arc of the Tiger Engine from inception to ship, including techniques for large-scale engine refactoring, data lifetime management, object system design, and source code layering. This talk revealed that the engine fork from blam! was driven by the recognition that the Halo codebase was fundamentally unsuited to the persistent, shared-world experience Bungie wanted to build.

The Beyond Light Engine Overhaul

When Destiny 2: Beyond Light launched in November 2020, Bungie shipped significant engine changes. The most important for networking was shifting the mission scripting model to run on the Physics Host rather than the Mission Host. This change unified the two server-side roles and enabled new behaviors: patrol missions now persist as players move between zones on a destination (previously, they would reset), and joining or leaving a fireteam in the Tower no longer required reloading the entire social space. These quality-of-life improvements came from reducing the separation between the Physics and Mission execution environments.

Weapon Sandbox and Server Validation

Destiny 2's weapon sandbox is a complex system with layers of damage modifiers. Weapon damage per bullet is a linear function of attack stat and Impact, with additional multipliers from buffs, debuffs, enemy type, and power level delta. Scout rifles, sniper rifles, and handcannons take additional penalties against Major and Ultra enemies (up to 25% for handcannons against Ultras), while auto rifles and pulse rifles do not.

DPS (damage per second) is generally calculated per archetype rather than per individual weapon: all Adaptive Sniper Rifles deal the same damage at the same rate of fire. Exotic weapons are exceptions with unique damage profiles. The server validates damage outcomes, but because hit registration is client-authoritative, the validation is largely confirming that the reported hit is plausible (correct weapon, correct target, reasonable position) rather than re-simulating the entire shot trajectory.

Season Pass Content Delivery

Destiny 2's seasonal content model delivers new activities, weapons, and narrative beats on a quarterly cadence. Seasonal content is pre-loaded in patches but time-gated through server-side flags. The Mission Host controls when seasonal activities become available, and the matchmaking server adds new playlists to the activity selection. Hotfixes can adjust weapon balance, ability tuning, and economy values without client patches by pushing server-side configuration changes. This split between client-side content delivery (assets, code) and server-side activation (timing, tuning) allows Bungie to control the content drip without requiring frequent large downloads.

2014 Destiny 1 launches. The hybrid P2P/server model debuts. The Physics Host runs on a player's console, causing mid-activity host migrations when that player disconnects. The Mission Host runs in Bungie's data center. The model is controversial but enables the responsive gunplay that defines the franchise.
2015 GDC talks reveal architecture details. Justin Truman presents the networked mission architecture, Chris Butcher presents engine lessons. The community gains its deepest understanding of how the shared world stitching, host handoff, and bubble-based instancing work.

The P2P mesh for player state is retained. PC launch on Battle.net is the first non-Blizzard title on the platform.
2019 Bungie splits from Activision, migrates to Steam. Self-publishing begins with Shadowkeep. The game goes free-to-play as "New Light." Cross-save enables character portability across platforms. The Bungie.net API ecosystem matures, with DIM becoming the de facto inventory management tool.

The Content Vault removes older campaigns and destinations to reduce game size and maintenance burden. PC networking migrates from Windows Sockets to Steam Network Sockets, hiding player IP addresses to combat DDoS in Trials.
