Risks

What can go wrong with each stack.

Unity + Photon

Unity changes pricing againMedium

They've done it 3 times in 3 years. Mitigation: build so the client can be swapped out.

Photon requires Windows ServerMedium

Limits hosting options and adds ~$20-40/mo per machine in OS licensing.

Scaling past PhotonLow

Albion handles 350K+ DAU on Photon. You'd need to be extremely successful to hit this ceiling.

Godot + Custom C#

No reference architecture existsHigh

No shipped MMO on this stack. Every networking problem is a novel problem.

Building netcode from scratchHigh

Reliable UDP, client prediction, interest management, zone handoff. Months of work before you have a playable game.

Team burnoutHigh

Networking infrastructure is unglamorous grind. Demoralizing before you see a boat on water.

Vendor riskLow

MIT license is irrevocable. Safest business decision of the three.

SpacetimeDB

Technology is brand newHigh

Hit 1.0 in March 2025. Expect breaking changes, sparse docs, and rough edges.

Rust is hard to learnMedium

The borrow checker will slow you down for months if choosing Rust. C# modules are now supported, which drops this barrier significantly. But Rust modules have the best docs and community support so far.

All state must fit in RAMMedium

RAM scales with world state size. A complex MMO could need significant RAM per instance; multi-instance sharding adds its own complexity.

Clockwork Labs could failMedium

VC-backed ($26M from a16z/Supercell), but still a startup. Source-available under BSL 1.1, converts to AGPL v3 in 2031. AGPL is copyleft, which may affect proprietary server code.

Verdict

What each stack actually means for the Wompworld team.

Unity + Photon Server

The safe bet. Albion Online is proof this works for exactly your type of game. The documentation exists, the reference architecture exists, the hiring pool exists. You pay more in licensing and accept vendor risk. But you trade money for certainty, and for a team learning game dev as they go, certainty is worth paying for.

Choose this if: Shipping matters more than owning every line of code.

Godot + Custom C# Server

Maximum freedom, maximum work. Zero vendor risk, zero licensing costs, you own everything. But you're building networking infrastructure that Photon gives you for free, with no reference MMO to guide you. The 6-12 extra months could be the difference between shipping and not shipping.

Choose this if: Someone on the team has multiplayer networking experience, or you refuse to depend on Unity/Photon on principle.

SpacetimeDB

The wildcard. If it works, it eliminates entire categories of bugs and months of infrastructure code. C# module support lowers the entry barrier vs pure Rust. But the tech is young (1.0 Mar 2025), the license is copyleft (AGPL v3), and nobody has proven this at Wompworld's target scale. High ceiling, uncertain floor.

Choose this if: You're comfortable being an early adopter on a platform that's well-funded but still maturing.

The Pragmatic Recommendation

Start with Unity + Photon, following the Albion playbook. Build the server as a standalone C# app that talks to the client over a clean protocol. If Unity does something unacceptable, port the client to Godot without touching server code. If SpacetimeDB matures in 12-18 months, evaluate migrating the server layer.

The question isn't which stack is "best" in the abstract. It's which one gives you the highest probability of having a playable game in 18 months. That's Unity + Photon, by a meaningful margin.
